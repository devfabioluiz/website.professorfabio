O que é um Banco de Dados?

Um banco de dados é uma coleção organizada de dados estruturados, geralmente armazenados e acessados eletronicamente de um sistema computacional. Os bancos de dados permitem armazenar, recuperar, modificar e excluir dados de forma eficiente e segura.

Um SGBD (Sistema de Gerenciamento de Banco de Dados) é o software responsável por gerenciar o banco de dados. Ele atua como uma interface entre o usuário e os dados, garantindo:

  • Persistência: os dados sobrevivem além da execução do programa
  • Concorrência: múltiplos usuários acessam os dados simultaneamente
  • Segurança: controle de acesso e permissões
  • Integridade: regras e restrições para manter os dados consistentes
  • Recuperação: mecanismos de backup e restauração

História e Evolução dos Bancos de Dados

A evolução dos bancos de dados acompanha a história da computação. Veja a linha do tempo:

Década Modelo Características
1960 Sistemas de Arquivos Dados armazenados em arquivos planos, sem estrutura padronizada
1960-1970 Modelo Hierárquico e em Rede Primeiros SGBDs comerciais (IMS, IDMS). Dados organizados em árvores ou grafos
1970 Modelo Relacional Edgar F. Codd publica o artigo "A Relational Model of Data for Large Shared Data Banks"
1980 SQL e SGBDs Relacionais SQL torna-se padrão. Surgem Oracle, DB2, PostgreSQL, MySQL
1990 Modelo Orientado a Objetos Tentativa de unir OO com bancos de dados. Surgem ODMG e produtos como ObjectStore
2000-2010 NoSQL Surgem MongoDB, Cassandra, Redis para cenários de Big Data e escalabilidade horizontal
2010-presente NewSQL e Poliglotas Bancos que unem SQL e escalabilidade (CockroachDB, Spanner). Múltiplos modelos no mesmo sistema

Modelos de Banco de Dados

Cada modelo de banco de dados oferece uma forma diferente de organizar e relacionar os dados. A tabela a seguir compara os principais modelos:

Modelo Estrutura Vantagens Desvantagens Exemplo
Arquivos Arquivos planos (CSV, texto) Simples, sem dependência de software externo Sem controle de concorrência, redundância, difícil de consultar Arquivos .csv, .txt
Rede Grafo direcionado (registros e sets) Acesso rápido a relacionamentos complexos Complexo de programar, difícil de modificar a estrutura IDMS
Hierárquico Árvore (pai-filho) Rápido para consultas previsíveis Rígido, não suporta relacionamentos muitos-para-muitos IMS (IBM)
Relacional Tabelas (relações) com linhas e colunas Flexível, SQL padronizado, integridade forte Escalabilidade vertical limitada, esquema rígido PostgreSQL, MySQL, Oracle
Orientado a Objetos Objetos com atributos e métodos Integração natural com OOP, tipos complexos Pouca adoção, ecossistema limitado ObjectDB, db4o
NoSQL Documentos, chave-valor, grafos, colunar Escalabilidade horizontal, flexível, alta performance Sem SQL padrão, consistência eventual, menos garantias ACID MongoDB, Cassandra, Redis, Neo4j

Modelo Relacional

Proposto por Edgar F. Codd em 1970, o modelo relacional organiza os dados em tabelas (também chamadas de relações). É o modelo mais utilizado atualmente.

Conceitos Fundamentais

  • Tabela (Relação): estrutura bidimensional com linhas e colunas
  • Linha (Tupla): representa um registro único na tabela
  • Coluna (Atributo): representa uma característica da entidade
  • Chave Primária (PK): identificador único de cada linha
  • Chave Estrangeira (FK): relaciona linhas entre tabelas diferentes
  • Domínio: conjunto de valores válidos para um atributo

Exemplo de Tabela Relacional


-- Tabela de alunos
CREATE TABLE alunos (
    id        SERIAL PRIMARY KEY,
    nome      VARCHAR(100) NOT NULL,
    email     VARCHAR(100) UNIQUE,
    idade     INTEGER CHECK (idade > 0),
    curso_id  INTEGER REFERENCES cursos(id)
);

-- Tabela de cursos
CREATE TABLE cursos (
    id     SERIAL PRIMARY KEY,
    nome   VARCHAR(80) NOT NULL,
    carga  INTEGER NOT NULL
);

As 12 Regras de Codd

Em 1985, Codd publicou 13 regras (numeradas de 0 a 12) que um SGBD deve seguir para ser considerado verdadeiramente relacional. Nenhum SGBD comercial atende todas perfeitamente, mas elas servem como guia de qualidade.

  • Regra 0 — Regra Fundamental: O sistema deve gerenciar os dados exclusivamente através de suas capacidades relacionais
  • Regra 1 — Informação: Toda informação deve ser representada como valores em tabelas
  • Regra 2 — Acesso Garantido: Todo valor deve ser acessível pela combinação de tabela, chave primária e coluna
  • Regra 3 — Tratamento Sistemático de Nulos: O SGBD deve tratar NULLs de forma uniforme e independente do tipo de dado
  • Regra 4 — Catálogo Online: A descrição do banco de dados (metadados) deve ser armazenada como dados comuns, acessível via SQL
  • Regra 5 — Sublinguagem de Dados: O sistema deve suportar uma linguagem com sintaxe linear para definição, manipulação, integridade e controle de transações
  • Regra 6 — Atualização de Views: Todas as views teoricamente atualizáveis devem ser atualizáveis pelo sistema
  • Regra 7 — Inserção, Atualização e Exclusão de Alto Nível: O sistema deve suportar operações que tratam conjuntos de tuplas de uma só vez
  • Regra 8 — Independência Física dos Dados: Mudanças no armazenamento físico não devem afetar a aplicação
  • Regra 9 — Independência Lógica dos Dados: Mudanças nas tabelas base não devem afetar a aplicação
  • Regra 10 — Independência de Integridade: Restrições de integridade devem ser definidas na linguagem de dados e armazenadas no catálogo
  • Regra 11 — Independência de Distribuição: A aplicação não deve saber se os dados estão distribuídos fisicamente
  • Regra 12 — Não-Subversão: Se o sistema tiver uma interface de baixo nível, ela não deve burlar as regras de integridade

PostgreSQL vs Outros SGBDs

O PostgreSQL é um dos SGBDs relacionais de código aberto mais avançados do mercado. Veja como ele se compara com outras opções populares:

Característica PostgreSQL MySQL Oracle SQL Server
Licença Open Source (PostgreSQL License) Open Source (GPL) Proprietária (cara) Proprietária (cara)
Conformidade ACID Completa Depende da engine (InnoDB sim) Completa Completa
Tipos de Dados Rico: JSON, Array, hstore, Geo, XML, UUID, etc. Básico, com JSON suporte recente Muito rico, mas proprietário Rico, integrado ao ecossistema .NET
Extensibilidade Extensões, linguagens procedurais (PL/pgSQL, PL/Python) Plugins, stored procedures PL/SQL, pacotes T-SQL, CLR integration
Performance Excelente para consultas complexas Muito rápida em leitura simples Alta performance em grandes volumes Excelente com ecossistema Microsoft
Replicação Streaming, lógica, síncrona e assíncrona Master-Slave, Group Replication Data Guard, GoldenGate Always On, Mirroring
Curva de Aprendizado Média Baixa Alta Média (se familiar com MS)

O PostgreSQL destaca-se por sua conformidade com os padrões SQL, extensibilidade e robustez, sendo a escolha ideal para este curso.

Passo a passo: Instalando o PostgreSQL

Vamos instalar o PostgreSQL no Windows. Siga os passos abaixo cuidadosamente.

Passo 1 — Baixar o instalador

Acesse o site oficial e faça o download do instalador para Windows:

  • Acesse https://www.postgresql.org/download/windows/
  • Clique em "Download the installer"
  • Selecione a versão mais recente (ex: PostgreSQL 17)
  • Escolha o instalador para Windows x86-64

Passo 2 — Executar o instalador

  • Execute o arquivo .exe baixado como Administrador
  • Escolha o diretório de instalação (padrão: C:\Program Files\PostgreSQL\17)
  • Selecione os componentes: PostgreSQL Server, pgAdmin 4, Command Line Tools
  • Defina o diretório de dados (padrão: C:\Program Files\PostgreSQL\17\data)

Passo 3 — Configurar senha e porta

  • Defina uma senha para o superusuário postgres
  • Mantenha a porta padrão: 5432
  • Mantenha o locale padrão
  • Conclua a instalação

Passo 4 — Verificar a instalação

Abra o PowerShell ou Prompt de Comando e execute:


psql --version

Você deve ver a versão do PostgreSQL instalada, algo como:


psql (PostgreSQL) 17.0

Passo a passo: Criando seu primeiro banco de dados

Após instalar o PostgreSQL, vamos criar nosso primeiro banco de dados.

Passo 1 — Acessar o PostgreSQL via terminal

Abra o PowerShell e conecte-se ao PostgreSQL com o superusuário postgres:


psql -U postgres

Digite a senha configurada durante a instalação.

Passo 2 — Criar o banco de dados

Dentro do terminal psql, execute o comando para criar um novo banco:


CREATE DATABASE minha_primeira_db;

Passo 3 — Listar todos os bancos de dados

Verifique se o banco foi criado com sucesso:


\l

A lista exibirá todos os bancos de dados existentes, incluindo o minha_primeira_db.

Passo 4 — Conectar ao banco criado

Para conectar-se ao banco recém-criado:


\c minha_primeira_db

O terminal exibirá a mensagem: You are now connected to database "minha_primeira_db" as user "postgres".

Passo 5 — Remover um banco de dados (se necessário)

Para excluir um banco de dados, primeiro desconecte-se dele e execute:


DROP DATABASE IF EXISTS minha_primeira_db;

3. Pesquise e liste 3 diferenças entre PostgreSQL e MySQL

Faça uma pesquisa e escreva pelo menos 3 diferenças significativas entre esses dois SGBDs.

Gabarito:
  • Tipos de dados: PostgreSQL oferece tipos nativos como JSONB, Array, UUID e hstore, enquanto MySQL tem suporte mais limitado a tipos avançados.
  • Conformidade ACID: PostgreSQL implementa ACID de forma completa e nativa. MySQL depende da engine InnoDB para suporte ACID completo.
  • Padrões SQL: PostgreSQL segue rigorosamente os padrões SQL, sendo mais fiel à especificação. MySQL historicamente priorizou performance sobre conformidade, embora tenha melhorado nas versões recentes.
  • Extensibilidade: PostgreSQL suporta extensões customizadas e múltiplas linguagens procedurais (PL/pgSQL, PL/Python, PL/Perl), enquanto MySQL tem um ecossistema de plugins mais restrito.
