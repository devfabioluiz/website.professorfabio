Anomalias em Tabela Única

Quando armazenamos todos os dados em uma única tabela (modelo não normalizado), diversos problemas podem surgir. Esses problemas são chamados de anomalias e comprometem a integridade e a consistência dos dados.

Considere a seguinte tabela não normalizada pedidos_ruim:


CREATE TABLE pedidos_ruim (
    pedido_id      INT,
    cliente_nome   VARCHAR(100),
    cliente_cidade VARCHAR(50),
    cliente_estado VARCHAR(2),
    produto_nome   VARCHAR(100),
    produto_preco  NUMERIC(10, 2),
    quantidade     INT,
    data_pedido    DATE
);

INSERT INTO pedidos_ruim VALUES
(1, 'João Silva',  'São Paulo',  'SP', 'Notebook',  3500.00, 1, '2025-01-15'),
(2, 'Maria Souza', 'Campinas',   'SP', 'Mouse',      150.00, 2, '2025-01-16'),
(3, 'João Silva',  'São Paulo',  'SP', 'Teclado',    250.00, 1, '2025-01-16'),
(4, 'Pedro Costa', 'Rio de Janeiro', 'RJ', 'Monitor', 1200.00, 1, '2025-01-17');

Anomalia de Atualização (Update Anomaly)

Ocorre quando a mesma informação está repetida em várias linhas e, ao atualizá-la, é necessário modificar múltiplos registros. Se algum registro não for atualizado, os dados ficam inconsistentes.

Exemplo: Se João Silva mudar de cidade, precisamos atualizar os pedidos 1 e 3. Se esquecermos de atualizar um deles, João Silva terá duas cidades diferentes nos registros:


-- Precisamos atualizar TODAS as linhas do João Silva
UPDATE pedidos_ruim
SET cliente_cidade = 'Guarulhos'
WHERE cliente_nome = 'João Silva';

-- Risco: se houver 500 pedidos do João,
-- a atualização pode ser lenta e propensa a erro

Anomalia de Inserção (Insert Anomaly)

Ocorre quando não é possível inserir um novo registro por falta de informações que não deveriam ser obrigatórias para aquela operação.

Exemplo: Queremos cadastrar um novo cliente "Ana Beatriz" que ainda não fez nenhum pedido. Como a tabela pedidos_ruim exige dados do pedido (pedido_id, produto, quantidade), não conseguimos inserir apenas os dados do cliente. Somos obrigados a inventar um pedido fictício ou deixar campos nulos.

Anomalia de Exclusão (Delete Anomaly)

Ocorre quando, ao remover um registro, perdemos informações que não deveriam ser excluídas.

Exemplo: Se excluirmos todos os pedidos do Pedro Costa (pedido_id = 4), perdemos completamente a informação de que o Pedro Costa é um cliente cadastrado. Os dados do cliente desaparecem junto com os pedidos:


-- Ao deletar o pedido 4, perdemos os dados do Pedro Costa
DELETE FROM pedidos_ruim WHERE pedido_id = 4;

-- Agora não sabemos mais que Pedro Costa existe como cliente

A solução para todas essas anomalias é dividir os dados em múltiplas tabelas relacionadas, processo conhecido como normalização.

Por que Múltiplas Tabelas?

Dividir os dados em múltiplas tabelas especializadas traz vantagens fundamentais para a qualidade do banco de dados:

  • Elimina redundância: cada dado é armazenado em um único local. O nome do cliente está apenas na tabela clientes, e não repetido em cada pedido.
  • Evita anomalias: atualizar o nome de um cliente requer modificar apenas um registro na tabela clientes.
  • Integridade dos dados: restrições como chaves estrangeiras garantem que relacionamentos sejam válidos.
  • Facilita a manutenção: alterações na estrutura de uma entidade não impactam as demais.
  • Melhora a performance: tabelas menores e mais enxutas consomem menos memória e aceleram consultas.
  • Escalabilidade: cada tabela pode crescer independentemente.

No exemplo anterior, a tabela pedidos_ruim seria dividida em:

  • clientes — armazena dados exclusivos do cliente (id, nome, cidade, estado)
  • produtos — armazena dados exclusivos do produto (id, nome, preço)
  • pedidos — armazena a relação: qual cliente fez qual pedido e quando
  • itens_pedido — armazena quais produtos estão em cada pedido e em qual quantidade

Chave Estrangeira (FOREIGN KEY)

Uma chave estrangeira (Foreign Key — FK) é uma coluna (ou conjunto de colunas) em uma tabela que referencia a chave primária de outra tabela. Ela estabelece o vínculo lógico entre as tabelas, garantindo a integridade referencial.

Sintaxe Básica


-- Tabela pai (referenciada)
CREATE TABLE clientes (
    id   SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL
);

-- Tabela filha (contém a FK)
CREATE TABLE pedidos (
    id         SERIAL PRIMARY KEY,
    cliente_id INT NOT NULL,
    data_pedido DATE DEFAULT CURRENT_DATE,
    CONSTRAINT fk_pedidos_clientes
        FOREIGN KEY (cliente_id)
        REFERENCES clientes (id)
);

Com a FK definida, o PostgreSQL impede:

  • Inserir um pedido com cliente_id que não existe na tabela clientes
  • Excluir um cliente que possui pedidos associados (a menos que ações adicionais sejam configuradas)

ON DELETE e ON UPDATE

As cláusulas ON DELETE e ON UPDATE definem o comportamento quando o registro referenciado é excluído ou atualizado:

Ação Comportamento
CASCADE Propaga a operação para os registros filhos (ex.: deletar cliente deleta seus pedidos)
SET NULL Define a FK como NULL nos registros filhos
SET DEFAULT Define a FK para o valor padrão nos registros filhos
RESTRICT Impede a operação se existirem registros filhos (comportamento padrão)
NO ACTION Similar ao RESTRICT, mas a verificação é adiada até o final da transação 

-- Exemplo com ON DELETE CASCADE e ON UPDATE CASCADE
CREATE TABLE pedidos (
    id         SERIAL PRIMARY KEY,
    cliente_id INT NOT NULL,
    data_pedido DATE DEFAULT CURRENT_DATE,
    CONSTRAINT fk_pedidos_clientes
        FOREIGN KEY (cliente_id)
        REFERENCES clientes (id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);
          

-- Exemplo com ON DELETE SET NULL
CREATE TABLE pedidos (
    id         SERIAL PRIMARY KEY,
    cliente_id INT,  -- agora permite NULL
    data_pedido DATE DEFAULT CURRENT_DATE,
    CONSTRAINT fk_pedidos_clientes
        FOREIGN KEY (cliente_id)
        REFERENCES clientes (id)
        ON DELETE SET NULL
);

Cuidado: use CASCADE com atenção, pois pode causar exclusões em cascata inesperadas.

Tipos de Relacionamentos

Os relacionamentos entre tabelas representam como as entidades do mundo real se conectam. Existem três tipos fundamentais:

Relacionamento 1:1 (Um para Um)

Cada registro na tabela A está associado a no máximo um registro na tabela B, e vice-versa.

Exemplo: Um usuário possui um único perfil, e um perfil pertence a um único usuário.


CREATE TABLE usuarios (
    id    SERIAL PRIMARY KEY,
    nome  VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL
);

CREATE TABLE perfis (
    id         SERIAL PRIMARY KEY,
    usuario_id INT UNIQUE NOT NULL,  -- UNIQUE garante 1:1
    bio        TEXT,
    avatar_url VARCHAR(200),
    CONSTRAINT fk_perfis_usuarios
        FOREIGN KEY (usuario_id)
        REFERENCES usuarios (id)
        ON DELETE CASCADE
);

Note o uso de UNIQUE na FK, que impede que o mesmo usuário tenha mais de um perfil.

Relacionamento 1:N (Um para Muitos)

Cada registro na tabela A pode estar associado a vários registros na tabela B, mas cada registro em B está associado a apenas um registro em A.

Exemplo: Um cliente pode fazer vários pedidos, mas cada pedido pertence a um único cliente.


CREATE TABLE clientes (
    id    SERIAL PRIMARY KEY,
    nome  VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE
);

CREATE TABLE pedidos (
    id         SERIAL PRIMARY KEY,
    cliente_id INT NOT NULL,
    data_pedido DATE DEFAULT CURRENT_DATE,
    valor_total NUMERIC(10, 2),
    CONSTRAINT fk_pedidos_clientes
        FOREIGN KEY (cliente_id)
        REFERENCES clientes (id)
);

Este é o tipo de relacionamento mais comum em bancos de dados relacionais.

Relacionamento N:N (Muitos para Muitos)

Cada registro na tabela A pode estar associado a vários registros na tabela B, e vice-versa. Este relacionamento requer uma tabela associativa (ou tabela de junção).

Exemplo: Um aluno pode se matricular em vários cursos, e um curso pode ter vários alunos matriculados.


-- Tabela alunos
CREATE TABLE alunos (
    id    SERIAL PRIMARY KEY,
    nome  VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE
);

-- Tabela cursos
CREATE TABLE cursos (
    id     SERIAL PRIMARY KEY,
    nome   VARCHAR(100) NOT NULL,
    carga_horaria INT NOT NULL
);

-- Tabela associativa para N:N
CREATE TABLE matriculas (
    aluno_id INT NOT NULL,
    curso_id INT NOT NULL,
    data_matricula DATE DEFAULT CURRENT_DATE,
    PRIMARY KEY (aluno_id, curso_id),
    CONSTRAINT fk_matriculas_alunos
        FOREIGN KEY (aluno_id)
        REFERENCES alunos (id)
        ON DELETE CASCADE,
    CONSTRAINT fk_matriculas_cursos
        FOREIGN KEY (curso_id)
        REFERENCES cursos (id)
        ON DELETE CASCADE
);

A tabela matriculas possui uma chave primária composta (aluno_id, curso_id) e duas chaves estrangeiras. Essa estrutura é o padrão para modelar relacionamentos N:N.

Integridade dos Dados

A integridade dos dados garante que as informações armazenadas no banco sejam precisas, consistentes e confiáveis. Existem três categorias principais:

Integridade de Entidade

Garante que cada tabela tenha uma chave primária única e não nula, identificando cada registro de forma exclusiva. Nenhuma linha pode ter PK duplicada ou nula. 


-- Garantido por PRIMARY KEY
CREATE TABLE produtos (
    id    SERIAL PRIMARY KEY,  -- garante unicidade e NOT NULL
    nome  VARCHAR(100) NOT NULL
);

Integridade Referencial

Garante que os relacionamentos entre tabelas sejam válidos. Uma chave estrangeira deve sempre referenciar um registro existente na tabela pai, ou ser NULL (se permitido). 


-- Garantido por FOREIGN KEY
CREATE TABLE pedidos (
    id         SERIAL PRIMARY KEY,
    cliente_id INT NOT NULL,
    CONSTRAINT fk_pedidos_clientes
        FOREIGN KEY (cliente_id)
        REFERENCES clientes (id)
);
-- O PostgreSQL rejeitará INSERT/UPDATE que violem esta regra

Integridade de Domínio

Garante que os valores em cada coluna pertençam ao conjunto de valores permitidos (domínio). É implementada através de restrições como: 


CREATE TABLE funcionarios (
    id       SERIAL PRIMARY KEY,
    nome     VARCHAR(100) NOT NULL,
    idade    INT CHECK (idade >= 18 AND idade <= 100),
    salario  NUMERIC(10, 2) CHECK (salario > 0),
    email    VARCHAR(100) UNIQUE NOT NULL,
    status   VARCHAR(20) DEFAULT 'Ativo'
        CHECK (status IN ('Ativo', 'Inativo', 'Férias')),
    data_admissao DATE NOT NULL
        CHECK (data_admissao <= CURRENT_DATE)
);

As constraints CHECK, NOT NULL, UNIQUE e DEFAULT são os mecanismos que garantem a integridade de domínio.

Normalização

A normalização é o processo de organizar os dados em tabelas para eliminar redundâncias e evitar anomalias. É aplicada em etapas chamadas Formas Normais (FN).

Partiremos de uma tabela não normalizada e aplicaremos cada forma normal passo a passo.

Tabela Não Normalizada

Considere a seguinte tabela que armazena pedidos de uma livraria:


CREATE TABLE pedidos_livraria (
    pedido_id     INT,
    cliente_nome  VARCHAR(100),
    cliente_telefone VARCHAR(20),
    cliente_cidade  VARCHAR(50),
    livro_titulo  VARCHAR(200),
    livro_autor   VARCHAR(100),
    livro_editora VARCHAR(100),
    livro_preco   NUMERIC(10, 2),
    quantidade    INT,
    data_pedido   DATE
);

1FN — Primeira Forma Normal

Regra: Todos os atributos devem ser atômicos (indivisíveis). Não pode haver grupos repetidos ou colunas multivaloradas.

Na tabela acima, todos os valores já são atômicos. Além disso, precisamos garantir que existe uma chave primária. Vamos definir pedido_id como PK e tornar os valores atômicos explícitos:


CREATE TABLE pedidos_livraria_1fn (
    pedido_id       INT,
    cliente_nome    VARCHAR(100) NOT NULL,
    cliente_telefone VARCHAR(20),
    cliente_cidade   VARCHAR(50),
    livro_titulo    VARCHAR(200) NOT NULL,
    livro_autor     VARCHAR(100),
    livro_editora   VARCHAR(100),
    livro_preco     NUMERIC(10, 2) NOT NULL,
    quantidade      INT NOT NULL,
    data_pedido     DATE NOT NULL,
    PRIMARY KEY (pedido_id, livro_titulo)
);

A chave primária composta (pedido_id, livro_titulo) é necessária porque um pedido pode conter vários livros, e um livro pode aparecer em vários pedidos.

2FN — Segunda Forma Normal

Regra: A tabela já deve estar na 1FN e todos os atributos não-chave devem ser totalmente dependentes da chave primária inteira (não apenas de parte dela).

Na nossa tabela 1FN, observe:

  • cliente_nome, cliente_telefone, cliente_cidade dependem apenas de pedido_id (não do livro)
  • data_pedido depende apenas de pedido_id
  • livro_autor, livro_editora, livro_preco dependem apenas de livro_titulo (não do pedido)
  • Apenas quantidade depende da combinação completa (pedido_id, livro_titulo)

Solução: dividir em três tabelas:


-- Tabela de clientes
CREATE TABLE clientes (
    cliente_id   SERIAL PRIMARY KEY,
    nome         VARCHAR(100) NOT NULL,
    telefone     VARCHAR(20),
    cidade       VARCHAR(50)
);

-- Tabela de livros
CREATE TABLE livros (
    livro_id  SERIAL PRIMARY KEY,
    titulo    VARCHAR(200) NOT NULL,
    autor     VARCHAR(100),
    editora   VARCHAR(100),
    preco     NUMERIC(10, 2) NOT NULL
);

-- Tabela de pedidos com itens
CREATE TABLE pedidos (
    pedido_id   INT NOT NULL,
    livro_id    INT NOT NULL,
    quantidade  INT NOT NULL,
    data_pedido DATE NOT NULL,
    PRIMARY KEY (pedido_id, livro_id),
    CONSTRAINT fk_pedidos_livros
        FOREIGN KEY (livro_id)
        REFERENCES livros (livro_id)
);

3FN — Terceira Forma Normal

Regra: A tabela já deve estar na 2FN e não pode haver dependência transitiva. Ou seja, um atributo não-chave não pode depender de outro atributo não-chave.

Na tabela clientes, suponha que adicionamos a coluna estado e codigo_estado. O codigo_estado determina o nome do estado — isso é uma dependência transitiva: cliente_id → codigo_estado → estado.

Solução: mover estado para sua própria tabela:


-- Tabela de estados
CREATE TABLE estados (
    codigo_estado VARCHAR(2) PRIMARY KEY,
    nome_estado   VARCHAR(50) NOT NULL
);

-- Tabela clientes ajustada
CREATE TABLE clientes (
    cliente_id    SERIAL PRIMARY KEY,
    nome          VARCHAR(100) NOT NULL,
    telefone      VARCHAR(20),
    cidade        VARCHAR(50),
    codigo_estado VARCHAR(2),
    CONSTRAINT fk_clientes_estados
        FOREIGN KEY (codigo_estado)
        REFERENCES estados (codigo_estado)
);

Agora, o nome do estado é armazenado apenas uma vez na tabela estados. Se o estado mudar de nome (raro, mas possível), atualizamos em um único lugar.

BCNF — Forma Normal de Boyce-Codd

Regra: Para toda dependência funcional X → Y, X deve ser uma superchave (chave candidata). A BCNF é uma versão mais rigorosa da 3FN.

A BCNF é necessária em casos específicos onde a 3FN não resolve todas as redundâncias. Exemplo: uma tabela de agendamento onde um professor pode estar em apenas uma sala, e uma sala pode ter vários professores em horários diferentes. 


-- Exemplo de tabela que está na 3FN mas não na BCNF
CREATE TABLE agendamentos (
    professor VARCHAR(100),
    sala      VARCHAR(10),
    horario   TIME,
    PRIMARY KEY (professor, horario)
);

-- Dependências:
-- professor, horario → sala   (PK determina sala)
-- sala, horario → professor   (em um horário, sala tem 1 professor)

-- Na BCNF, quebramos em:
CREATE TABLE alocacao_professor (
    professor VARCHAR(100),
    sala      VARCHAR(10),
    PRIMARY KEY (professor)
);

CREATE TABLE alocacao_sala (
    sala      VARCHAR(10),
    horario   TIME,
    professor VARCHAR(100),
    PRIMARY KEY (sala, horario)
);

Resumo das Formas Normais

Forma Normal Requisito Principal Problema que Resolve
1FN Valores atômicos, sem grupos repetidos Colunas multivaloradas
2FN Dependência funcional total da PK Dependência parcial (atributos que dependem de parte da PK composta)
3FN Sem dependências transitivas Atributos que dependem de outros atributos não-chave
BCNF Todo determinante é chave candidata Casos específicos não cobertos pela 3FN

Na prática, a maioria dos bancos de dados bem projetados atende até a 3FN. A BCNF e formas superiores (4FN, 5FN) são aplicadas em cenários mais específicos.

Consultas Multitabulares

Quando os dados estão distribuídos em várias tabelas, precisamos de mecanismos para combiná-los em uma única consulta. A forma mais fundamental é o produto cartesiano.

Produto Cartesiano

O produto cartesiano combina todas as linhas da tabela A com todas as linhas da tabela B. Se a tabela A tem 5 linhas e a tabela B tem 10 linhas, o resultado terá 5 × 10 = 50 linhas. 


-- Produto cartesiano: todas as combinações
SELECT * FROM clientes, pedidos;

O resultado bruto do produto cartesiano raramente é útil. Precisamos filtrar para manter apenas as combinações que fazem sentido: 


-- Filtrando o produto cartesiano com WHERE
SELECT
    clientes.nome AS cliente,
    pedidos.id AS pedido_id,
    pedidos.data_pedido
FROM clientes, pedidos
WHERE clientes.id = pedidos.cliente_id;

Esta é a forma tradicional de fazer consultas multitabulares. A condição clientes.id = pedidos.cliente_id na cláusula WHERE conecta as duas tabelas pela chave estrangeira, produzindo o que chamamos de junção implícita (implicit join).

Exemplo Completo com Três Tabelas

Vamos criar e popular tabelas relacionadas para demonstrar consultas multitabulares:


-- Criar as tabelas
CREATE TABLE clientes (
    id    SERIAL PRIMARY KEY,
    nome  VARCHAR(100) NOT NULL,
    cidade VARCHAR(50)
);

CREATE TABLE pedidos (
    id         SERIAL PRIMARY KEY,
    cliente_id INT NOT NULL,
    data_pedido DATE DEFAULT CURRENT_DATE,
    CONSTRAINT fk_pedidos_clientes
        FOREIGN KEY (cliente_id)
        REFERENCES clientes (id)
);

CREATE TABLE itens_pedido (
    pedido_id  INT NOT NULL,
    produto    VARCHAR(100) NOT NULL,
    quantidade INT NOT NULL,
    preco_unit NUMERIC(10, 2) NOT NULL,
    PRIMARY KEY (pedido_id, produto),
    CONSTRAINT fk_itens_pedidos
        FOREIGN KEY (pedido_id)
        REFERENCES pedidos (id)
        ON DELETE CASCADE
);

-- Inserir dados
INSERT INTO clientes (nome, cidade) VALUES
    ('Ana Costa', 'São Paulo'),
    ('Bruno Lima', 'Campinas'),
    ('Carla Mendes', 'São Paulo');

INSERT INTO pedidos (cliente_id, data_pedido) VALUES
    (1, '2025-06-01'),
    (1, '2025-06-10'),
    (2, '2025-06-05'),
    (3, '2025-06-12');

INSERT INTO itens_pedido (pedido_id, produto, quantidade, preco_unit) VALUES
    (1, 'Notebook', 1, 3500.00),
    (1, 'Mouse',    2,  150.00),
    (2, 'Teclado',  1,  250.00),
    (3, 'Monitor',  1, 1200.00),
    (4, 'Notebook', 1, 3500.00),
    (4, 'Webcam',   1,  300.00);

Consulta unindo as três tabelas:


-- Consulta com 3 tabelas usando produto cartesiano filtrado
SELECT
    c.nome        AS cliente,
    p.id          AS pedido,
    p.data_pedido AS data,
    i.produto,
    i.quantidade,
    i.preco_unit,
    (i.quantidade * i.preco_unit) AS subtotal
FROM clientes c, pedidos p, itens_pedido i
WHERE c.id = p.cliente_id
  AND p.id = i.pedido_id
ORDER BY p.id, i.produto;

Consulta com filtro adicional — apenas clientes de São Paulo:


SELECT
    c.nome AS cliente,
    COUNT(p.id) AS total_pedidos
FROM clientes c, pedidos p
WHERE c.id = p.cliente_id
  AND c.cidade = 'São Paulo'
GROUP BY c.nome;

Dependências funcionais identificadas:

  • venda_id → data_venda
  • venda_id → cliente_nome, cliente_cpf
  • venda_id → vendedor_nome, vendedor_setor
  • cliente_cpf → cliente_nome
  • vendedor_nome → vendedor_setor
  • produto_nome → produto_preco
  • (venda_id, produto_nome) → quantidade

1FN: Todos valores atômicos. PK composta: (venda_id, produto_nome).

2FN: Separar atributos que dependem parcialmente da PK:


CREATE TABLE clientes (
    cliente_id SERIAL PRIMARY KEY,
    nome       VARCHAR(100) NOT NULL,
    cpf        VARCHAR(14) UNIQUE NOT NULL
);

CREATE TABLE vendedores (
    vendedor_id SERIAL PRIMARY KEY,
    nome        VARCHAR(100) NOT NULL,
    setor       VARCHAR(50) NOT NULL
);

CREATE TABLE produtos (
    produto_id SERIAL PRIMARY KEY,
    nome       VARCHAR(100) NOT NULL,
    preco      NUMERIC(10, 2) NOT NULL
);

CREATE TABLE vendas (
    venda_id    SERIAL PRIMARY KEY,
    cliente_id  INT NOT NULL,
    vendedor_id INT NOT NULL,
    data_venda  DATE NOT NULL,
    CONSTRAINT fk_vendas_clientes
        FOREIGN KEY (cliente_id) REFERENCES clientes (cliente_id),
    CONSTRAINT fk_vendas_vendedores
        FOREIGN KEY (vendedor_id) REFERENCES vendedores (vendedor_id)
);

CREATE TABLE itens_venda (
    venda_id   INT NOT NULL,
    produto_id INT NOT NULL,
    quantidade INT NOT NULL,
    PRIMARY KEY (venda_id, produto_id),
    CONSTRAINT fk_itens_vendas
        FOREIGN KEY (venda_id) REFERENCES vendas (venda_id)
        ON DELETE CASCADE,
    CONSTRAINT fk_itens_produtos
        FOREIGN KEY (produto_id) REFERENCES produtos (produto_id)
);

3FN: Verificar dependências transitivas:

  • Na tabela clientes: cpf (não-chave) → nome. Como cpf é UNIQUE (chave candidata), está OK. Se não fosse, teríamos que criar tabela separada.
  • Na tabela vendedores: nome (não-chave) → setor. Se nome não for UNIQUE, há dependência transitiva. Solução: manter setor na tabela vendedores se cada vendedor tem um único setor; caso contrário, normalizar.

O esquema acima já atende à 3FN, pois não há dependências transitivas entre atributos não-chave nas tabelas criadas.
