Módulo 7 (Parte 1): Views e Triggers

Tópicos do nosso plano de curso abordados nessa aula: Criação de views, Modificação de views (CREATE OR REPLACE), Exclusão de views, Alteração de dados através de views, O que é um trigger, Tipos de triggers (BEFORE, AFTER, INSTEAD OF), Trigger INSERT, Trigger UPDATE, Trigger DELETE.

O que são Views?

Uma view (visão) é uma tabela virtual baseada no resultado de uma consulta SQL. Ela não armazena dados fisicamente — sempre que você consulta uma view, o SGBD executa a query que a define e retorna os dados atualizados das tabelas base.

As principais vantagens de usar views são:

  • Segurança: você pode expor apenas colunas específicas, ocultando dados sensíveis como senhas ou CPF
  • Simplicidade: consultas complexas com múltiplos JOINs ficam encapsuladas em uma view simples de consultar
  • Reusabilidade: a lógica da consulta é definida uma única vez e reutilizada em vários lugares
  • Abstração: mudanças na estrutura das tabelas base podem ser transparentes para as aplicações que usam a view
  • Consistência: todos os usuários e aplicações utilizam a mesma definição da consulta

Pense nas views como "atalhos nomeados" para consultas frequentes. Em vez de reescrever um SELECT complexo toda vez, você cria a view uma vez e faz SELECT * FROM view.

Criando Views

A sintaxe básica para criar uma view é:

View Simples

Uma view que filtra clientes ativos de uma tabela:


CREATE VIEW clientes_ativos AS
SELECT * FROM clientes
WHERE status = 'ativo';

Agora você pode consultar a view como se fosse uma tabela normal:


SELECT * FROM clientes_ativos;

View com JOINs

Uma view que consolida dados de múltiplas tabelas com INNER JOIN:


CREATE VIEW resumo_pedidos AS
SELECT
    c.nome AS cliente,
    p.data_pedido,
    p.valor_total,
    p.status
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id;

Consultar pedidos de um cliente específico agora fica trivial:


SELECT * FROM resumo_pedidos
WHERE cliente = 'João Silva'
ORDER BY data_pedido DESC;

View com Agregações

Views também podem conter funções de agregação como COUNT, SUM, AVG, MAX e MIN:


CREATE VIEW vendas_por_categoria AS
SELECT
    categoria,
    COUNT(*) AS total_produtos,
    SUM(valor) AS receita_total,
    AVG(valor) AS preco_medio,
    MAX(valor) AS produto_mais_caro,
    MIN(valor) AS produto_mais_barato
FROM produtos
GROUP BY categoria;

Consultar quais categorias geram mais receita:


SELECT categoria, receita_total
FROM vendas_por_categoria
ORDER BY receita_total DESC;

Criando Views a partir de outras Views

É possível criar views que referenciam outras views, construindo camadas de abstração:


CREATE VIEW top_categorias AS
SELECT categoria, receita_total
FROM vendas_por_categoria
WHERE receita_total > 5000;

Modificando Views

Para alterar a definição de uma view existente sem precisar excluí-la e recriá-la, utilizamos o comando CREATE OR REPLACE VIEW. Isso preserva as permissões já concedidas sobre a view. 


-- View original
CREATE VIEW clientes_ativos AS
SELECT * FROM clientes
WHERE status = 'ativo';

-- Substituindo a view para incluir apenas colunas específicas
CREATE OR REPLACE VIEW clientes_ativos AS
SELECT id, nome, email, cidade
FROM clientes
WHERE status = 'ativo';

Importante: O PostgreSQL também oferece ALTER VIEW para alterar propriedades auxiliares da view (como renomear, alterar dono, definir opções), mas para alterar a query da view usa-se CREATE OR REPLACE VIEW. 


-- Renomear uma view
ALTER VIEW clientes_ativos RENAME TO clientes_ativos_v2;

-- Alterar o dono da view
ALTER VIEW clientes_ativos_v2 OWNER TO outro_usuario;

Excluindo Views

Para remover uma view do banco de dados, utiliza-se o comando DROP VIEW. O uso de IF EXISTS evita erros caso a view não exista. 


-- Excluir uma view
DROP VIEW clientes_ativos;

-- Excluir com verificação de existência (recomendado)
DROP VIEW IF EXISTS clientes_ativos;

-- Excluir múltiplas views de uma vez
DROP VIEW IF EXISTS view1, view2, view3;

Atenção: Excluir uma view não afeta os dados das tabelas base. Apenas a definição da consulta virtual é removida. Views que dependem de outras views também não são excluídas automaticamente — elas se tornarão inválidas.

Para verificar quais views dependem de outras, use:


SELECT
    dependent_ns.nspname AS dependent_schema,
    dependent_view.relname AS dependent_view,
    source_ns.nspname AS source_schema,
    source_view.relname AS source_view
FROM pg_depend
JOIN pg_rewrite ON pg_depend.objid = pg_rewrite.oid
JOIN pg_class AS dependent_view ON pg_rewrite.ev_class = dependent_view.oid
JOIN pg_class AS source_view ON pg_depend.refobjid = source_view.oid
JOIN pg_namespace AS dependent_ns ON dependent_ns.oid = dependent_view.relnamespace
JOIN pg_namespace AS source_ns ON source_ns.oid = source_view.relnamespace
WHERE dependent_view.relkind = 'v';

Alterando Dados através de Views

Em muitos casos, é possível executar INSERT, UPDATE e DELETE diretamente através de uma view. No PostgreSQL, uma view é automaticamente atualizável se atender a estas condições:

  • A view é baseada em exatamente uma tabela (ou view atualizável)
  • Todas as colunas da view referenciam diretamente colunas da tabela base (sem expressões, funções ou agregações)
  • A query da view não contém: DISTINCT, GROUP BY, HAVING, LIMIT, OFFSET, UNION, INTERSECT, EXCEPT
  • Não há funções de janela (WINDOW), nem funções de agregação no SELECT

Inserindo dados via View


CREATE VIEW clientes_simples AS
SELECT id, nome, email
FROM clientes;

INSERT INTO clientes_simples (nome, email)
VALUES ('Maria Souza', 'maria@email.com');

Atualizando dados via View


UPDATE clientes_simples
SET email = 'maria.nova@email.com'
WHERE id = 1;

Excluindo dados via View


DELETE FROM clientes_simples
WHERE id = 1;

WITH CHECK OPTION

Ao criar uma view com WITH CHECK OPTION, você garante que qualquer INSERT ou UPDATE feito através da view respeite a condição WHERE da view. Isso impede que dados inseridos pela view "desapareçam" da visualização. 


CREATE VIEW produtos_caros AS
SELECT * FROM produtos
WHERE preco > 1000
WITH CHECK OPTION;

-- Isso funciona (preço > 1000)
INSERT INTO produtos_caros (nome, preco)
VALUES ('Notebook Gamer', 5000);

-- Isso falha (preço não é > 1000)
INSERT INTO produtos_caros (nome, preco)
VALUES ('Mouse', 50);
-- ERRO: new row violates check option for view "produtos_caros"

O WITH CASCADED CHECK OPTION (padrão) propaga a verificação para views subjacentes. O WITH LOCAL CHECK OPTION verifica apenas a view atual.

O que são Triggers?

Um trigger (gatilho) é um procedimento armazenado que é executado automaticamente pelo SGBD quando um evento específico (INSERT, UPDATE ou DELETE) ocorre em uma tabela.

Triggers são fundamentais para:

  • Auditoria: registrar automaticamente quem fez qual alteração e quando
  • Validação: impor regras de negócio complexas que vão além de constraints
  • Integridade: manter dados derivados ou desnormalizados sincronizados
  • Log: manter histórico de alterações em tabelas críticas
  • Enforcement: aplicar políticas de segurança ou regras de acesso

A analogia: Pense em um trigger como um "alarme de carro" — você não precisa ligá-lo manualmente; ele dispara automaticamente quando alguém tenta abrir a porta (evento).

Estrutura de um Trigger

No PostgreSQL, os triggers funcionam em duas partes:

  1. Função Trigger: uma função escrita em PL/pgSQL (ou outra linguagem procedural) que contém a lógica a ser executada
  2. Trigger propriamente dito: que associa a função a uma tabela e define quando ela será disparada

Este é o template básico de uma função trigger e seu respectivo trigger:


-- 1. Criar a função que o trigger executará
CREATE OR REPLACE FUNCTION nome_da_funcao()
RETURNS TRIGGER AS $$
BEGIN
    -- Lógica do trigger aqui
    -- NEW representa a nova linha (INSERT/UPDATE)
    -- OLD representa a linha antiga (UPDATE/DELETE)
    -- TG_OP contém a operação: 'INSERT', 'UPDATE' ou 'DELETE'
    RETURN NEW;  -- ou RETURN OLD para DELETE
END;
$$ LANGUAGE plpgsql;

-- 2. Criar o trigger que chama a função
CREATE TRIGGER nome_do_trigger
BEFORE INSERT ON tabela
FOR EACH ROW
EXECUTE FUNCTION nome_da_funcao();

Variáveis especiais disponíveis em funções trigger:

Variável Significado Disponível em
NEW Registro da nova linha (após a operação) INSERT, UPDATE
OLD Registro da linha antiga (antes da operação) UPDATE, DELETE
TG_OP Operação que disparou o trigger ('INSERT', 'UPDATE', 'DELETE', 'TRUNCATE') Todos
TG_TABLE_NAME Nome da tabela onde o trigger foi disparado Todos
TG_WHEN Quando o trigger é disparado ('BEFORE', 'AFTER', 'INSTEAD OF') Todos
TG_LEVEL 'ROW' para FOR EACH ROW, 'STATEMENT' para FOR EACH STATEMENT Todos

Tipos de Triggers

Os triggers podem ser classificados pelo momento de execução e pelo evento disparador:

BEFORE INSERT — Validar dados antes da inserção

Impede a inserção de um produto com preço negativo:


CREATE OR REPLACE FUNCTION validar_preco_produto()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.preco < 0 THEN
        RAISE EXCEPTION 'O preço do produto não pode ser negativo: %', NEW.preco;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_valida_preco
BEFORE INSERT ON produtos
FOR EACH ROW
EXECUTE FUNCTION validar_preco_produto();

AFTER INSERT — Registrar log de auditoria

Registra toda inserção em uma tabela de log:


CREATE OR REPLACE FUNCTION log_insert_produto()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO log_operacoes (tabela, operacao, registro_id, data_hora)
    VALUES ('produtos', 'INSERT', NEW.id, NOW());
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_log_insert
AFTER INSERT ON produtos
FOR EACH ROW
EXECUTE FUNCTION log_insert_produto();

BEFORE UPDATE — Atualizar campo de data automaticamente

Atualiza o campo updated_at sempre que uma linha é modificada:


CREATE OR REPLACE FUNCTION atualizar_timestamp()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_update_timestamp
BEFORE UPDATE ON produtos
FOR EACH ROW
EXECUTE FUNCTION atualizar_timestamp();

AFTER UPDATE — Registrar alterações

Registra em log quais campos foram alterados:


CREATE OR REPLACE FUNCTION log_update_produto()
RETURNS TRIGGER AS $$
BEGIN
    IF OLD.preco <> NEW.preco THEN
        INSERT INTO log_alteracoes (tabela, registro_id, campo, valor_antigo, valor_novo, data_hora)
        VALUES ('produtos', NEW.id, 'preco', OLD.preco::TEXT, NEW.preco::TEXT, NOW());
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_log_update
AFTER UPDATE ON produtos
FOR EACH ROW
EXECUTE FUNCTION log_update_produto();

BEFORE DELETE — Impedir exclusão com condições

Impede a exclusão de clientes que ainda possuem pedidos:


CREATE OR REPLACE FUNCTION impedir_exclusao_cliente()
RETURNS TRIGGER AS $$
DECLARE
    pedidos_count INTEGER;
BEGIN
    SELECT COUNT(*) INTO pedidos_count
    FROM pedidos
    WHERE cliente_id = OLD.id;

    IF pedidos_count > 0 THEN
        RAISE EXCEPTION 'Não é possível excluir o cliente ID % pois ele possui % pedido(s) ativo(s).', OLD.id, pedidos_count;
    END IF;
    RETURN OLD;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_impede_delete_cliente
BEFORE DELETE ON clientes
FOR EACH ROW
EXECUTE FUNCTION impedir_exclusao_cliente();

AFTER DELETE — Limpeza de dados relacionados

Remove automaticamente registros dependentes:


CREATE OR REPLACE FUNCTION limpar_dependentes()
RETURNS TRIGGER AS $$
BEGIN
    DELETE FROM telefones WHERE cliente_id = OLD.id;
    DELETE FROM enderecos WHERE cliente_id = OLD.id;
    RETURN OLD;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_after_delete_cliente
AFTER DELETE ON clientes
FOR EACH ROW
EXECUTE FUNCTION limpar_dependentes();

INSTEAD OF — Para Views não atualizáveis

Permite INSERT, UPDATE e DELETE em views complexas que não seriam naturalmente atualizáveis:


-- View que junta clientes e pedidos (não é naturalmente atualizável)
CREATE VIEW clientes_com_pedidos AS
SELECT
    c.id AS cliente_id,
    c.nome,
    p.id AS pedido_id,
    p.valor_total
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id;

-- Função que trata INSERT na view
CREATE OR REPLACE FUNCTION insere_cliente_view()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO clientes (nome) VALUES (NEW.nome);
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_instead_of_insert
INSTEAD OF INSERT ON clientes_com_pedidos
FOR EACH ROW
EXECUTE FUNCTION insere_cliente_view();

Passo a passo: Criando um sistema de auditoria com triggers

Vamos construir um sistema completo de auditoria que registra automaticamente todas as operações (INSERT, UPDATE, DELETE) na tabela clientes.

Passo 1 — Criar a tabela de auditoria


CREATE TABLE auditoria (
    id               SERIAL PRIMARY KEY,
    tabela           VARCHAR(100) NOT NULL,
    operacao         VARCHAR(10) NOT NULL,
    registro_id      INTEGER,
    dados_antigos    JSONB,
    dados_novos      JSONB,
    usuario          VARCHAR(100) DEFAULT CURRENT_USER,
    data_hora        TIMESTAMP DEFAULT NOW()
);

Passo 2 — Criar a tabela de clientes de exemplo


CREATE TABLE clientes (
    id     SERIAL PRIMARY KEY,
    nome   VARCHAR(100) NOT NULL,
    email  VARCHAR(150),
    status VARCHAR(20) DEFAULT 'ativo'
);

Passo 3 — Criar a função de auditoria


CREATE OR REPLACE FUNCTION fn_auditoria()
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        INSERT INTO auditoria (tabela, operacao, registro_id, dados_novos)
        VALUES (TG_TABLE_NAME, 'INSERT', NEW.id, row_to_json(NEW)::JSONB);

    ELSIF TG_OP = 'UPDATE' THEN
        INSERT INTO auditoria (tabela, operacao, registro_id, dados_antigos, dados_novos)
        VALUES (TG_TABLE_NAME, 'UPDATE', NEW.id, row_to_json(OLD)::JSONB, row_to_json(NEW)::JSONB);

    ELSIF TG_OP = 'DELETE' THEN
        INSERT INTO auditoria (tabela, operacao, registro_id, dados_antigos)
        VALUES (TG_TABLE_NAME, 'DELETE', OLD.id, row_to_json(OLD)::JSONB);
    END IF;

    RETURN NULL;
END;
$$ LANGUAGE plpgsql;

Passo 4 — Criar os triggers na tabela clientes


CREATE TRIGGER trg_auditoria_insert
AFTER INSERT ON clientes
FOR EACH ROW
EXECUTE FUNCTION fn_auditoria();

CREATE TRIGGER trg_auditoria_update
AFTER UPDATE ON clientes
FOR EACH ROW
EXECUTE FUNCTION fn_auditoria();

CREATE TRIGGER trg_auditoria_delete
AFTER DELETE ON clientes
FOR EACH ROW
EXECUTE FUNCTION fn_auditoria();

Passo 5 — Testar o sistema

Execute operações e verifique os registros de auditoria:


-- Inserir clientes
INSERT INTO clientes (nome, email) VALUES ('João Silva', 'joao@email.com');
INSERT INTO clientes (nome, email) VALUES ('Maria Souza', 'maria@email.com');

-- Atualizar um cliente
UPDATE clientes SET email = 'joao.novo@email.com' WHERE id = 1;

-- Excluir um cliente
DELETE FROM clientes WHERE id = 2;

-- Verificar os registros de auditoria
SELECT * FROM auditoria ORDER BY data_hora DESC;

Passo 6 — Listar todos os triggers do banco


SELECT
    tgname AS trigger_name,
    relname AS table_name,
    CASE WHEN tgfoid IS NOT NULL THEN proname END AS function_name,
    CASE
        WHEN (t.tgtype::int & 2) = 2 AND (t.tgtype::int & 1) = 0 THEN 'BEFORE'
        WHEN (t.tgtype::int & 2) = 2 AND (t.tgtype::int & 1) = 1 THEN 'AFTER'
        WHEN (t.tgtype::int & 2) = 0 THEN 'INSTEAD OF'
    END AS when_trigger
FROM pg_trigger t
JOIN pg_class c ON t.tgrelid = c.oid
LEFT JOIN pg_proc p ON t.tgfoid = p.oid
WHERE NOT t.tgisinternal
ORDER BY relname, tgname;

Dever de Casa

1. Crie uma view que mostre clientes e seus totais de pedidos

A view deve exibir o nome do cliente, quantidade de pedidos e valor total gasto. Considere as tabelas clientes (id, nome) e pedidos (id, cliente_id, valor).

Gabarito: 

CREATE VIEW clientes_total_pedidos AS
SELECT
    c.nome,
    COUNT(p.id) AS total_pedidos,
    COALESCE(SUM(p.valor), 0) AS valor_total_gasto
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
GROUP BY c.id, c.nome
ORDER BY valor_total_gasto DESC;

O LEFT JOIN garante que clientes sem pedidos também apareçam na view. O COALESCE converte NULL em 0 para o valor total.

2. Crie um trigger BEFORE INSERT que impede a inserção de produto com preço negativo

Considere a tabela produtos (id, nome, preco, estoque). O trigger deve lançar uma exceção com mensagem descritiva quando o preço for menor que zero.

Gabarito: 

CREATE OR REPLACE FUNCTION validar_preco_produto()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.preco < 0 THEN
        RAISE EXCEPTION 'Erro: O preço do produto "%" não pode ser negativo (R$ %).', NEW.nome, NEW.preco;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_valida_preco_produto
BEFORE INSERT ON produtos
FOR EACH ROW
EXECUTE FUNCTION validar_preco_produto();

O trigger também pode ser estendido para UPDATE com BEFORE INSERT OR UPDATE para validar alterações de preço.

3. Crie um trigger AFTER UPDATE que registra em uma tabela de log as alterações de preço dos produtos

Crie a tabela log_precos e o trigger que registra: id do produto, preço antigo, preço novo, data da alteração e usuário que realizou a mudança.

Gabarito: 

-- Passo 1: Criar a tabela de log
CREATE TABLE log_precos (
    id           SERIAL PRIMARY KEY,
    produto_id   INTEGER NOT NULL,
    preco_antigo NUMERIC(10,2),
    preco_novo   NUMERIC(10,2),
    usuario      VARCHAR(100) DEFAULT CURRENT_USER,
    data_alteracao TIMESTAMP DEFAULT NOW()
);

-- Passo 2: Criar a função trigger
CREATE OR REPLACE FUNCTION registrar_alteracao_preco()
RETURNS TRIGGER AS $$
BEGIN
    IF OLD.preco <> NEW.preco THEN
        INSERT INTO log_precos (produto_id, preco_antigo, preco_novo)
        VALUES (NEW.id, OLD.preco, NEW.preco);
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Passo 3: Criar o trigger
CREATE TRIGGER trg_log_alteracao_preco
AFTER UPDATE ON produtos
FOR EACH ROW
EXECUTE FUNCTION registrar_alteracao_preco();

A condição IF OLD.preco <> NEW.preco garante que só registramos quando o preço realmente mudou, evitando logs desnecessários.
