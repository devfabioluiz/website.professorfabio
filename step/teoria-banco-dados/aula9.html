Modelo de Segurança no PostgreSQL

O PostgreSQL implementa um modelo de segurança robusto baseado em Roles (papéis), Privileges (privilégios) e Object Ownership (propriedade de objetos). Entender esse modelo é fundamental para administrar bancos de dados de forma segura.

Conceitos Fundamentais

  • Roles: entidades que podem representar usuários individuais ou grupos de usuários. No PostgreSQL, não há distinção conceitual entre "usuário" e "grupo" — ambos são roles.
  • Privilégios: permissões específicas que determinam o que uma role pode fazer (SELECT, INSERT, UPDATE, DELETE, CONNECT, USAGE, EXECUTE, etc.).
  • Object Ownership: o criador de um objeto (tabela, schema, banco de dados) é o seu proprietário e tem todos os privilégios sobre ele por padrão. O proprietário pode conceder privilégios a outras roles.
  • Herança: uma role pode herdar privilégios de outras roles das quais é membro, simplificando o gerenciamento de permissões.

Métodos de Autenticação

O PostgreSQL suporta diversos métodos de autenticação para controlar como os usuários se conectam ao banco:

Método Descrição Uso Típico
password Senha enviada em texto claro Apenas em conexões locais seguras (não recomendado em produção)
md5 Senha enviada com hash MD5 (obsoleto, evitar) Compatibilidade com versões antigas
scram-sha-256 Mecanismo moderno de desafio-resposta com SHA-256 (recomendado) Conexões remotas e produção
peer Mapeia usuário do SO para role do PostgreSQL (Linux) Conexões locais no Linux
trust Sem autenticação — qualquer um se conecta (perigoso!) Apenas desenvolvimento local isolado
cert Autenticação via certificado SSL/TLS Ambientes corporativos com PKI

Esses métodos são configurados no arquivo pg_hba.conf, localizado no diretório de dados do PostgreSQL.

Gerenciando Usuários (Roles)

No PostgreSQL, todo acesso ao banco é feito através de uma role. Vamos aprender a criar, modificar e remover roles.

CREATE ROLE

O comando CREATE ROLE cria uma nova role. Por padrão, uma role criada com CREATE ROLE não tem permissão de login. Para permitir login, usamos o atributo LOGIN. 


-- Cria uma role com permissão de login e senha
CREATE ROLE estagiario WITH LOGIN PASSWORD 'senha123';

-- Cria uma role sem permissão de login (será usada como grupo)
CREATE ROLE desenvolvedores;

CREATE USER

CREATE USER é um alias para CREATE ROLE com o atributo LOGIN implícito. Ambos produzem o mesmo resultado, mas CREATE USER é mais intuitivo para criar usuários que farão login. 


-- Equivalente a: CREATE ROLE joao WITH LOGIN PASSWORD 'senha456';
CREATE USER joao WITH PASSWORD 'senha456';

-- Criando múltiplos usuários
CREATE USER maria WITH PASSWORD 'senha789';
CREATE USER carlos WITH PASSWORD 'senha321' CREATEDB;

ALTER ROLE

O comando ALTER ROLE modifica atributos de uma role existente. 


-- Conceder privilégios de superusuário
ALTER ROLE joao WITH SUPERUSER;

-- Definir validade da senha
ALTER ROLE joao VALID UNTIL '2025-12-31';

-- Renomear uma role
ALTER ROLE estagiario RENAME TO estagiario_antigo;

-- Alterar senha
ALTER ROLE maria WITH PASSWORD 'nova_senha_segura';

-- Remover privilégio de superusuário
ALTER ROLE joao WITH NOSUPERUSER;

-- Conceder permissão para criar bancos de dados
ALTER ROLE carlos WITH CREATEDB;

DROP ROLE

O comando DROP ROLE remove uma role do sistema. Use IF EXISTS para evitar erros caso a role não exista. 


-- Remove uma role (se não existir, gera erro)
DROP ROLE estagiario_antigo;

-- Remove uma role com segurança (não gera erro se não existir)
DROP ROLE IF EXISTS estagiario;

Atributos de Role

Ao criar ou alterar uma role, podemos definir diversos atributos que controlam suas capacidades:

Atributo Descrição
LOGIN / NOLOGIN Permite ou nega a capacidade de se conectar ao banco de dados
SUPERUSER / NOSUPERUSER Superusuário ignora todas as verificações de permissão (use com extrema cautela)
CREATEDB / NOCREATEDB Permite ou nega a criação de bancos de dados
CREATEROLE / NOCREATEROLE Permite ou nega a criação, alteração e remoção de outras roles
REPLICATION Permite iniciar replicação de streaming e backups físicos
INHERIT / NOINHERIT Determina se a role herda automaticamente privilégios das roles das quais é membro
BYPASSRLS / NOBYPASSRLS Ignora políticas de Row Level Security (RLS)
PASSWORD Define a senha da role (apenas para roles com LOGIN)
VALID UNTIL Define uma data de expiração para a senha da role

Roles e Herança de Privilégios

Uma das features mais poderosas do PostgreSQL é a capacidade de usar roles como grupos. Você cria uma role sem permissão de login, concede privilégios a ela, e depois adiciona usuários como membros dessa role. Os membros herdam automaticamente os privilégios do grupo.

Criando uma estrutura de grupo


-- 1. Cria uma role que funcionará como grupo (sem LOGIN)
CREATE ROLE desenvolvedores;

-- 2. Cria usuários individuais (com LOGIN)
CREATE USER joao WITH PASSWORD 'senha456';
CREATE USER maria WITH PASSWORD 'senha789';

-- 3. Adiciona usuários como membros do grupo
GRANT desenvolvedores TO joao;
GRANT desenvolvedores TO maria;

Verificando membros de uma role


-- Lista todas as roles do sistema
\du

-- Consulta via SQL os membros de roles
SELECT
    r.rolname AS role,
    array_agg(m.rolname) AS membros
FROM pg_roles r
LEFT JOIN pg_auth_members am ON r.oid = am.roleid
LEFT JOIN pg_roles m ON am.member = m.oid
WHERE r.rolcanlogin = false
GROUP BY r.rolname;

Ativando uma role

Para assumir temporariamente uma role da qual você é membro (e usar seus privilégios), use SET ROLE: 


-- Assume a role desenvolvedores (usando seus privilégios)
SET ROLE desenvolvedores;

-- Volta à role original
RESET ROLE;

Herança INHERIT vs NOINHERIT

Por padrão, o atributo INHERIT está ativado e os privilégios de grupos são herdados automaticamente. Com NOINHERIT, o usuário precisa usar SET ROLE explicitamente para acessar os privilégios do grupo. 


-- Usuário que NÃO herda automaticamente os privilégios do grupo
CREATE USER pedro WITH PASSWORD 'senha000' NOINHERIT;
GRANT desenvolvedores TO pedro;
-- pedro precisa usar SET ROLE desenvolvedores para ter os privilégios

Comando GRANT — Concedendo Privilégios

O comando GRANT concede privilégios específicos a roles. Os privilégios podem ser concedidos em diferentes níveis: banco de dados, schema, tabela, coluna, sequência e função.

Nível de Banco de Dados


-- Permite que a role se conecte ao banco de dados
GRANT CONNECT ON DATABASE meubanco TO desenvolvedores;

-- Permite criar schemas dentro do banco
GRANT CREATE ON DATABASE meubanco TO desenvolvedores;

Nível de Schema


-- Permite acessar objetos dentro do schema
GRANT USAGE ON SCHEMA public TO desenvolvedores;

-- Permite criar objetos (tabelas, views, funções) no schema
GRANT CREATE ON SCHEMA public TO desenvolvedores;

Nível de Tabela


-- Concede privilégios específicos em uma tabela
GRANT SELECT, INSERT, UPDATE ON clientes TO desenvolvedores;
GRANT ALL PRIVILEGES ON produtos TO admin;

-- Concede DELETE em todas as tabelas do schema
GRANT DELETE ON ALL TABLES IN SCHEMA public TO gerentes;

-- Privilégios em tabelas que serão criadas no futuro
ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON TABLES TO leitura;

Nível de Coluna

É possível conceder privilégios em colunas específicas de uma tabela, permitindo um controle de acesso muito granular: 


-- Permite SELECT apenas nas colunas nome e email da tabela clientes
GRANT SELECT (nome, email) ON clientes TO estagiario;

-- Permite UPDATE apenas na coluna telefone
GRANT UPDATE (telefone) ON clientes TO estagiario;

Nível de Sequência


-- Permite usar a sequência (necessário para INSERT com SERIAL)
GRANT USAGE ON SEQUENCE produtos_id_seq TO dev;

-- Concede em todas as sequências do schema
GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO dev;

Nível de Função


-- Permite executar uma função
GRANT EXECUTE ON FUNCTION calcular_desconto(numeric) TO dev;

-- Concede em todas as funções do schema
GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA public TO dev;
Resumo dos Privilégios Disponíveis:
  • SELECT: permite consultar dados (SELECT)
  • INSERT: permite inserir novos registros
  • UPDATE: permite modificar registros existentes
  • DELETE: permite excluir registros
  • TRUNCATE: permite esvaziar a tabela completamente
  • REFERENCES: permite criar chaves estrangeiras referenciando a tabela
  • TRIGGER: permite criar triggers na tabela
  • CONNECT: permite conectar-se ao banco de dados
  • CREATE: permite criar objetos (schemas, tabelas, etc.)
  • USAGE: permite usar schemas, sequências e tipos
  • EXECUTE: permite executar funções e procedures
  • ALL PRIVILEGES: concede todos os privilégios aplicáveis ao objeto

Comando REVOKE — Revogando Privilégios

O comando REVOKE remove privilégios previamente concedidos com GRANT. É uma prática de segurança essencial, especialmente ao remover usuários ou alterar funções. 


-- Revoga um privilégio específico
REVOKE INSERT ON clientes FROM estagiario;

-- Revoga todos os privilégios em todas as tabelas do schema
REVOKE ALL ON ALL TABLES IN SCHEMA public FROM ex_funcionario;

-- Revoga acesso ao banco de dados
REVOKE CONNECT ON DATABASE meubanco FROM ex_funcionario;

-- Revoga privilégio em uma coluna específica
REVOKE SELECT (email) ON clientes FROM estagiario;

-- Revoga privilégio de executar uma função
REVOKE EXECUTE ON FUNCTION calcular_desconto(numeric) FROM estagiario;
Atenção: Revogar privilégios de um proprietário de objeto não surte efeito — o proprietário sempre mantém todos os privilégios sobre seus objetos. Para remover completamente o acesso, é necessário transferir a propriedade (ALTER TABLE ... OWNER TO ...) ou excluir o objeto.

Passo a passo: Configurando Segurança em um Banco

Vamos configurar um sistema de segurança completo para um banco de dados, criando uma hierarquia de roles com diferentes níveis de acesso.

Passo 1 — Criar as roles


-- Role de administração (acesso total)
CREATE ROLE admin WITH LOGIN PASSWORD 'admin_senha_segura' SUPERUSER;

-- Role de desenvolvimento (DML completo)
CREATE ROLE dev WITH LOGIN PASSWORD 'dev_senha_segura';

-- Role de leitura (apenas SELECT)
CREATE ROLE leitura WITH LOGIN PASSWORD 'leitura_senha_segura';

-- Role de aplicação (INSERT, UPDATE, SELECT, sem DELETE)
CREATE ROLE app WITH LOGIN PASSWORD 'app_senha_segura';

Passo 2 — Conceder acesso ao banco e schema


-- Todos precisam de CONNECT ao banco
GRANT CONNECT ON DATABASE meubanco TO dev, leitura, app;

-- Todos precisam de USAGE no schema public
GRANT USAGE ON SCHEMA public TO dev, leitura, app;

Passo 3 — Conceder privilégios nas tabelas


-- Dev: acesso completo a dados (DML)
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO dev;

-- Leitura: apenas consultas
GRANT SELECT ON ALL TABLES IN SCHEMA public TO leitura;

-- App: operações de negócio, sem DELETE
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO app;

-- Conceder acesso a sequências (necessário para SERIAL)
GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO dev, app;

Passo 4 — Configurar privilégios para objetos futuros


-- Para que novas tabelas herdem automaticamente os privilégios:
ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO dev;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON TABLES TO leitura;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT, INSERT, UPDATE ON TABLES TO app;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT USAGE ON SEQUENCES TO dev, app;

Passo 5 — Verificar a configuração

Para verificar os privilégios concedidos, utilize as consultas a seguir:


-- Listar privilégios de tabela por role
SELECT
    grantee,
    table_schema,
    table_name,
    privilege_type
FROM information_schema.table_privileges
WHERE table_schema = 'public'
ORDER BY grantee, table_name, privilege_type;

O que é Backup e Restauração?

Backup é o processo de criar uma cópia de segurança dos dados e da estrutura do banco de dados. Restauração é o processo inverso: recuperar os dados a partir de um backup.

Backups são a última linha de defesa contra perda de dados. Sem um plano de backup adequado, você corre o risco de perder todo o trabalho em caso de:

  • Falhas de hardware: discos rígidos queimados, servidores danificados
  • Erros humanos: DROP TABLE acidental, DELETE sem WHERE
  • Desastres naturais: incêndios, inundações, quedas de energia
  • Ataques maliciosos: ransomware, invasões, sabotagem
  • Migração de servidores: mover o banco para outro ambiente
Regra de ouro do DBA: "Se você não testou a restauração do seu backup, você não tem um backup — você tem apenas uma esperança."

Tipos de Backup

O PostgreSQL oferece três abordagens principais para backup:

Tipo Ferramenta Descrição Vantagens Desvantagens
Lógico pg_dump
pg_dumpall		 Gera comandos SQL para recriar a estrutura e os dados do banco Portável entre versões e arquiteturas, permite restaurar objetos individuais, pode ser editado manualmente Mais lento que backup físico, pode bloquear operações em tabelas grandes
Físico Cópia do diretório de dados Cópia em nível de arquivo do diretório PGDATA Extremamente rápido, ideal para volumes muito grandes de dados Restauração somente no mesmo SO e arquitetura, não permite restaurar objetos individuais, requer que o servidor esteja parado
Contínuo (WAL) WAL Archiving + pg_basebackup Arquiva continuamente os Write-Ahead Logs, permitindo Point-In-Time Recovery (PITR) Permite restaurar para qualquer ponto no tempo, perda mínima de dados (RPO próximo de zero), ideal para ambientes críticos Mais complexo de configurar e gerenciar, requer mais espaço em disco, manutenção contínua

Write-Ahead Log (WAL)

O WAL é o mecanismo central de durabilidade do PostgreSQL. Toda alteração é primeiro escrita no log (WAL) antes de ser aplicada aos arquivos de dados. Esse log pode ser arquivado para backup contínuo, permitindo restaurar o banco para qualquer ponto no tempo (Point-In-Time Recovery — PITR).

Ferramentas de Backup do PostgreSQL

pg_dump — Backup Lógico de um Banco Específico

O pg_dump é a ferramenta mais utilizada para backups lógicos. Ele gera um arquivo contendo os comandos SQL necessários para recriar o banco de dados.

Backup em formato texto (SQL puro)


pg_dump -U postgres -d meubanco -f backup.sql

Backup em formato customizado (binário, comprimido)

O formato customizado (-Fc) é mais eficiente, gera arquivos menores e permite restaurar objetos seletivamente com pg_restore.


pg_dump -U postgres -d meubanco -Fc -f backup.dump

Backup apenas do schema (estrutura)


pg_dump -U postgres -d meubanco --schema-only -f schema.sql

Backup apenas dos dados


pg_dump -U postgres -d meubanco --data-only -f dados.sql

Backup de tabelas específicas


pg_dump -U postgres -d meubanco -t clientes -t pedidos -f tabelas.sql

Backup com compressão


# Formato texto com compressão gzip
pg_dump -U postgres -d meubanco | gzip > backup.sql.gz

# Formato customizado com nível de compressão 9 (máximo)
pg_dump -U postgres -d meubanco -Fc -Z9 -f backup_compressed.dump

pg_dumpall — Backup Lógico de Todo o Cluster

O pg_dumpall faz backup de todos os bancos de dados do cluster, incluindo objetos globais como roles e tablespaces. 


# Backup completo do cluster (todos os bancos + roles + tablespaces)
pg_dumpall -U postgres -f backup_total.sql

# Backup apenas das roles (objetos globais)
pg_dumpall -U postgres --roles-only -f roles.sql

Opções úteis do pg_dump

Opção Descrição
-U usuario Usuário de conexão
-d banco Nome do banco de dados
-f arquivo Arquivo de saída
-F formato Formato: p (plain/texto), c (custom), t (tar), d (directory)
-t tabela Fazer backup apenas de tabela(s) específica(s)
-Z 0-9 Nível de compressão (0 = sem, 9 = máxima)
--schema-only Apenas estrutura, sem dados
--data-only Apenas dados, sem estrutura
--no-owner Não incluir comandos de propriedade (útil para restaurar em outro usuário)
--no-privileges Não incluir comandos GRANT/REVOKE
--verbose Exibe progresso detalhado durante o dump

Restauração de Backups

A restauração é o processo de recuperar um banco de dados a partir de um backup. O método depende do formato do backup.

Restaurando backup em formato texto (SQL)

Para arquivos gerados com pg_dump -Fp ou pg_dumpall:


# Restaurar em um banco existente
psql -U postgres -d meubanco -f backup.sql

# Restaurar com saída detalhada (verbose)
psql -U postgres -d meubanco -f backup.sql -v ON_ERROR_STOP=1

Restaurando backup em formato customizado (.dump)

Para arquivos gerados com pg_dump -Fc, use o pg_restore:


# Restaurar backup customizado em um banco existente
pg_restore -U postgres -d meubanco backup.dump

# Restaurar apenas uma tabela específica do backup customizado
pg_restore -U postgres -d meubanco -t clientes backup.dump

# Listar o conteúdo do backup antes de restaurar
pg_restore -l backup.dump

Passo a passo: Criar banco e restaurar backup

Este é o fluxo completo para migrar ou recuperar um banco de dados:


# Passo 1: Criar um novo banco de dados vazio
createdb -U postgres novo_banco

# Passo 2: Restaurar o backup no novo banco (formato texto)
psql -U postgres -d novo_banco -f backup.sql

# OU para formato customizado:
pg_restore -U postgres -d novo_banco backup.dump

# Passo 3: Verificar se a restauração foi bem-sucedida
psql -U postgres -d novo_banco -c "\dt"
psql -U postgres -d novo_banco -c "SELECT count(*) FROM clientes;"

Backup e restauração com pipe (sem arquivo intermediário)

Para migrar diretamente entre bancos sem criar arquivo no disco:


# Backup de um banco e restauração direta em outro
pg_dump -U postgres -d banco_origem | psql -U postgres -d banco_destino

Opções úteis do pg_restore

Opção Descrição
-d banco Nome do banco de destino
-t tabela Restaurar apenas tabela(s) específica(s)
-l Listar conteúdo do backup (sem restaurar)
-C Criar o banco de dados antes de restaurar
-c Limpar (drop) objetos antes de recriá-los
--no-owner Não restaurar propriedade dos objetos
--no-privileges Não restaurar privilégios (GRANT/REVOKE)
-j n Restaurar com n jobs paralelos (apenas formato directory)
--verbose Exibir progresso detalhado
Boas práticas de backup:
  • Automatize backups com cron (Linux) ou Task Scheduler (Windows)
  • Armazene backups em local diferente do servidor de banco de dados
  • Teste a restauração periodicamente para garantir que os backups são válidos
  • Mantenha múltiplas gerações de backup (ex: diário dos últimos 7 dias + semanal dos últimos 30 dias)
  • Documente o procedimento de restauração — em uma emergência, não haverá tempo para pesquisar

Gabarito: 

-- Criar as roles
CREATE ROLE admin_app WITH LOGIN PASSWORD 'admin123';

-- Acesso ao banco e schema
GRANT CONNECT ON DATABASE loja TO admin_app, operador, relatorios;
GRANT USAGE ON SCHEMA public TO admin_app, operador, relatorios;

-- admin_app: acesso total
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO admin_app;

-- operador: DML em vendas e produtos, SELECT em clientes
GRANT SELECT, INSERT, UPDATE ON vendas, produtos TO operador;
GRANT SELECT ON clientes TO operador;

-- relatorios: apenas leitura

GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO admin_app, operador;
