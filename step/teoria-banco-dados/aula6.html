Operadores para Subconsultas

Além do IN que vimos na aula anterior, o SQL oferece operadores poderosos para comparar um valor com o resultado de uma subconsulta. Vamos explorar EXISTS, ANY/SOME e ALL.

EXISTS — Verificando a existência de registros

O operador EXISTS retorna TRUE se a subconsulta retornar pelo menos uma linha. É um operador de teste de existência — ele não se importa com quantas linhas retornam, apenas se existe alguma. É muito eficiente para consultas correlacionadas, pois o banco de dados pode parar a busca assim que encontrar a primeira correspondência. 


-- Clientes que fizeram pelo menos um pedido acima de 1000
SELECT nome FROM clientes c
WHERE EXISTS (
    SELECT 1 FROM pedidos p
    WHERE p.cliente_id = c.id
      AND p.valor_total > 1000
);

-- Categorias que possuem produtos sem estoque
SELECT nome FROM categorias cat
WHERE EXISTS (
    SELECT 1 FROM produtos p
    WHERE p.categoria_id = cat.id
      AND p.estoque = 0
);

-- NOT EXISTS: clientes que NUNCA fizeram pedido
SELECT nome FROM clientes c
WHERE NOT EXISTS (
    SELECT 1 FROM pedidos p
    WHERE p.cliente_id = c.id
);

Dica: Use SELECT 1 ou SELECT * dentro do EXISTS — o valor retornado não importa, apenas a existência de linhas.

ANY e SOME — Comparando com qualquer valor da lista

Os operadores ANY e SOME são sinônimos. Eles comparam um valor com qualquer valor retornado pela subconsulta. A condição é verdadeira se a comparação for verdadeira para pelo menos um dos valores. 


-- Produtos com preço maior que QUALQUER produto da categoria 'Eletrônicos'
-- (ou seja, maior que o menor preço de Eletrônicos)
SELECT nome, preco FROM produtos
WHERE preco > ANY (
    SELECT preco FROM produtos WHERE categoria = 'Eletrônicos'
);

-- Funcionários que ganham mais que PELO MENOS UM gerente
SELECT nome, salario FROM funcionarios
WHERE salario > ANY (
    SELECT salario FROM funcionarios WHERE cargo = 'Gerente'
);

Entenda a semântica: X > ANY (subconsulta) significa "X é maior que pelo menos um dos valores". É equivalente a X > MIN(subconsulta).

ALL — Comparando com todos os valores da lista

O operador ALL compara um valor com todos os valores retornados pela subconsulta. A condição só é verdadeira se a comparação for verdadeira para cada um dos valores. 


-- Produto mais caro que TODOS os eletrônicos
-- (ou seja, maior que o MAIOR preço dos eletrônicos)
SELECT nome, preco FROM produtos
WHERE preco > ALL (
    SELECT preco FROM produtos WHERE categoria = 'Eletrônicos'
);

-- Funcionário que ganha mais que TODOS os gerentes
SELECT nome, salario FROM funcionarios
WHERE salario > ALL (
    SELECT salario FROM funcionarios WHERE cargo = 'Gerente'
);

Entenda a semântica: X > ALL (subconsulta) significa "X é maior que todos os valores". É equivalente a X > MAX(subconsulta).

Operador Significado Equivalente com agregação
X > ANY (...) X é maior que pelo menos um X > (SELECT MIN(...) ...)
X > ALL (...) X é maior que todos X > (SELECT MAX(...) ...)
X = ANY (...) X é igual a pelo menos um X IN (...)
X != ALL (...) X é diferente de todos X NOT IN (...)

UNION e UNION ALL — Combinando resultados de consultas

Os operadores UNION e UNION ALL combinam os resultados de duas ou mais consultas SELECT em um único conjunto de resultados. A diferença entre eles é crucial:

  • UNION: combina os resultados e remove duplicatas (mais lento, pois precisa ordenar e comparar)
  • UNION ALL: combina os resultados e mantém todas as linhas, inclusive duplicatas (mais rápido)

Regras para usar UNION

  • Todas as consultas devem retornar o mesmo número de colunas
  • As colunas correspondentes devem ter tipos de dados compatíveis
  • A ordenação (ORDER BY) só pode aparecer no final, após o último SELECT
  • Os nomes das colunas no resultado são definidos pela primeira consulta

UNION — Removendo duplicatas


-- Lista única de e-mails de clientes e fornecedores (sem repetições)
SELECT nome, email FROM clientes
UNION
SELECT nome, email FROM fornecedores
ORDER BY nome;

-- Cidades onde temos clientes OU fornecedores
SELECT cidade FROM clientes
UNION
SELECT cidade FROM fornecedores;

UNION ALL — Mantendo todas as linhas


-- Todos os contatos, mesmo que duplicados (ex: mesma pessoa é cliente e fornecedor)
SELECT nome, email FROM clientes
UNION ALL
SELECT nome, email FROM fornecedores;

-- Combinando vendas de diferentes regiões (tabelas separadas)
SELECT data_venda, valor, 'Norte' AS regiao FROM vendas_norte
UNION ALL
SELECT data_venda, valor, 'Sul'   AS regiao FROM vendas_sul
UNION ALL
SELECT data_venda, valor, 'Leste' AS regiao FROM vendas_leste
ORDER BY data_venda;

Quando usar UNION vs UNION ALL?

Situação Use
Precisa eliminar duplicatas UNION
Tem certeza que não há duplicatas UNION ALL (mais rápido)
Precisa contar todas as ocorrências UNION ALL
Alta performance é crítica UNION ALL

INNER JOIN — Relacionando tabelas

O INNER JOIN é o tipo de junção mais comum. Ele retorna apenas as linhas que possuem correspondência em ambas as tabelas envolvidas, com base na condição especificada na cláusula ON.

Sintaxe básica do INNER JOIN


SELECT tabela1.coluna, tabela2.coluna
FROM tabela1
INNER JOIN tabela2 ON tabela1.chave = tabela2.chave_estrangeira;

Exemplo prático: Clientes e seus pedidos


-- Lista os pedidos com o nome do cliente
SELECT c.nome AS cliente, p.data_pedido, p.valor_total
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
ORDER BY p.data_pedido DESC;

INNER JOIN com múltiplas tabelas

É possível encadear vários JOINs para navegar por múltiplas tabelas relacionadas, construindo consultas complexas que abrangem todo o modelo de dados. 


-- Detalhamento completo do pedido: cliente, produtos e quantidades
SELECT
    c.nome      AS cliente,
    p.id        AS pedido,
    p.data_pedido,
    pr.nome     AS produto,
    ip.quantidade,
    pr.preco,
    (ip.quantidade * pr.preco) AS subtotal
FROM clientes c
INNER JOIN pedidos p        ON c.id  = p.cliente_id
INNER JOIN itens_pedido ip  ON p.id  = ip.pedido_id
INNER JOIN produtos pr      ON ip.produto_id = pr.id
ORDER BY p.data_pedido DESC;

INNER JOIN com filtros adicionais


-- Pedidos de clientes de São Paulo com valor acima de 500
SELECT c.nome, p.data_pedido, p.valor_total
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
WHERE c.cidade = 'São Paulo'
  AND p.valor_total > 500
ORDER BY p.valor_total DESC;

INNER JOIN com GROUP BY e agregações


-- Total gasto por cliente (apenas clientes que fizeram pedidos)
SELECT c.nome, COUNT(p.id) AS qtd_pedidos, SUM(p.valor_total) AS total_gasto
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
GROUP BY c.id, c.nome
HAVING SUM(p.valor_total) > 1000
ORDER BY total_gasto DESC;

LEFT JOIN — Incluindo todos os registros da esquerda

O LEFT JOIN (ou LEFT OUTER JOIN) retorna todas as linhas da tabela à esquerda (a primeira tabela mencionada), mesmo que não haja correspondência na tabela à direita. Quando não há correspondência, as colunas da tabela direita são preenchidas com NULL.

Visualizando o LEFT JOIN

Imagine dois conjuntos: A (tabela esquerda) e B (tabela direita). O LEFT JOIN retorna: todo o conjunto A + as interseções com B. O que está em B mas não em A é ignorado. 


-- TODOS os clientes, com ou sem pedidos
SELECT c.nome, p.data_pedido, p.valor_total
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
ORDER BY c.nome;

Encontrando registros SEM correspondência

Um padrão muito comum com LEFT JOIN é identificar registros da tabela esquerda que não possuem correspondência na tabela direita. Basta filtrar por WHERE coluna_da_direita IS NULL. 


-- Clientes que NUNCA fizeram um pedido
SELECT c.nome, c.email
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
WHERE p.id IS NULL;

-- Produtos que nunca foram vendidos
SELECT p.nome, p.categoria
FROM produtos p
LEFT JOIN itens_pedido ip ON p.id = ip.produto_id
WHERE ip.produto_id IS NULL;

LEFT JOIN com agregações


-- Quantidade de pedidos por cliente (incluindo quem tem 0 pedidos)
SELECT c.nome, COUNT(p.id) AS total_pedidos
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
GROUP BY c.id, c.nome
ORDER BY total_pedidos DESC;

RIGHT JOIN — Incluindo todos os registros da direita

O RIGHT JOIN (ou RIGHT OUTER JOIN) é o oposto do LEFT JOIN: ele retorna todas as linhas da tabela à direita, mesmo sem correspondência na esquerda. Na prática, a maioria dos desenvolvedores prefere usar LEFT JOIN (apenas invertendo a ordem das tabelas) por ser mais legível, mas o RIGHT JOIN existe e é importante conhecê-lo. 


-- Todos os pedidos, mesmo sem cliente associado (ex: pedidos órfãos)
SELECT c.nome, p.id AS pedido_id, p.data_pedido
FROM clientes c
RIGHT JOIN pedidos p ON c.id = p.cliente_id;

-- Equivalente usando LEFT JOIN (invertendo as tabelas)
SELECT c.nome, p.id AS pedido_id, p.data_pedido
FROM pedidos p
LEFT JOIN clientes c ON c.id = p.cliente_id;

Dica: Na prática, prefira sempre LEFT JOIN e organize as tabelas de forma que a tabela "principal" (da qual você quer todos os registros) fique à esquerda. Isso torna o código mais legível e padronizado.

FULL JOIN — A união completa

O FULL JOIN (ou FULL OUTER JOIN) retorna todas as linhas de ambas as tabelas. Quando há correspondência, as colunas são combinadas. Quando não há, as colunas da tabela sem correspondência são preenchidas com NULL. É a união dos comportamentos do LEFT JOIN e do RIGHT JOIN. 


-- Todos os clientes e todos os pedidos (combinados quando possível)
SELECT c.nome, p.id AS pedido_id, p.data_pedido
FROM clientes c
FULL JOIN pedidos p ON c.id = p.cliente_id;

Encontrando registros que NÃO correspondem em nenhuma direção

Com o FULL JOIN, podemos encontrar registros órfãos em ambas as tabelas simultaneamente — algo impossível com apenas LEFT ou RIGHT JOIN. 


-- Clientes sem pedidos E pedidos sem cliente (dados inconsistentes)
SELECT c.nome, p.id AS pedido_id
FROM clientes c
FULL JOIN pedidos p ON c.id = p.cliente_id
WHERE c.id IS NULL OR p.id IS NULL;

Comparação Visual de Todos os JOINs

A tabela a seguir resume o que cada tipo de JOIN retorna, comparando com a teoria dos conjuntos:

Tipo de JOIN O que retorna Equivalente em conjuntos Quando usar
INNER JOIN Apenas linhas com correspondência em ambas as tabelas Interseção (A ∩ B) Quando você precisa apenas dos registros que existem nos dois lados
LEFT JOIN Todas as linhas da tabela esquerda + correspondências da direita A inteiro + interseção Quando a tabela esquerda é a principal e você quer tudo dela
RIGHT JOIN Todas as linhas da tabela direita + correspondências da esquerda B inteiro + interseção Similar ao LEFT JOIN, mas a tabela direita é a principal
FULL JOIN Todas as linhas de ambas as tabelas, com ou sem correspondência União (A ∪ B) Quando você quer todos os registros de ambos os lados
LEFT JOIN + WHERE IS NULL Apenas registros da esquerda sem correspondência na direita A - B Encontrar registros órfãos ou faltantes

Dever de Casa

1. INNER JOIN entre 3 tabelas

Escreva uma consulta que retorne o nome do cliente, a data do pedido, o nome do produto e a quantidade de cada item pedido. Use INNER JOIN entre as tabelas clientes, pedidos, itens_pedido e produtos.

Gabarito: 

SELECT
    c.nome AS cliente,
    p.data_pedido,
    pr.nome AS produto,
    ip.quantidade
FROM clientes c
INNER JOIN pedidos p       ON c.id = p.cliente_id
INNER JOIN itens_pedido ip ON p.id = ip.pedido_id
INNER JOIN produtos pr     ON ip.produto_id = pr.id
ORDER BY p.data_pedido DESC;

2. LEFT JOIN listando clientes que nunca fizeram pedido

Escreva uma consulta que retorne o nome e o e-mail de todos os clientes que nunca realizaram um pedido. Use LEFT JOIN e verificação de NULL.

Gabarito: 

SELECT c.nome, c.email
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
WHERE p.id IS NULL
ORDER BY c.nome;

3. UNION combinando resultados de duas consultas diferentes

Escreva uma consulta que retorne uma lista única de nomes e e-mails, combinando os clientes que são de "São Paulo" com os fornecedores que são de "São Paulo". Identifique a origem com uma coluna "tipo".

Gabarito: 

SELECT nome, email, 'Cliente' AS tipo
FROM clientes
WHERE cidade = 'São Paulo'
UNION
SELECT nome, email, 'Fornecedor' AS tipo
FROM fornecedores
WHERE cidade = 'São Paulo'
ORDER BY nome;

4. Subconsulta com EXISTS para encontrar produtos que já foram pedidos

Escreva uma consulta que retorne o nome e o preço de todos os produtos que já apareceram em pelo menos um pedido. Use EXISTS com subconsulta correlacionada.

Gabarito: 

SELECT nome, preco
FROM produtos p
WHERE EXISTS (
    SELECT 1 FROM itens_pedido ip
    WHERE ip.produto_id = p.id
)
ORDER BY nome;
