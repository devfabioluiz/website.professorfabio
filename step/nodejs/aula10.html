Escalabilidade: Processos Filhos e Cluster

Tópicos do nosso plano de curso abordados nessa aula: Motivos para escalar aplicações, Processos filhos, Exec, spawn e fork.

Por que Escalar Aplicações?

Escalar uma aplicação significa fazê-la suportar mais usuários, mais requisições e mais carga de processamento sem perder performance ou estabilidade.

Motivos comuns para escalar:

  • Aumento de usuários simultâneos
  • Picos de acesso (Black Friday, campanhas)
  • Processamento pesado (imagens, relatórios, IA)
  • Alta disponibilidade (aplicação não pode cair)
  • Tolerância a falhas (se um processo cair, outro assume)

O Problema do Single Thread

O Node.js roda JavaScript em uma única thread (Event Loop). Isso é eficiente para operações I/O (rede, arquivos, banco), mas tarefas pesadas de CPU (cálculos, criptografia, compressão) podem bloquear o Event Loop e travar toda a aplicação.

Demonstração do bloqueio

const http = require('http');

const server = http.createServer((req, res) => {
  if (req.url === '/pesado') {

    let resultado = 0;
    for (let i = 0; i < 1e9; i++) {
      resultado += i;
    }

    res.end(`Resultado: ${resultado}`);
    return;
  }

  res.end('Rota normal - resposta rápida');
});

server.listen(3000, () => {
  console.log('Servidor rodando');
});

Quando alguém acessar /pesado, TODAS as outras requisições ficarão congeladas até o cálculo terminar. Isso acontece porque a CPU está 100% ocupada na única thread disponível.

Escalabilidade Vertical vs Horizontal

Vertical (Scale Up) Horizontal (Scale Out)
Aumenta recursos da máquina (RAM, CPU) Adiciona mais máquinas/processos
Mais simples de implementar Mais complexo (load balancer, estado)
Tem limite físico Escala "infinitamente"
Ponto único de falha Alta disponibilidade

O Node.js facilita a escalabilidade horizontal com o módulo cluster e o módulo child_process.

Módulo child_process

O módulo child_process permite criar processos filhos que executam comandos ou scripts em paralelo, sem bloquear o processo principal. Existem 4 métodos principais:

Método Usa Shell? Buffer? Quando usar
exec() Sim Sim (buffer) Comandos simples, saída pequena
spawn() Não (por padrão) Não (stream) Comandos com muita saída, processos longos
fork() Não Não Executar outro arquivo Node.js (comunicação IPC)
execFile() Não Sim (buffer) Executar arquivo binário sem shell

exec() — Executando Comandos do Sistema

const { exec } = require('child_process');

exec('dir', { cwd: 'C:\\' }, (erro, stdout, stderr) => {
  if (erro) {
    console.log('Erro:', erro.message);
    return;
  }

  if (stderr) {
    console.log('Stderr:', stderr);
    return;
  }

  console.log('Saída:');
  console.log(stdout);
});

🤓 Explicando o código 📝

🔹 cwd

Define o diretório de trabalho do comando. Útil para executar comandos em pastas específicas.

🔹 stdout vs stderr

stdout é a saída normal do comando. stderr é a saída de erro. Sempre verifique ambos.

🔹 Cuidado com buffer overflow

O exec() armazena toda a saída em um buffer na memória. Se o comando gerar gigabytes de saída, o processo pode travar. Para saída grande, use spawn().

spawn() — Processando Saída como Stream

const { spawn } = require('child_process');

const comando = spawn('ping', ['-n', '5', 'google.com']);

comando.stdout.on('data', (dados) => {
  console.log(`[stdout]: ${dados}`);
});

comando.stderr.on('data', (dados) => {
  console.log(`[stderr]: ${dados}`);
});

comando.on('close', (codigo) => {
  console.log(`Processo encerrado com código ${codigo}`);
});

Com spawn(), a saída chega como stream (evento data), permitindo processar gigabytes de saída sem consumir memória.

fork() — Comunicação entre Processos Node.js

fork() é um caso especial de spawn() projetado para executar arquivos Node.js. Ele cria um canal de comunicação IPC (Inter-Process Communication) entre o pai e o filho.

Arquivo pai (main.js)

const { fork } = require('child_process');

const filho = fork('./worker.js');

filho.send({ tarefa: 'calcular', dados: [1, 2, 3, 4, 5] });

filho.on('message', (mensagem) => {
  console.log('Resultado recebido do filho:', mensagem);
});

filho.on('close', (codigo) => {
  console.log('Filho encerrado com código:', codigo);
});

Arquivo filho (worker.js)

process.on('message', (mensagem) => {
  console.log('Mensagem recebida do pai:', mensagem);

  if (mensagem.tarefa === 'calcular') {
    const soma = mensagem.dados.reduce((a, b) => a + b, 0);
    const media = soma / mensagem.dados.length;

    process.send({
      tarefa: 'calcular',
      resultado: { soma, media }
    });
  }
});

🤓 Explicando o código 📝

🔹 process.send() e process.on('message')

No processo filho, process.send() envia mensagens para o pai. process.on('message') escuta mensagens vindas do pai. No pai, usamos filho.send() e filho.on('message').

🔹 IPC (Inter-Process Communication)

O canal IPC usa pipes do sistema operacional. As mensagens são serializadas com JSON.stringify(), então você pode enviar qualquer dado serializável (objetos, arrays, números, strings).

🔹 Casos de uso do fork()

  • Processamento de imagens e vídeos
  • Geração de relatórios pesados
  • Cálculos matemáticos intensivos
  • Processamento em lote (batch)

Módulo Cluster — Multiplicando o Servidor

O módulo cluster permite criar múltiplas instâncias (workers) do seu servidor, cada uma rodando em um núcleo (core) da CPU. Todas compartilham a mesma porta.

Se sua máquina tem 8 núcleos, você pode ter 8 workers processando requisições simultaneamente — um aumento de até 8x na capacidade.

Como o Cluster funciona

  1. Um processo Master é criado
  2. O Master cria Workers (um por núcleo da CPU)
  3. O Master escuta a porta (ex: 3000) e distribui requisições entre os Workers (round-robin)
  4. Se um Worker morrer, o Master cria um novo automaticamente

Servidor com Cluster

const cluster = require('cluster');
const http = require('http');
const numCPUs = require('os').cpus().length;

if (cluster.isMaster) {
  console.log(`Master ${process.pid} está rodando`);

  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  cluster.on('exit', (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} morreu`);
    console.log('Criando novo worker...');
    cluster.fork();
  });

} else {

  http.createServer((req, res) => {
    res.writeHead(200);
    res.end(`Resposta do Worker ${process.pid}\n`);
  }).listen(3000);

  console.log(`Worker ${process.pid} iniciado`);
}

Resultado

Master 12345 está rodando
Worker 12346 iniciado
Worker 12347 iniciado
Worker 12348 iniciado
Worker 12349 iniciado
...

Acesse http://localhost:3000 várias vezes. Cada requisição será respondida por um Worker diferente (round-robin).

Passo 1 — Criar o Worker de Cálculo Pesado

Esse worker vai rodar em um processo separado, liberando o Event Loop do servidor principal.

Criar workers/calculo.js:

mkdir workers
type nul > workers\calculo.js
            
process.on("message", (dados) => {
  const { tipo, valor } = dados;

  switch (tipo) {
    case "fatorial":
      let fatorial = 1n;

      for (let i = 2n; i <= BigInt(valor); i++) {
        fatorial *= i;
      }

      process.send({
        tipo: "fatorial",
        entrada: valor,
        resultado: fatorial.toString(),
      });
      break;

    case "primos":
      const primos = [];
      const limite = valor;

      for (let n = 2; n <= limite; n++) {
        let ehPrimo = true;

        for (let d = 2; d * d <= n; d++) {
          if (n % d === 0) {
            ehPrimo = false;
            break;
          }
        }

        if (ehPrimo) {
          primos.push(n);
        }
      }

      process.send({
        tipo: "primos",
        entrada: valor,
        resultado: primos.length,
      });
      break;

    default:
      process.send({
        erro: "Tipo de cálculo não suportado",
      });
  }

  process.exit(0);
});

Passo 2 — Criar Controller de Cálculo

Criar src/controllers/calculoController.js:

type nul > src\controllers\calculoController.js
            
const { fork } = require("child_process");
const path = require("path");
const logger = require("../utils/logger");

const executarCalculo = (req, res) => {
  const { tipo, valor } = req.body;

  if (!tipo || !valor) {
    return res.status(400).json({
      erro: "Informe 'tipo' (fatorial ou primos) e 'valor'",
    });
  }

  const tiposValidos = ["fatorial", "primos"];

  if (!tiposValidos.includes(tipo)) {
    return res.status(400).json({
      erro: `Tipo inválido. Use: ${tiposValidos.join(", ")}`,
    });
  }

  logger.info(`Iniciando cálculo ${tipo}(${valor}) em processo filho`);

  const caminhoWorker = path.join(
    __dirname, "../../workers/calculo.js"
  );

  const worker = fork(caminhoWorker);

  worker.send({ tipo, valor });

  worker.on("message", (resultado) => {
    res.json(resultado);
    logger.info(`Cálculo ${tipo} concluído`);
  });

  worker.on("error", (erro) => {
    logger.erro(`Erro no worker: ${erro.message}`);
    res.status(500).json({ erro: "Erro no processamento" });
  });
};

module.exports = { executarCalculo };

require("dotenv").config();

const cluster = require("cluster");
const os = require("os");

const numCPUs = os.cpus().length;

if (cluster.isMaster) {
  console.log(`Master ${process.pid} rodando`);
  console.log(`Criando ${numCPUs} workers...`);

  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  cluster.on("exit", (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} caiu (código: ${code})`);
    console.log("Reiniciando worker...");
    cluster.fork();
  });

} else {
  const app = require("./src/app");

  const conectarBanco = require("./src/database/database");

  conectarBanco();

  app.listen(process.env.PORT || 3000, () => {
    console.log(`Worker ${process.pid} ouvindo na porta ${process.env.PORT || 3000}`);
  });
}

const http = require('http');

const totalRequisicoes = 100;
let completas = 0;

for (let i = 0; i < totalRequisicoes; i++) {

🤓 Explicando o código 📝

🔹 cluster.isMaster

Determina se o processo atual é o Master. O Master gerencia os Workers; os Workers executam o servidor.

🔹 cluster.fork()

Cria um novo Worker. O código do arquivo atual é executado novamente, mas desta vez cluster.isMaster será false, caindo no bloco else do Worker.

🔹 cluster.on('exit')

Permite detectar quando um Worker morre e criar um substituto. Isso garante zero downtime — sempre há Workers ativos.

🔹 Por que o fork() no controller?

Tarefas pesadas de CPU NÃO devem rodar no mesmo processo do servidor. Usamos fork() para criar um processo isolado que faz o cálculo. O servidor continua respondendo outras requisições enquanto o cálculo roda em paralelo.

Desenvolva:

  • Um arquivo workers/relatorio.js que:
    • Recebe um array de produtos via process.on('message')
    • Calcula estatísticas: total de produtos, valor total em estoque, preço médio, produto mais caro e mais barato
    • Envia o resultado de volta com process.send()
  • Uma rota POST /relatorio-produtos que:
    • Busca os produtos do banco (MongoDB ou PostgreSQL)
    • Envia para o worker processar
    • Retorna o relatório gerado pelo worker
Gabarito:

workers/relatorio.js

process.on("message", (produtos) => {
  if (!produtos || produtos.length === 0) {
    process.send({ erro: "Nenhum produto para analisar" });
    process.exit(0);
  }

  const precos = produtos.map((p) => p.preco);
  const total = precos.reduce((a, b) => a + b, 0);
  const media = total / produtos.length;
  const max = Math.max(...precos);
  const min = Math.min(...precos);

  const maisCaro = produtos.find((p) => p.preco === max);
  const maisBarato = produtos.find((p) => p.preco === min);

  process.send({
    totalProdutos: produtos.length,
    valorTotalEstoque: total,
    precoMedio: media.toFixed(2),
    maisCaro: { nome: maisCaro.nome, preco: maisCaro.preco },
    maisBarato: { nome: maisBarato.nome, preco: maisBarato.preco },
  });

  process.exit(0);
});
