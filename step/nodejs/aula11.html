Boas Práticas em Produção

Antes de colocar uma aplicação no ar, é essencial seguir boas práticas que garantem segurança, performance e manutenibilidade.

1. Variáveis de Ambiente

NUNCA coloque senhas, chaves de API ou URLs de banco no código. Use variáveis de ambiente com o arquivo .env e o pacote dotenv. 

JWT_SECRET=uma-string-longa-e-aleatoria
MONGO_URL=mongodb+srv://...
PG_PASSWORD=senha-do-banco-postgres
PORT=3000
NODE_ENV=production

IMPORTANTE: adicione o arquivo .env ao .gitignore. Ele NUNCA deve ser commitado. 

node_modules/
.env
logs/
uploads/

2. Scripts npm úteis

{
  "scripts": {
    "dev": "node server.js",
    "start": "node server.js",
    "migrate": "node src/database/migrate.js",
    "seed": "node src/database/seeds.js",
    "benchmark": "node benchmark.js"
  }
}
Comando Convenção
npm start Inicia a aplicação em produção
npm run dev Inicia em modo desenvolvimento (com nodemon, etc.)
npm test Executa os testes
npm run build Compila/cria assets para produção

3. Tratamento de Erros Global

process.on('uncaughtException', (erro) => {
  console.error('Erro não tratado:', erro);
  process.exit(1);
});

process.on('unhandledRejection', (erro) => {
  console.error('Promise rejeitada sem catch:', erro);
  process.exit(1);
});

4. Middleware de Erro no Express

app.use((erro, req, res, next) => {
  console.error('Erro:', erro.message);

  res.status(erro.status || 500).json({
    erro: erro.message || 'Erro interno do servidor'
  });
});

O Express reconhece middlewares com 4 parâmetros como error handlers. Eles só são executados quando next(erro) é chamado.

5. Helmet — Segurança HTTP Headers

npm install helmet

const helmet = require('helmet');

app.use(helmet());

O helmet adiciona automaticamente headers de segurança que protegem contra XSS, clickjacking, sniffing de MIME type e outras vulnerabilidades comuns.

6. CORS — Controle de Acesso

npm install cors

const cors = require('cors');

app.use(cors({
  origin: 'https://meu-frontend.com',
  methods: ['GET', 'POST', 'PUT', 'DELETE']
}));

7. Compression — Reduzindo o Tamanho das Respostas

npm install compression

const compression = require('compression');

app.use(compression());

8. Rate Limiting — Proteção contra Abusos

npm install express-rate-limit

const rateLimit = require('express-rate-limit');

const limitador = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  message: { erro: 'Muitas requisições. Tente novamente mais tarde.' }
});

app.use('/api/', limitador);

Publicando um Pacote no npm

O npm não serve apenas para instalar pacotes — você também pode publicar os seus próprios. Isso é útil para compartilhar bibliotecas, utilitários e ferramentas com a comunidade ou com sua equipe.

Preparando o pacote

Estrutura de um pacote npm publicável:

meu-pacote/
│
├── index.js          # ponto de entrada
├── package.json      # metadados do pacote
├── README.md         # documentação
└── LICENSE           # licença

package.json para publicação

{
  "name": "meu-utilitario-node",
  "version": "1.0.0",
  "description": "Utilitário para formatação de dados",
  "main": "index.js",
  "keywords": ["node", "util", "formatacao"],
  "author": "Seu Nome",
  "license": "MIT",
  "repository": {
    "type": "git",
    "url": "https://github.com/seuusuario/meu-pacote.git"
  },
  "files": [
    "index.js",
    "src/"
  ]
}

🤓 Explicando o código 📝

🔹 name

Deve ser único no npm. Use @seu-usuario/nome para criar pacotes com escopo (scoped packages), que não precisam ter nome único globalmente.

🔹 version

Segue o SemVer: MAJOR.MINOR.PATCH. Ex: 1.2.3. MAJOR: mudanças que quebram compatibilidade. MINOR: novas features compatíveis. PATCH: correções de bugs.

🔹 main

Arquivo que será carregado quando alguém fizer require('seu-pacote').

🔹 files

Lista de arquivos que serão incluídos no pacote. Use para evitar enviar arquivos desnecessários (testes, configs, .env, etc.).

Passo a passo para publicar

npm login

npm publish

npm version patch

npm publish

npm version minor

npm publish

npm version major

npm publish

Atualizando um pacote

npm version patch
npm publish

Isso atualiza automaticamente a versão no package.json, cria uma tag no Git e publica a nova versão no npm.

Deploy com Railway (Passo a Passo)

O Railway é uma das plataformas mais simples para deploy. Ele se conecta diretamente ao GitHub e faz deploy automático a cada push.

1. Preparar o projeto para deploy

Certifique-se de que o package.json tem o script start: 

{
  "scripts": {
    "start": "node server.js"
  },
  "engines": {
    "node": ">=18.0.0"
  }
}

Teste localmente:

npm start

2. Criar conta no Railway

Acesse: https://railway.app

Faça login com sua conta do GitHub.

3. Criar novo projeto

  1. Clique em New Project
  2. Selecione Deploy from GitHub repo
  3. Conecte o repositório correto

4. Configurar variáveis de ambiente

No painel do Railway, vá em Variables e adicione:

PORT=3000
NODE_ENV=production
MONGO_URL=mongodb+srv://...
JWT_SECRET=sua-chave-secreta
PG_HOST=seu-host-postgres
PG_PASSWORD=sua-senha
PG_DATABASE=meubanco
SESSION_SECRET=outro-segredo

5. Configurar banco de dados

O Railway oferece plugins de banco de dados:

  • Clique em New → Database
  • Escolha MongoDB ou PostgreSQL
  • As variáveis de conexão são geradas automaticamente
  • Copie os valores para as variáveis de ambiente do seu projeto

6. Deploy automático

Após conectar o repositório, o Railway faz o deploy automaticamente. Cada push na branch conectada (geralmente main) dispara um novo deploy.

O Railway detecta automaticamente que é um projeto Node.js (pelo package.json) e executa:

  1. npm install
  2. npm start

7. Acessar a aplicação

O Railway gera um domínio público automaticamente, como: 

https://seu-projeto.up.railway.app

Teste suas rotas:

curl https://seu-projeto.up.railway.app/produtos

Deploy com Render

O Render segue um processo similar:

  1. Acesse https://render.com e faça login com GitHub
  2. Clique em New Web Service
  3. Conecte seu repositório
  4. Configure:
    • Build Command: npm install
    • Start Command: npm start
  5. Adicione as variáveis de ambiente em Environment
  6. Clique em Create Web Service

Checklist Pré-Deploy

Antes de fazer deploy, verifique:

  1. ✅ .env está no .gitignore
  2. ✅ Não há senhas ou chaves hardcoded no código
  3. ✅ O script start está no package.json
  4. ✅ engines.node está definido no package.json
  5. ✅ Helmet e CORS estão configurados
  6. ✅ Rate limiting está ativo nas rotas sensíveis (login, registro)
  7. ✅ O middleware de erro global está configurado
  8. ✅ A aplicação funciona com NODE_ENV=production
  9. ✅ As variáveis de ambiente estão configuradas na plataforma de deploy
  10. ✅ O banco de dados está acessível da nuvem (IP whitelist se necessário)

Atualize src/app.js:

const express = require("express");
const path = require("path");
const helmet = require("helmet");
const cors = require("cors");
const compression = require("compression");
const rateLimit = require("express-rate-limit");
const cookieParser = require("cookie-parser");
const session = require("express-session");

const usuarioRoutes = require("./routes/usuarioRoutes");
const produtoRoutes = require("./routes/produtoRoutes");
const produtoPgRoutes = require("./routes/produtoPgRoutes");
const arquivoRoutes = require("./routes/arquivoRoutes");
const authRoutes = require("./routes/authRoutes");
const streamRoutes = require("./routes/streamRoutes");
const cepRoutes = require("./routes/cepRoutes");
const sessaoRoutes = require("./routes/sessaoRoutes");
const calculoRoutes = require("./routes/calculoRoutes");

const loggerMiddleware = require("./middlewares/loggerMiddleware");

const app = express();

app.use(helmet());

app.use(cors());

app.use(compression());

const limitadorGeral = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 200,
  message: { erro: "Muitas requisições. Tente novamente mais tarde." },
});

const limitadorAuth = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 10,
  message: { erro: "Muitas tentativas. Aguarde 15 minutos." },
});

app.use(limitadorGeral);

app.use(loggerMiddleware);

app.use(express.static(path.join(__dirname, "../public")));

app.use(cookieParser());

app.use(session({
  secret: process.env.SESSION_SECRET || "segredo-super-seguro",
  resave: false,
  saveUninitialized: false,
  cookie: {
    maxAge: 30 * 60 * 1000,
    httpOnly: true,
    secure: process.env.NODE_ENV === "production",
    sameSite: "lax",
  },
}));

app.use(express.json());
app.use(express.urlencoded({ extended: true }));

app.use(authRoutes);
app.use(produtoRoutes);
app.use(produtoPgRoutes);
app.use(usuarioRoutes);
app.use(arquivoRoutes);
app.use(streamRoutes);
app.use(cepRoutes);
app.use(sessaoRoutes);
app.use(calculoRoutes);

app.use("/registrar", limitadorAuth);
app.use("/login", limitadorAuth);

app.use((erro, req, res, next) => {
  console.error("Erro interno:", erro.message);

  res.status(erro.status || 500).json({
    erro: process.env.NODE_ENV === "production"
      ? "Erro interno do servidor"
      : erro.message,
  });
});

module.exports = app;

🤓 Explicando o código 📝

🔹 helmet()

Adiciona 11 headers de segurança HTTP automaticamente, incluindo: X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, X-XSS-Protection e Content-Security-Policy.

🔹 cors()

Por padrão, navegadores bloqueiam requisições entre origens diferentes. O CORS libera isso de forma controlada. Em produção, restrinja a origin ao domínio do seu frontend.

🔹 compression()

Comprime respostas com gzip, reduzindo o tráfego em até 70%. Isso acelera o carregamento e reduz custos de banda.

🔹 rateLimit

Cria dois limitadores: um geral (200 req/15min) e um para autenticação (10 req/15min). Isso protege contra brute force no login.

🔹 secure e sameSite nos cookies

secure: true em produção garante que cookies só trafeguem em HTTPS. sameSite: "lax" protege contra CSRF.

🔹 Middleware de erro

Em produção, NÃO exponha detalhes internos do erro. Mostre apenas uma mensagem genérica. Em desenvolvimento, mostre o erro completo.
