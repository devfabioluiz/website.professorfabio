Iniciando nosso Projeto Final

Node.js

O Projeto Final

Neste projeto final você criará uma API REST completa que servirá como backend para os projetos finais de React e Angular.

A API deve ser criada dentro de uma branch chamada projeto-final no repositório projeto-node-step.

A API terá os seguintes recursos:

  • Produtos — CRUD completo (listar, obter, criar, atualizar, deletar)
  • Autenticação — registro de usuário, login com JWT e perfil do usuário logado
  • Proteção de rotas — endpoints administrativos protegidos por JWT e verificação de permissão (admin)
  • Banco MongoDB no Atlas — banco de dados profissional grátis na nuvem
  • Deploy no Vercel — publicação automática via GitHub

☁️ PASSO 1 — CRIAR BANCO NO MONGODB ATLAS

Antes de escrever qualquer código, vamos criar o banco de dados. Usaremos o MongoDB Atlas, que oferece MongoDB grátis na nuvem (tier M0). Esse banco será usado tanto em desenvolvimento quanto em produção.

📦 Passo a passo: Criar banco no MongoDB Atlas

  1. Acesse https://mongodb.com/atlas e clique em "Try Free"
  2. Faça login com GitHub ou crie uma conta. Após o login, clique em "Create a cluster"
  3. Escolha o plano M0 Sandbox (grátis), selecione AWS e a região mais próxima (ex: South America — São Paulo)
  4. Clique em "Create Cluster" e aguarde ~3 minutos enquanto o cluster é provisionado
  5. Crie um usuário no banco: vá em Database Access → Add New Database User, defina usuário e senha (guarde essa senha!)
  6. Libere o acesso de IPs: vá em Network Access → Add IP Address e clique em "Allow Access from Anywhere" (0.0.0.0/0)
  7. Obtenha a string de conexão: vá em Database → Connect → Connect your application 
    mongodb+srv://seuusuario:senha@cluster0.xxxxx.mongodb.net/projeto-final?retryWrites=true&w=majority
  8. Cole essa URL em um bloco de notas — você vai usar nos próximos passos. Substitua seuusuario e senha pelos dados que você criou.

Passo 5 — Criar o arquivo de conexão com o banco

Diferente das aulas anteriores, aqui usamos um padrão de conexão cacheada. Isso permite que o MongoDB funcione tanto no servidor local quanto no ambiente serverless do Vercel sem criar múltiplas conexões. 

type nul > src\database\connect.js
const mongoose = require("mongoose");

let cached = global.mongoose;

if (!cached) {
  cached = global.mongoose = { conn: null, promise: null };
}

async function conectar() {
  if (cached.conn) return cached.conn;

  if (!cached.promise) {
    const MONGODB_URL =
      process.env.MONGODB_URL ||
      "mongodb://localhost:27017/projeto-final";

    cached.promise = mongoose.connect(MONGODB_URL, {
      serverSelectionTimeoutMS: 5000,
      bufferCommands: false,
    });
  }

  cached.conn = await cached.promise;
  return cached.conn;
}

async function desconectar() {
  if (cached.conn) {
    await cached.conn.disconnect();
    cached.conn = null;
    cached.promise = null;
  }
}

module.exports = { conectar, desconectar };

🤓 Explicando o código 📝

🔹 Conexão cacheada

Em ambiente serverless (Vercel), cada requisição pode rodar em um processo diferente. O padrão de cache com global.mongoose garante que a conexão seja reutilizada enquanto o processo estiver ativo, evitando criar uma nova conexão a cada requisição.

🔹 serverSelectionTimeoutMS: 5000

Tempo máximo de 5 segundos para conectar ao MongoDB. Se o banco não responder nesse período, a conexão falha rapidamente em vez de ficar esperando indefinidamente.

🔹 bufferCommands: false

Desabilita o buffer de comandos do Mongoose. Em ambiente serverless (Vercel), evita que comandos acumulem enquanto não há conexão, fazendo a requisição falhar imediatamente se o banco estiver offline.

🔹 desconectar()

Função auxiliar para encerrar a conexão manualmente quando necessário (ex: ao final do servidor local ou do script de seed).

🔹 fallback local

Se a variável MONGODB_URL não estiver configurada, o código usa mongodb://localhost:27017/projeto-final como fallback para desenvolvimento local.

    const { id } = req.params;
    const produto = await Produto.findById(id);

  } catch (erro) {
    res.status(500).json({ erro: erro.message });
  }

  try {
    const produto = await Produto.create(req.body);

    const { id } = req.params;

      runValidators: true,
    });

      return res.status(404).json({ erro: "Produto não encontrado" });

    res.json(produto);

Passo 9 — Criar o Controller de Autenticação

type nul > src\controllers\authController.js
const bcrypt = require("bcryptjs");
const jwt = require("jsonwebtoken");
const Usuario = require("../models/Usuario");

const JWT_SECRET = process.env.JWT_SECRET || "segredo";

const registrar = async (req, res) => {
  const { nome, email, senha, role } = req.body;

  const existe = await Usuario.findOne({ email });
  if (existe) {
    return res.status(400).json({ erro: "Email já cadastrado" });
  }

  const senhaHash = await bcrypt.hash(senha, 10);
  const usuario = await Usuario.create({
    nome,
    email,
    senha: senhaHash,
    role: role || "user",
  });

  const token = jwt.sign(
    { id: usuario._id, email: usuario.email, role: usuario.role },
    JWT_SECRET,
    { expiresIn: "7d" },
  );

  res.status(201).json({
    mensagem: "Usuário registrado com sucesso",
    token,
    usuario: {
      id: usuario._id,
      nome: usuario.nome,
      email: usuario.email,
      role: usuario.role,
    },
  });
};

const login = async (req, res) => {
  const { email, senha } = req.body;

  const usuario = await Usuario.findOne({ email });
  if (!usuario) {
    return res.status(401).json({ erro: "Email ou senha inválidos" });
  }

  const senhaValida = await bcrypt.compare(senha, usuario.senha);
  if (!senhaValida) {
    return res.status(401).json({ erro: "Email ou senha inválidos" });
  }

  const token = jwt.sign(
    { id: usuario._id, email: usuario.email, role: usuario.role },
    JWT_SECRET,
    { expiresIn: "7d" },
  );

  res.json({
    mensagem: "Login realizado com sucesso",
    token,
    usuario: {
      id: usuario._id,
      nome: usuario.nome,
      email: usuario.email,
      role: usuario.role,
    },
  });
};

const perfil = async (req, res) => {
  const usuario = await Usuario.findById(req.usuarioId).select("-senha");
  if (!usuario) {
    return res.status(404).json({ erro: "Usuário não encontrado" });
  }
  res.json({
    usuario: {
      id: usuario._id,
      nome: usuario.nome,
      email: usuario.email,
      role: usuario.role,
    },
  });
};

module.exports = { registrar, login, perfil };

Passo 10 — Criar o Middleware de Autenticação

type nul > src\middlewares\autenticar.js
const jwt = require("jsonwebtoken");

const JWT_SECRET = process.env.JWT_SECRET || "segredo";

const autenticar = (req, res, next) => {
  const authHeader = req.headers.authorization;

  if (!authHeader) {
    return res.status(401).json({ erro: "Token não informado" });
  }

  const token = authHeader.split(" ")[1];

  try {
    const payload = jwt.verify(token, JWT_SECRET);
    req.usuarioId = payload.id;
    req.usuarioEmail = payload.email;
    req.usuarioRole = payload.role;
    next();
  } catch (erro) {
    return res.status(401).json({ erro: "Token inválido ou expirado" });
  }
};

module.exports = autenticar;

Passo 14 — Criar o app.js (aplicação Express)

O app.js configura o Express com todos os middlewares de segurança e as rotas. Este arquivo é compartilhado entre o servidor local e o Vercel. 

type nul > app.js
const express = require("express");
const helmet = require("helmet");
const cors = require("cors");
const compression = require("compression");
const rateLimit = require("express-rate-limit");
const { conectar } = require("./src/database/connect");

const produtoRoutes = require("./src/routes/produtoRoutes");
const authRoutes = require("./src/routes/authRoutes");

const app = express();

// ─── Middlewares de Segurança ─────────────────────────
app.use(helmet());
app.use(cors({ origin: process.env.CORS_ORIGIN || "*" }));
app.use(compression());

// ─── Rate Limiting ────────────────────────────────────
const limitador = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  message: { erro: "Muitas requisições. Tente novamente mais tarde." },
});
app.use("/api/", limitador);

app.use(express.json());

// ─── Conexão com MongoDB ───────────────────────────────
app.use(async (req, res, next) => {
  try {
    await conectar();
    next();
  } catch (erro) {
    console.error("Falha na conexão MongoDB:", erro.message);
    res.status(500).json({
      erro: "Erro ao conectar ao banco de dados",
      detalhe: erro.message,
    });
  }
});

// ─── Rota Raiz ────────────────────────────────────────
app.get("/", (req, res) => {
  res.json({
    mensagem: "API do Projeto Final funcionando!",
    versao: "1.0.0",
    endpoints: {
      produtos: {
        listar: "GET /produtos",
        obter: "GET /produtos/:id",
        criar: "POST /produtos",
        atualizar: "PUT /produtos/:id",
        deletar: "DELETE /produtos/:id",
      },
      autenticacao: {
        registrar: "POST /registrar",
        login: "POST /login",
        perfil: "GET /me",
      },
    },
  });
});

// ─── Rotas da API ─────────────────────────────────────
app.use(produtoRoutes);
app.use(authRoutes);

// ─── Middleware de Erro Global ─────────────────────────
app.use((erro, req, res, next) => {
  console.error("Erro interno:", erro.message);
  res.status(erro.status || 500).json({
    erro:
      process.env.NODE_ENV === "production"
        ? "Erro interno do servidor"
        : erro.message,
  });
});

module.exports = app;

🤓 Explicando o código 📝

🔹 helmet()

Adiciona automaticamente headers de segurança HTTP que protegem contra XSS, clickjacking, sniffing de MIME type e outras vulnerabilidades comuns.

🔹 cors()

Por padrão, navegadores bloqueiam requisições entre origens diferentes. O CORS libera isso de forma controlada. Em produção, restrinja a origin ao domínio do seu frontend.

🔹 compression()

Comprime respostas com gzip, reduzindo o tráfego em até 70%. Isso acelera o carregamento e reduz custos de banda.

🔹 rateLimit()

Limita a 100 requisições por 15 minutos. Protege contra ataques de brute force e DoS.

🔹 Middleware de conexão MongoDB

Um middleware global que chama conectar() em toda requisição. Em ambiente serverless (Vercel), como o processo pode ser reutilizado, a conexão cacheada no global.mongoose evita criar múltiplas conexões. Se o banco estiver offline, retorna erro 500 imediatamente.

🔹 Middleware de erro

Captura qualquer erro não tratado nos controllers. Em produção, não expõe detalhes internos do erro.

require("dotenv").config();

const { conectar, desconectar } = require("./src/database/connect");
const app = require("./app");

const PORT = process.env.PORT || 3000;

conectar()
  .then(() => {
    console.log("MongoDB conectado com sucesso");
    app.listen(PORT, () => {
      console.log(`Servidor rodando em http://localhost:${PORT}`);

  .catch((erro) => {
    console.error("Erro ao conectar no MongoDB:", erro.message);
    process.exit(1);
  });

desconectar()
  .then(() => {
    console.log("MongoDB desconectado com sucesso");
  })
  .catch((erro) => {
    console.error("Erro ao desconectar do MongoDB:", erro.message);
  });

module.exports = app;

  await Usuario.create({ nome: "Usuário Teste", email: "usuario@email.com", senha: senhaHash, role: "user" });

  console.log("Usuários criados!");
  console.log("Admin: admin@email.com / senha: admin123");
  console.log("User:  usuario@email.com / senha: admin123");
  console.log("Banco populado com sucesso!");

  await mongoose.disconnect();
}

seed().catch((erro) => {
  console.error("Erro ao popular banco:", erro);
  process.exit(1);
});

3. Helmet — Segurança HTTP Headers

O helmet já está configurado no app.js. Ele adiciona automaticamente 11 headers de segurança, incluindo: X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, X-XSS-Protection e Content-Security-Policy.

4. CORS — Controle de Acesso

O cors já está configurado no app.js. Em produção, restrinja a origin ao domínio do seu frontend via variável de ambiente CORS_ORIGIN.

5. Compression — Reduzindo o Tamanho das Respostas

O compression já está configurado no app.js. Ele comprime respostas com gzip, reduzindo o tráfego em até 70%.

6. Rate Limiting — Proteção contra Abusos

O express-rate-limit já está configurado no app.js com limite de 100 requisições a cada 15 minutos.

7. Tratamento de Erros Global

O Node.js permite capturar erros não tratados em todo o processo: 

process.on("uncaughtException", (erro) => {
  console.error("Erro não tratado:", erro);
  process.exit(1);
});

process.on("unhandledRejection", (erro) => {
  console.error("Promise rejeitada sem catch:", erro);
  process.exit(1);
});

8. Middleware de Erro no Express

O middleware de erro global já está configurado no app.js. O Express reconhece middlewares com 4 parâmetros como error handlers. Eles só são executados quando next(erro) é chamado.
