RxJS e Operadores na Prática

O que é RxJS?

RxJS é uma biblioteca para trabalhar com dados assíncronos usando Observables.

Diferente de valores simples, um Observable pode emitir vários valores ao longo do tempo.

Exemplo simples




import { of } from 'rxjs';

of(1, 2, 3).subscribe(console.log);

Transformando dados com operadores

No mundo real, raramente usamos Observable puro. Usamos operadores.

map




import { of, map } from 'rxjs';

of(1, 2, 3)
.pipe(map(v => v * 10))
.subscribe(console.log);

filter




import { of, filter } from 'rxjs';

of(1,2,3,4)
.pipe(filter(v => v % 2 === 0))
.subscribe(console.log);

tap (debug)




import { of, tap } from 'rxjs';

of(1,2,3)
.pipe(tap(v => console.log('antes:', v)))
.subscribe();

Controlando eventos (debounceTime)

Evita execuções excessivas (muito usado em busca). 




debounceTime(500)

Chamadas HTTP com switchMap

O switchMap cancela chamadas anteriores e mantém apenas a última. 




switchMap(termo => this.http.get(`/api?q=${termo}`))

import { Component } from '@angular/core';
import { FormControl } from '@angular/forms';
import { debounceTime, switchMap, tap } from 'rxjs';
import { HttpClient } from '@angular/common/http';

@Component({
selector: 'app-busca-produtos',
templateUrl: './busca-produtos.html'
})
export class BuscaProdutos {
campoBusca = new FormControl('');
resultados: any[] = [];

constructor(private http: HttpClient) {
this.campoBusca.valueChanges
.pipe(
debounceTime(500),
tap(valor => console.log('Buscando:', valor)),
switchMap(valor =>
this.http.get(`https://fakestoreapi.com/products`)
)
)
.subscribe(res => {
this.resultados = res;
});
}
}

O que foi usado aqui?

  • debounceTime → evita excesso
  • tap → debug
  • switchMap → chamada HTTP segura

Resumo

  • RxJS é essencial para Angular
  • Observables representam fluxos de dados
  • Operadores transformam e controlam esses fluxos
  • debounceTime + switchMap = padrão de mercado

Signals vs RxJS — Quando usar cada um?

Com as versões mais recentes do Angular, surgiu uma dúvida muito comum:

Signals substituem RxJS?

Resposta curta: NÃO.

Eles resolvem problemas diferentes e, na prática, são usados juntos.

Diferença principal

Signals RxJS
Estado local e reatividade simples Fluxos assíncronos complexos
Mais simples e direto Mais poderoso e flexível
Atualização automática Baseado em streams

Quando usar Signals

  • Estado de UI (contador, toggle, loading)
  • Dados locais de um componente
  • Substituir variáveis simples
  • Reatividade direta sem complexidade

contador = signal(0);

incrementar() {
  this.contador.update(v => v + 1);
}

Quando usar RxJS

  • Chamadas HTTP
  • Eventos (input, scroll, clique)
  • Fluxos assíncronos complexos
  • Combinar múltiplas fontes de dados

this.campoBusca.valueChanges.pipe(
  debounceTime(300),
  switchMap(valor => this.http.get(`/api?q=${valor}`))
);

Comparação prática

Imagine um campo de busca:

  • Signal: guarda o valor atual
  • RxJS: controla o fluxo de busca (tempo, requisições, etc)

Usando os dois juntos

Em aplicações reais, é comum usar RxJS para buscar dados e Signals para armazenar o estado. 


resultados = signal([]);

this.campoBusca.valueChanges.pipe(
  debounceTime(300),
  switchMap(valor => this.http.get(`/api?q=${valor}`))
).subscribe(res => {
  this.resultados.set(res);
});

Resumo prático

  • Signals = estado simples
  • RxJS = fluxo assíncrono
  • Signals não substituem RxJS
  • RxJS não substitui Signals
  • Os dois juntos = arquitetura moderna Angular
