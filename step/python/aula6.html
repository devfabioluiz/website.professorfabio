Passo 1 — ORM: Object-Relational Mapping

ORM é a ponte entre o mundo orientado a objetos (Python) e o mundo relacional (banco de dados SQL). Em vez de escrever SELECT * FROM usuarios WHERE nome = 'João', você escreve Usuario.objects.filter(nome='João').

O Django traduz automaticamente suas chamadas Python para SQL. Isso significa que o mesmo código funciona com SQLite, PostgreSQL, MySQL, Oracle, etc. — sem mudar nada.

1.1 — O Model (tabela no banco)

Vamos criar um model mais completo no arquivo usuarios/models.py. Apague o conteúdo antigo e coloque: 


from django.db import models

class Categoria(models.Model):
    nome = models.CharField(max_length=50)

    def __str__(self):
        return self.nome

class Usuario(models.Model):
    nome = models.CharField(max_length=100)
    email = models.EmailField(unique=True)
    idade = models.IntegerField(default=0)
    ativo = models.BooleanField(default=True)
    categoria = models.ForeignKey(Categoria, on_delete=models.CASCADE, null=True, blank=True)
    criado_em = models.DateTimeField(auto_now_add=True)

    def __str__(self):
        return self.nome

Explicação dos campos:

  • CharField(max_length=50) — texto curto (VARCHAR no SQL)
  • EmailField(unique=True) — e-mail com validação e índice único
  • IntegerField(default=0) — número inteiro com valor padrão
  • BooleanField(default=True) — verdadeiro/falso
  • ForeignKey(Categoria, on_delete=models.CASCADE) — relacionamento 1:N (um Usuario pertence a uma Categoria)
  • DateTimeField(auto_now_add=True) — data/hora preenchida automaticamente na criação

Dentro do shell, teste cada operação:


from usuarios.models import Usuario, Categoria

# ========== CREATE (criar) ==========
# Criando uma categoria
cat_dev = Categoria.objects.create(nome="Desenvolvedores")
cat_design = Categoria.objects.create(nome="Designers")

# Criando usuários
Usuario.objects.create(nome="Ana", email="ana@email.com", idade=25, categoria=cat_dev)
Usuario.objects.create(nome="Bruno", email="bruno@email.com", idade=30, categoria=cat_dev)
Usuario.objects.create(nome="Carla", email="carla@email.com", idade=22, categoria=cat_design)
Usuario.objects.create(nome="Diego", email="diego@email.com", idade=35, categoria=cat_design)
Usuario.objects.create(nome="Eduarda", email="eduarda@email.com", idade=28, categoria=cat_dev)

# ========== READ (ler / consultar) ==========

# all() — todos os registros
todos = Usuario.objects.all()
print(todos)   # QuerySet com todos os usuários

# get() — um único registro (lança erro se não encontrar)
ana = Usuario.objects.get(nome="Ana")
print(ana.email)   # ana@email.com

# filter() — filtra com condições
devs = Usuario.objects.filter(categoria__nome="Desenvolvedores")
print(devs)   # Ana, Bruno, Eduarda

# exclude() — exclui da consulta
nao_devs = Usuario.objects.exclude(categoria__nome="Desenvolvedores")
print(nao_devs)   # Carla, Diego

# first() e last() — primeiro e último registro
primeiro = Usuario.objects.first()
ultimo = Usuario.objects.last()

# count() — conta registros
total = Usuario.objects.count()
print(total)   # 5

# exists() — verifica se existe
tem_admin = Usuario.objects.filter(nome="Admin").exists()
print(tem_admin)   # False

# order_by() — ordena
por_idade = Usuario.objects.all().order_by('idade')   # crescente
por_idade_desc = Usuario.objects.all().order_by('-idade')   # decrescente

# values() — retorna dicionários (mais leve, sem objetos)
nomes = Usuario.objects.values('nome', 'email')
for item in nomes:
    print(item)   # {'nome': 'Ana', 'email': 'ana@email.com'}

# values_list() — retorna tuplas
lista_plana = Usuario.objects.values_list('nome', flat=True)
print(list(lista_plana))   # ['Ana', 'Bruno', 'Carla', 'Diego', 'Eduarda']

# ========== UPDATE (atualizar) ==========

# Método 1: instância
ana = Usuario.objects.get(nome="Ana")
ana.idade = 26
ana.save()

# Método 2: update() no QuerySet (mais eficiente para vários)
Usuario.objects.filter(categoria__nome="Designers").update(idade=40)

# ========== DELETE (deletar) ==========

# Método 1: instância
diego = Usuario.objects.get(nome="Diego")
diego.delete()

# Método 2: delete() no QuerySet
Usuario.objects.filter(idade__gt=30).delete()   # deleta quem tem idade > 30

# ========== FILTROS AVANÇADOS ==========

# __gt (greater than / maior que)
maiores = Usuario.objects.filter(idade__gt=25)

# __gte (greater than or equal / maior ou igual)
maiores_ou_25 = Usuario.objects.filter(idade__gte=25)

# __lt (less than / menor que)
menores = Usuario.objects.filter(idade__lt=25)

# __lte (less than or equal / menor ou igual)
menores_ou_25 = Usuario.objects.filter(idade__lte=25)

# __contains (contém — case sensitive)
contem_na = Usuario.objects.filter(nome__contains="na")

# __icontains (contém — case insensitive)
contem_na_i = Usuario.objects.filter(nome__icontains="NA")

# __startswith / __endswith (começa com / termina com)
comeca_com_a = Usuario.objects.filter(nome__startswith="A")

# __in (está na lista)
ids = [1, 3, 5]
usuarios_por_id = Usuario.objects.filter(id__in=ids)

# __range (intervalo — funciona com datas e números)
entre_20_e_30 = Usuario.objects.filter(idade__range=(20, 30))

# _______________ AGREGACOES _______________

from django.db.models import Count, Avg, Max, Min, Sum

# Quantos usuários por categoria
por_categoria = Categoria.objects.annotate(total=Count('usuario'))
for cat in por_categoria:
    print(f"{cat.nome}: {cat.total} usuários")

# Média de idade
media = Usuario.objects.aggregate(media_idade=Avg('idade'))
print(media)   # {'media_idade': 28.0}

# idade máxima, mínima e soma
estatisticas = Usuario.objects.aggregate(
    maximo=Max('idade'),
    minimo=Min('idade'),
    soma=Sum('idade')
)
print(estatisticas)

Passo 2 — Admin do Django

O Django Admin é um painel administrativo completo gerado automaticamente. Ele lê seus models e cria telas de CRUD com filtros, buscas, ordenação e paginação — tudo sem você escrever HTML. No mercado de trabalho, usamos o Admin para:

  • Gerenciar dados internamente (equipe de suporte, por exemplo)
  • Fazer testes manuais durante o desenvolvimento
  • Prototipar rapidamente um backoffice
  • Visualizar dados sem abrir o banco diretamente

2.1 — Registrar models no Admin

Arquivo: usuarios/admin.py


from django.contrib import admin
from .models import Usuario, Categoria

# Registro simples
admin.site.register(Categoria)

# Registro com personalização
@admin.register(Usuario)
class UsuarioAdmin(admin.ModelAdmin):
    # Colunas na listagem
    list_display = ('id', 'nome', 'email', 'idade', 'ativo', 'categoria', 'criado_em')

    # Filtros laterais
    list_filter = ('ativo', 'categoria', 'criado_em')

    # Campo de busca
    search_fields = ('nome', 'email')

    # Campos clicáveis na listagem
    list_display_links = ('id', 'nome')

    # Paginação: quantos itens por página
    list_per_page = 20

    # Ordenação padrão
    ordering = ('-criado_em',)

    # Campos somente leitura (não editáveis)
    readonly_fields = ('criado_em',)

    # Organizar campos no formulário de edição
    fieldsets = (
        ('Dados Pessoais', {
            'fields': ('nome', 'email', 'idade', 'categoria')
        }),
        ('Status', {
            'fields': ('ativo', 'criado_em')
        }),
    )

Passo 3 — Templates (aprofundamento)

Na Aula 4 criamos um template simples. Agora vamos explorar o poder real do sistema de templates do Django, começando pela funcionalidade mais importante para projetos reais: herança de templates.

3.1 — Template base (herança)

A ideia é criar um layout comum (cabeçalho, menu, rodapé) e fazer cada página apenas preencher o conteúdo específico. Sem herança, você repetiria HTML em toda página. Com herança, o layout fica em um só lugar.

Crie o arquivo usuarios/templates/base.html: 


<!DOCTYPE html>
<html lang="pt-BR">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>{% block title %}Meu Site{% endblock %}</title>
</head>
<body>
    <header>
        <h1>Sistema de Usuários</h1>
        <nav>
            <a href="/">Home</a> |
            <a href="/usuarios/">Lista</a> |
            {% if user.is_authenticated %}
                Olá, {{ user.username }} |
                <a href="/logout/">Sair</a>
            {% else %}
                <a href="/login/">Entrar</a>
            {% endif %}
        </nav>
        <hr>
    </header>

    <main>
        {% block content %}
        <p>Conteúdo padrão (será substituído pelas páginas filhas).</p>
        {% endblock %}
    </main>

    <footer>
        <hr>
        <p>&copy; 2025 - Todos os direitos reservados.</p>
    </footer>
</body>
</html>

3.3 — Template tags na View (como passamos dados)

No render(), o terceiro argumento é o contexto — um dicionário Python cujas chaves viram variáveis dentro do template. 


def listar_template(request):
    usuarios = Usuario.objects.all().order_by('nome')
    total = usuarios.count()
    hoje = date.today()

    contexto = {
        'usuarios': usuarios,
        'total': total,
        'hoje': hoje,
        'titulo': 'Lista de Usuários',
    }
    return render(request, 'usuarios/lista.html', contexto)

Passo 4 — Login e Autenticação

O Django tem um sistema de autenticação completo e pronto para uso: login, logout, criação de usuários, grupos, permissões. Tudo já incluso via django.contrib.auth. Você só precisa criar as views e templates.

Adicione no arquivo usuarios/urls.py:

4.2 — Criar as views de autenticação

Adicione no arquivo usuarios/views.py (mantendo o que já existe):


from django.shortcuts import render, redirect
from django.contrib.auth import authenticate, login, logout
from django.contrib.auth.decorators import login_required
from django.contrib import messages
from rest_framework.decorators import api_view
from rest_framework.response import Response
from .models import Usuario
from .serializers import UsuarioSerializer

# ... (manter listar_usuarios, criar_usuario, listar_template) ...

def home(request):
    return render(request, 'base.html')

def fazer_login(request):
    if request.method == 'POST':
        username = request.POST.get('username')
        password = request.POST.get('password')

        user = authenticate(request, username=username, password=password)

        if user is not None:
            login(request, user)
            messages.success(request, f'Bem-vindo, {user.username}!')
            return redirect('lista_usuarios')
        else:
            messages.error(request, 'Usuário ou senha inválidos.')

    return render(request, 'usuarios/login.html')

def fazer_logout(request):
    logout(request)
    messages.info(request, 'Você saiu do sistema.')
    return redirect('home')

4.4 — Protegendo views com @login_required

O decorator @login_required bloqueia o acesso de usuários não autenticados. Se alguém tentar acessar a página sem login, é redirecionado automaticamente para a página de login. 


from django.contrib.auth.decorators import login_required

@login_required(login_url='/login/')
def listar_template(request):
    usuarios = Usuario.objects.all()
    contexto = {'usuarios': usuarios}
    return render(request, 'usuarios/lista.html', contexto)

Note que definimos login_url='/login/' para que o Django saiba para onde enviar o usuário não autenticado.

Passo 5 — Permissões

Autenticação diz quem é o usuário. Permissão diz o que ele pode fazer. O Django tem um sistema de permissões baseado em models: cada model gera automaticamente 3 permissões — add (criar), change (editar) e delete (deletar).

5.1 — Verificar permissão na view


from django.contrib.auth.decorators import permission_required
from django.contrib.auth.models import User

# Permite acesso apenas a quem tem a permissão especificada
@permission_required('usuarios.add_usuario', login_url='/login/')
def criar_usuario_view(request):
    if request.method == 'POST':
        # lógica de criação
        pass
    return render(request, 'usuarios/form.html')

O formato da permissão é: app_label.codename. Exemplos:

  • usuarios.add_usuario — pode criar usuário
  • usuarios.change_usuario — pode editar usuário
  • usuarios.delete_usuario — pode deletar usuário
  • usuarios.view_usuario — pode visualizar usuário

5.2 — Verificar permissão no template


{% if perms.usuarios.add_usuario %}
    <a href="/admin/usuarios/usuario/add/">Adicionar Usuário</a>
{% endif %}

{% if perms.usuarios.delete_usuario %}
    <button>Excluir</button>

5.3 — Grupos (atribuir permissões em lote)

Em projetos reais, você não atribui permissões usuário por usuário. Você cria grupos (ex: "Editores", "Administradores") e associa usuários a grupos. Grupos recebem permissões; usuários herdam do grupo. 


from django.contrib.auth.models import Group, Permission

# Criar um grupo
editores, created = Group.objects.get_or_create(name='Editores')

# Buscar permissões específicas
perm_add = Permission.objects.get(codename='add_usuario')
perm_change = Permission.objects.get(codename='change_usuario')
perm_view = Permission.objects.get(codename='view_usuario')

# Atribuir permissões ao grupo
editores.permissions.add(perm_add, perm_change, perm_view)

# Adicionar um usuário existente ao grupo
user = User.objects.get(username='maria')
user.groups.add(editores)

# Verificar se usuário tem permissão (via grupo ou direta)
print(user.has_perm('usuarios.add_usuario'))   # True
print(user.has_perm('usuarios.delete_usuario'))   # False

No admin do Django, vá em "Groups" para gerenciar grupos e permissões graficamente, sem código.

5.4 — Programaticamente na view (sem decorator)


def deletar_usuario(request, pk):
    if not request.user.has_perm('usuarios.delete_usuario'):
        messages.error(request, 'Você não tem permissão para excluir.')
        return redirect('lista_usuarios')

    usuario = get_object_or_404(Usuario, pk=pk)
    usuario.delete()
    messages.success(request, 'Usuário excluído.')
    return redirect('lista_usuarios')

Passo 6 — Sessões

Uma sessão é um pequeno armazenamento temporário no servidor vinculado ao navegador do usuário por um cookie. Serve para guardar dados entre requisições, como carrinho de compras, preferências ou etapa de um formulário longo.

O Django já vem com django.contrib.sessions instalado (está no INSTALLED_APPS). Funciona sem configuração adicional.

6.1 — Ler e escrever na sessão


# ESCREVER na sessão (como um dicionário)
request.session['tema'] = 'escuro'
request.session['visitou_pagina'] = True
request.session['carrinho'] = ['produto1', 'produto2']

# LER da sessão
tema = request.session.get('tema', 'claro')   # 'claro' é o valor padrão
carrinho = request.session.get('carrinho', [])

# DELETAR um valor específico
if 'tema' in request.session:
    del request.session['tema']

# LIMPAR toda a sessão
request.session.flush()

# Definir tempo de expiração (em segundos)
request.session.set_expiry(3600)   # expira em 1 hora

6.2 — Contador de visitas (exemplo prático)


def home(request):
    # Incrementa o contador a cada visita
    visitas = request.session.get('visitas', 0)
    visitas += 1
    request.session['visitas'] = visitas

    return render(request, 'base.html', {'visitas': visitas})

No template base.html:


{% if visitas %}
    <p>Você visitou esta página {{ visitas }} vez(es) nesta sessão.</p>
{% endif %}

6.3 — Onde as sessões são armazenadas?

Por padrão, o Django armazena sessões no banco de dados (tabela django_session). Você pode mudar isso no settings.py com a variável SESSION_ENGINE. Opções comuns:

EngineOnde armazenaQuando usar
django.contrib.sessions.backends.dbBanco de dados (padrão)Uso geral
django.contrib.sessions.backends.cacheCache (Redis, Memcached)Alta performance
django.contrib.sessions.backends.fileArquivos no discoSem banco de dados
django.contrib.sessions.backends.signed_cookiesNo cookie do navegadorSem armazenamento no servidor

Passo 7 — Migrations

Migrations são arquivos que descrevem mudanças no banco de dados (criar tabelas, adicionar colunas, criar índices, etc.) escritos em Python. Elas são versionáveis (vão pro Git) e permitem que todos os desenvolvedores do time tenham o banco sincronizado.

Sempre que você altera um model, precisa gerar e aplicar uma migration. Esse fluxo de 2 passos é algo que você fará todos os dias como dev Django.

7.1 — O ciclo de vida de uma migration

  1. Você altera um model em models.py
  2. Gera a migration: python manage.py makemigrations
  3. Aplica a migration: python manage.py migrate

7.2 — Comandos essenciais

ComandoO que faz
python manage.py makemigrationsGera arquivos de migration baseados nas mudanças dos models
python manage.py migrateAplica as migrations pendentes no banco
python manage.py showmigrationsLista todas as migrations e quais foram aplicadas
python manage.py sqlmigrate usuarios 0001Mostra o SQL que aquela migration executa
python manage.py migrate usuarios 0001Volta (ou avança) para uma migration específica
python manage.py migrate usuarios zeroDesfaz TODAS as migrations do app usuarios

7.3 — Exemplo prático completo

Situação: você precisa adicionar um campo telefone ao model Usuario.

Passo 1: Edite usuarios/models.py e adicione:


class Usuario(models.Model):
    # ... campos existentes ...
    telefone = models.CharField(max_length=20, blank=True, null=True)

Passo 2: Gere a migration:


python manage.py makemigrations

Saída esperada:


Migrations for 'usuarios':
  usuarios\migrations\0002_usuario_telefone.py
    - Add field telefone to usuario

Passo 3: Veja qual SQL será executado:


python manage.py sqlmigrate usuarios 0002

Saída (com SQLite):


ALTER TABLE "usuarios_usuario" ADD COLUMN "telefone" varchar(20) NULL;

Passo 4: Aplique no banco:


python manage.py migrate

Passo 5: Confira que foi aplicada:


python manage.py showmigrations usuarios

7.4 — Cuidados importantes com migrations

  • Nunca delete arquivos de migration que já foram aplicados em produção. Isso quebra o histórico.
  • Sempre faça commit das migrations no Git. Elas são código e precisam ser versionadas.
  • Se der conflito de migration no merge, resolva manualmente ou use python manage.py makemigrations --merge.
  • Para reverter uma migration não aplicada em produção, delete o arquivo .py da migration e rode makemigrations novamente.
  • Não edite migrations manualmente a menos que saiba exatamente o que está fazendo. O Django gera migrations corretas automaticamente na maioria dos casos.

Dever de Casa

  1. Expandir o Admin: adicione ao UsuarioAdmin as ações em massa para ativar e desativar usuários. Teste selecionando vários usuários na listagem do admin.
  2. Template com herança: crie uma nova página /usuarios/{{ id }}/ que exiba os detalhes de um único usuário (nome, e-mail, idade, categoria, data de criação). Use herança de template (extendendo base.html).
  3. Proteger a página de detalhes: use o decorator @login_required na view que você criou no exercício 2. Teste acessando sem login e depois logado.
