1. História do PostgreSQL

O PostgreSQL é um dos sistemas de banco de dados relacionais de código aberto mais avançados do mundo. Sua história é marcada por décadas de pesquisa acadêmica e desenvolvimento comunitário:

  • 1986: O projeto POSTGRES nasce na Universidade da Califórnia em Berkeley, liderado pelo professor Michael Stonebraker (vencedor do Prêmio Turing 2014). O objetivo era superar as limitações do modelo relacional tradicional, introduzindo suporte a tipos de dados complexos, regras e herança de tabelas.
  • 1994: Andrew Yu e Jolly Chen desenvolvem o Postgres95, substituindo a linguagem de consulta original POSTQUEL por SQL. O nome é alterado para refletir a compatibilidade SQL.
  • 1996: O projeto é renomeado para PostgreSQL (o "SQL" no final enfatiza o suporte nativo ao padrão SQL). A primeira versão de código aberto (6.0) é lançada, marcando o início da era comunitária.
  • 1997-presente: O PostgreSQL Global Development Group mantém e evolui o projeto, com lançamentos anuais de novas versões principais. Cada release introduz funcionalidades de nível empresarial: MVCC, WAL, PITR, replicação, particionamento nativo, índices avançados (GIN, GiST, BRIN, SP-GiST), busca full-text, JSONB, window functions, CTEs recursivas e muito mais.

Filosofia: O PostgreSQL segue o padrão SQL mais fielmente que qualquer outro SGBD, é extensível (tipos, funções, operadores, linguagens procedurais customizados), é ACID-compliant e é 100% gratuito sob a licença PostgreSQL (similar à MIT).

Hoje o PostgreSQL é adotado por gigantes como Apple, Spotify, Instagram, Reddit, Skype, Uber, Netflix e milhares de outras organizações, sendo frequentemente citado como o DBMS que mais cresce em popularidade segundo rankings como DB-Engines e StackOverflow Developer Survey.

2. Instalando o PostgreSQL

O PostgreSQL está disponível para Linux, Windows, macOS e outras plataformas. Veremos a instalação no Windows usando o instalador oficial da EDB (EnterpriseDB).

Passo a Passo no Windows

  1. Acesse postgresql.org/download/windows
  2. Clique em "Download the installer" — isso redireciona para o instalador mantido pela EnterpriseDB (EDB)
  3. Selecione a versão mais recente estável (ex: PostgreSQL 16.x)
  4. Execute o instalador postgresql-16.x-windows-x64.exe
  5. Diretório de instalação: Padrão C:\Program Files\PostgreSQL\16
  6. Componentes: Selecione PostgreSQL Server, pgAdmin 4, Command Line Tools (psql, pg_dump, etc.) e Stack Builder
  7. Diretório de dados (data directory): Padrão C:\Program Files\PostgreSQL\16\data
  8. Senha do superusuário: Defina e guarde com segurança — é a senha do usuário postgres
  9. Porta: Padrão 5432
  10. Locale: Selecione Portuguese, Brazil (ou pt_BR)
  11. Conclua a instalação e, opcionalmente, execute o Stack Builder para instalar extensões adicionais

Verificando a Instalação

Após a instalação, abra o Prompt de Comando ou PowerShell e execute: 


# Verificar versão do psql
psql --version

# Verificar se o servidor está respondendo
pg_isready

# Conectar ao banco padrão como superusuário
psql -U postgres

# Dentro do psql, verificar a versão do servidor
SELECT version();

Instalação no Linux (Ubuntu/Debian)


# Adicionar repositório oficial
sudo sh -c 'echo "deb https://apt.postgresql.org/pub/repos/apt $(lsb_release -cs)-pgdg main" > /etc/apt/sources.list.d/pgdg.list'
wget --quiet -O - https://www.postgresql.org/media/keys/ACCC4CF8.asc | sudo apt-key add -

# Atualizar e instalar
sudo apt update
sudo apt install postgresql-16 postgresql-client-16

# Iniciar e habilitar o serviço
sudo systemctl start postgresql
sudo systemctl enable postgresql

# Verificar status
sudo systemctl status postgresql

Instalação no macOS (via Homebrew)


# Instalar PostgreSQL
brew install postgresql@16

# Iniciar o serviço
brew services start postgresql@16

# Verificar
pg_isready

3. Configuração do Servidor

O PostgreSQL utiliza três arquivos principais de configuração, localizados no diretório de dados (PGDATA) por padrão:

postgresql.conf — Configurações do Servidor

Arquivo principal de parâmetros de runtime. As configurações mais importantes são:

Parâmetro Descrição Valor Padrão
listen_addresses Endereços IP nos quais o servidor escuta (use '*' para todas as interfaces) 'localhost'
port Porta TCP do servidor 5432
max_connections Número máximo de conexões simultâneas 100
shared_buffers Memória compartilhada para cache de dados (recomendação: 25% da RAM) 128MB
effective_cache_size Estimativa da memória disponível para cache do SO (usada pelo planner) 4GB
work_mem Memória por operação (ordenamento, hash). Aumente para queries complexas 4MB
maintenance_work_mem Memória para operações de manutenção (VACUUM, CREATE INDEX) 64MB
wal_level Nível de detalhamento do WAL: minimal, replica ou logical replica
wal_buffers Memória para buffer do WAL -1 (auto)
checkpoint_timeout Intervalo máximo entre checkpoints 5min
log_statement Quais statements logar: none, ddl, mod, all 'none'
log_destination Destino dos logs: stderr, csvlog, syslog, eventlog 'stderr'
timezone Fuso horário do servidor 'GMT'

Dica prática: Use o site PGTune para gerar configurações otimizadas baseadas no hardware disponível.

pg_hba.conf — Autenticação Baseada em Host

O arquivo pg_hba.conf controla quem pode se conectar, de onde e com qual método de autenticação. Cada linha segue o formato: 

TYPE  DATABASE  USER  ADDRESS  METHOD

Métodos de autenticação disponíveis:

Método Descrição
trust Sem senha — qualquer um conecta (apenas desenvolvimento local!)
md5 Senha com hash MD5 (compatibilidade com versões antigas)
scram-sha-256 Senha com SCRAM-SHA-256 (recomendado, mais seguro)
peer Autentica pelo usuário do SO (apenas conexões locais Unix/Linux)
cert Autenticação via certificado SSL/TLS
reject Rejeita conexão incondicionalmente
password Senha em texto plano (evitar — inseguro)
ldap Autenticação via servidor LDAP
gss Autenticação via GSSAPI (Kerberos)

Exemplo de configuração pg_hba.conf típica:


# TYPE  DATABASE        USER            ADDRESS                 METHOD

# Conexões locais (Unix socket) — peer para usuário SO postgres
local   all             postgres                                peer

# Conexões locais para outros usuários — senha scram
local   all             all                                     scram-sha-256

# Conexões remotas IPv4 do mesmo host — senha scram
host    all             all             127.0.0.1/32            scram-sha-256

# Conexões remotas IPv6 do mesmo host — senha scram
host    all             all             ::1/128                 scram-sha-256

# Conexões da rede local (192.168.1.0/24) — senha scram
host    all             all             192.168.1.0/24          scram-sha-256

# Conexões de um IP específico para banco de produção
host    producao        app_user        10.0.0.50/32            scram-sha-256

# Bloquear tudo que não foi explicitamente permitido (opcional)
host    all             all             all                     reject

Após alterar o pg_hba.conf, recarregue a configuração sem reiniciar o servidor: 


# Recarregar configuração (sem interromper conexões)
pg_ctl reload

# Ou via SQL
SELECT pg_reload_conf();

pg_ident.conf — Mapeamento de Nomes de Usuário

Permite mapear nomes de usuário do sistema operacional para nomes de usuário do PostgreSQL quando o método de autenticação peer é utilizado: 


# MAPNAME       SYSTEM-USERNAME         PG-USERNAME
mapa_dev        fabio                   postgres
mapa_dev        dev1                    postgres
mapa_app        apprunner               app_user

Para usar o mapeamento, adicione a opção map=mapa_dev na linha correspondente do pg_hba.conf.

Verificando Configurações Atuais


-- Ver todos os parâmetros de configuração
SHOW ALL;

-- Ver parâmetro específico
SHOW shared_buffers;
SHOW max_connections;
SHOW wal_level;

-- Ver parâmetros de um contexto
SELECT name, setting, unit, context, boot_val, reset_val
FROM pg_settings
WHERE name LIKE '%wal%'
ORDER BY name;

-- Ver localização dos arquivos de configuração
SELECT name, setting FROM pg_settings
WHERE name IN ('config_file', 'hba_file', 'ident_file', 'data_directory');

4. Criando um Banco de Dados

O PostgreSQL permite criar bancos de dados de várias formas: via SQL, via linha de comando (createdb) ou via interface gráfica (pgAdmin). Cada banco de dados é um container isolado com seu próprio catálogo, schemas e configurações.

Via SQL (CREATE DATABASE)


-- Criação básica
CREATE DATABASE meubanco;

-- Criação com owner, encoding, locale e outras opções
CREATE DATABASE meubanco
    OWNER joao
    ENCODING 'UTF8'
    LC_COLLATE 'pt_BR.UTF-8'
    LC_CTYPE 'pt_BR.UTF-8'
    TEMPLATE template0
    CONNECTION LIMIT 50;

-- Criação com tablespace customizado
CREATE DATABASE meubanco
    OWNER maria
    TABLESPACE fast_disk
    ENCODING 'UTF8';

-- Criação permitindo conexões apenas do usuário dono
CREATE DATABASE backup_db
    OWNER postgres
    ALLOW_CONNECTIONS false;

Via Linha de Comando (createdb)


# Criação básica (conecta como usuário atual do SO)
createdb meubanco

# Especificando owner
createdb -O joao meubanco

# Com encoding e locale
createdb -E UTF8 -l pt_BR.UTF-8 -O joao meubanco

# Com descrição (comment)
createdb meubanco -O joao --comment="Banco de dados de produção"

# Com connection limit
createdb meubanco -O joao -N 50

Template Databases

O PostgreSQL cria bancos de dados sempre copiando um banco template. Existem dois templates padrão:

  • template0: Banco "limpo" — contém apenas os objetos padrão do sistema, sem dados de usuário. Ideal para criar bancos com encoding ou locale diferentes do template1.
  • template1: Banco "customizável" — você pode adicionar objetos (tabelas, funções, extensões) que serão automaticamente incluídos em todos os bancos criados posteriormente a partir dele. É o template padrão se nenhum for especificado. 

-- Criar banco a partir do template0 (com encoding específico)
CREATE DATABASE db_latin1
    TEMPLATE template0
    ENCODING 'LATIN1'
    LC_COLLATE 'pt_BR.ISO-8859-1';

-- Adicionar extensão ao template1 (todos os novos bancos a herdarão)
\c template1
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- Novo banco herda automaticamente as extensões do template1
CREATE DATABASE novo_banco;
-- Extensões uuid-ossp e pgcrypto já estarão disponíveis

Gerenciando Schemas

Dentro de cada banco de dados, os objetos são organizados em schemas (namespaces lógicos): 


-- Criar schema
CREATE SCHEMA financeiro;
CREATE SCHEMA rh AUTHORIZATION joao;

-- Criar tabela em schema específico
CREATE TABLE financeiro.contas (
    id SERIAL PRIMARY KEY,
    descricao VARCHAR(200),
    valor NUMERIC(12,2)
);

-- Definir schema padrão para busca
SET search_path TO financeiro, public;

-- Ver schemas existentes
SELECT schema_name FROM information_schema.schemata;

-- Ver search_path atual
SHOW search_path;

Listando e Gerenciando Bancos


-- Listar bancos de dados
SELECT datname, datowner, encoding, datcollate, datctype, datconnlimit
FROM pg_database
WHERE datistemplate = false
ORDER BY datname;

-- Via terminal psql
\l

-- Renomear banco
ALTER DATABASE meubanco RENAME TO novo_nome;

-- Alterar owner
ALTER DATABASE meubanco OWNER TO maria;

-- Alterar connection limit
ALTER DATABASE meubanco CONNECTION LIMIT 200;

-- Excluir banco
DROP DATABASE meubanco;
-- Com IF EXISTS (evita erro se não existir)
DROP DATABASE IF EXISTS meubanco;

-- Excluir banco mesmo com conexões ativas
DROP DATABASE meubanco WITH (FORCE);

5. Dialeto SQL PostgreSQL (PL/pgSQL)

O PostgreSQL implementa uma das variantes mais completas do SQL padrão. Embora compartilhe fundamentos com Oracle, MySQL e SQL Server, existem diferenças importantes no dialeto. Abaixo, exploramos o DML essencial com foco nas particularidades do PostgreSQL.

SELECT — Consultando Dados

O PostgreSQL estende o SELECT com várias funcionalidades avançadas: 


-- SELECT básico com filtro e ordenação
SELECT nome, email FROM clientes
WHERE ativo = true
ORDER BY nome;

-- SELECT com LIMIT e OFFSET (paginação)
SELECT * FROM produtos
ORDER BY preco DESC
LIMIT 10 OFFSET 20;

-- SELECT com DISTINCT ON (específico do PostgreSQL)
SELECT DISTINCT ON (categoria) categoria, nome, preco
FROM produtos
ORDER BY categoria, preco DESC;

-- SELECT com CASE
SELECT
    nome,
    preco,
    CASE
        WHEN preco < 100 THEN 'Barato'
        WHEN preco < 500 THEN 'Médio'
        WHEN preco < 1000 THEN 'Caro'
        ELSE 'Premium'
    END AS faixa_preco
FROM produtos;

-- SELECT com funções de string
SELECT
    UPPER(nome) AS nome_maiusculo,
    LOWER(email) AS email_minusculo,
    LENGTH(nome) AS tamanho_nome,
    SUBSTRING(nome FROM 1 FOR 10) AS nome_curto,
    CONCAT(nome, ' <', email, '>') AS contato
FROM clientes;

-- SELECT com funções de data/hora
SELECT
    CURRENT_DATE AS hoje,
    CURRENT_TIME AS agora_hora,
    CURRENT_TIMESTAMP AS agora_completo,
    NOW() AS timestamp_atual,
    NOW() - INTERVAL '30 days' AS mes_passado,
    DATE_TRUNC('month', NOW()) AS inicio_mes,
    EXTRACT(YEAR FROM data_cadastro) AS ano_cadastro
FROM clientes;

-- SELECT com subqueries (subselects)
SELECT nome, (
    SELECT COUNT(*) FROM pedidos WHERE cliente_id = clientes.id
) AS total_pedidos
FROM clientes;

-- SELECT com WITH (CTE - Common Table Expression)
WITH vendas_mensais AS (
    SELECT
        DATE_TRUNC('month', data_pedido) AS mes,
        SUM(valor_total) AS total_vendas
    FROM pedidos
    GROUP BY DATE_TRUNC('month', data_pedido)
)
SELECT mes, total_vendas,
    LAG(total_vendas) OVER (ORDER BY mes) AS vendas_mes_anterior,
    total_vendas - LAG(total_vendas) OVER (ORDER BY mes) AS diferenca
FROM vendas_mensais
ORDER BY mes DESC;

INSERT — Inserindo Dados

O PostgreSQL oferece várias formas de inserir dados, incluindo recursos avançados: 


-- INSERT básico com valores
INSERT INTO produtos (nome, preco, estoque, categoria)
VALUES ('Notebook Gamer', 3500.00, 10, 'Eletrônicos');

-- INSERT múltiplo (várias linhas de uma vez)
INSERT INTO produtos (nome, preco, estoque, categoria)
VALUES
    ('Mouse Wireless', 150.00, 50, 'Periféricos'),
    ('Teclado Mecânico', 400.00, 30, 'Periféricos'),
    ('Monitor 27"', 1800.00, 15, 'Eletrônicos'),
    ('Webcam HD', 250.00, 40, 'Periféricos');

-- INSERT com SELECT (copiar dados entre tabelas)
INSERT INTO produtos_arquivados (id, nome, preco, data_arquivamento)
SELECT id, nome, preco, CURRENT_DATE
FROM produtos
WHERE estoque = 0;

-- INSERT com DEFAULT
INSERT INTO clientes (nome, email, data_cadastro, ativo)
VALUES ('João Silva', 'joao@email.com', DEFAULT, DEFAULT);

-- INSERT com RETURNING (retorna valores gerados)
INSERT INTO clientes (nome, email)
VALUES ('Maria Souza', 'maria@email.com')
RETURNING id, nome, data_cadastro;

-- INSERT com RETURNING + INTO (armazena resultado em variável PL/pgSQL)
DO $$
DECLARE
    novo_id INTEGER;
BEGIN
    INSERT INTO clientes (nome, email)
    VALUES ('Pedro Alves', 'pedro@email.com')
    RETURNING id INTO novo_id;

    RAISE NOTICE 'Novo cliente cadastrado com ID: %', novo_id;
END;
$$;

UPSERT (INSERT ... ON CONFLICT)

O UPSERT (uma combinação de UPDATE + INSERT) é uma das funcionalidades mais úteis do PostgreSQL. Ele insere uma linha e, se houver conflito com uma constraint (como PRIMARY KEY ou UNIQUE), executa uma ação alternativa (UPDATE ou DO NOTHING): 


-- UPSERT com ON CONFLICT DO UPDATE
-- Se já existir produto com id=1, atualiza o preço
INSERT INTO produtos (id, nome, preco, estoque)
VALUES (1, 'Mouse', 50.00, 100)
ON CONFLICT (id)
DO UPDATE SET
    preco = EXCLUDED.preco,
    estoque = EXCLUDED.estoque;

-- ON CONFLICT DO NOTHING (ignora se já existir)
INSERT INTO produtos (id, nome, preco)
VALUES (1, 'Mouse', 50.00)
ON CONFLICT (id) DO NOTHING;

-- ON CONFLICT com constraint específica
INSERT INTO clientes (email, nome)
VALUES ('joao@email.com', 'João Atualizado')
ON CONFLICT (email)
DO UPDATE SET nome = EXCLUDED.nome;

-- ON CONFLICT com condição WHERE
INSERT INTO estoque (produto_id, quantidade)
VALUES (10, 100)
ON CONFLICT (produto_id)
DO UPDATE SET quantidade = estoque.quantidade + EXCLUDED.quantidade
WHERE estoque.produto_id = 10;

-- UPSERT retornando o registro final (inserido ou atualizado)
INSERT INTO produtos (id, nome, preco)
VALUES (1, 'Mouse Pro', 89.90)
ON CONFLICT (id)
DO UPDATE SET preco = EXCLUDED.preco, nome = EXCLUDED.nome
RETURNING id, nome, preco,
    CASE WHEN xmax = 0 THEN 'Inserido' ELSE 'Atualizado' END AS acao;

UPDATE — Atualizando Dados


-- UPDATE básico com condição
UPDATE produtos
SET preco = preco * 1.1
WHERE categoria = 'Eletrônicos';

-- UPDATE múltiplas colunas
UPDATE clientes
SET
    email = 'joao.novo@email.com',
    telefone = '(11) 99999-0000',
    atualizado_em = NOW()
WHERE id = 1;

-- UPDATE com subquery
UPDATE produtos
SET preco = (
    SELECT AVG(preco) FROM produtos WHERE categoria = 'Eletrônicos'
)
WHERE estoque = 0;

-- UPDATE com FROM (atualizar usando JOIN)
UPDATE produtos p
SET preco = p.preco * 0.9
FROM categorias c
WHERE p.categoria_id = c.id
AND c.nome = 'Eletrônicos'
AND p.estoque > 50;

-- UPDATE com RETURNING
UPDATE produtos
SET estoque = estoque - 5
WHERE id = 10 AND estoque >= 5
RETURNING id, nome, estoque;

DELETE — Removendo Dados


-- DELETE básico com condição
DELETE FROM log_acesso
WHERE data < CURRENT_DATE - INTERVAL '90 days';

-- DELETE com subquery (remove clientes sem pedidos)
DELETE FROM clientes
WHERE id NOT IN (SELECT DISTINCT cliente_id FROM pedidos);

-- DELETE com USING (delete baseado em JOIN)
DELETE FROM itens_pedido i
USING pedidos p
WHERE i.pedido_id = p.id
AND p.status = 'cancelado'
AND p.data_pedido < CURRENT_DATE - INTERVAL '1 year';

-- DELETE com RETURNING (útil para arquivar antes de deletar)
DELETE FROM clientes
WHERE id = 5
RETURNING id, nome, email;

TRUNCATE — Esvaziando Tabelas Rapidamente


-- TRUNCATE básico (mais rápido que DELETE sem WHERE)
TRUNCATE TABLE log_acesso;

-- TRUNCATE múltiplas tabelas
TRUNCATE TABLE log_acesso, log_erros, cache_sessao;

-- TRUNCATE com RESTART IDENTITY (reseta sequências)
TRUNCATE TABLE produtos RESTART IDENTITY;

-- TRUNCATE com CASCADE (trunca também tabelas dependentes)
TRUNCATE TABLE clientes CASCADE;

-- TRUNCATE dentro de transação (pode ser desfeito com ROLLBACK!)
BEGIN;
TRUNCATE TABLE log_acesso;
-- Se algo der errado...
ROLLBACK;
-- Diferente de outros SGBDs, TRUNCATE é transacional no PostgreSQL

Transações


-- Transação explícita
BEGIN;
UPDATE contas SET saldo = saldo - 1000 WHERE id = 1;
UPDATE contas SET saldo = saldo + 1000 WHERE id = 2;
-- Se tudo OK:
COMMIT;
-- Se algo deu errado:
-- ROLLBACK;

-- Transação com SAVEPOINT
BEGIN;
INSERT INTO pedidos (cliente_id, valor_total) VALUES (1, 500);
SAVEPOINT sp_pedido_criado;

INSERT INTO itens_pedido (pedido_id, produto_id, quantidade, preco_unit)
VALUES (currval('pedidos_id_seq'), 10, 2, 250);

-- Se houver erro no item, volta ao savepoint sem perder o pedido
ROLLBACK TO sp_pedido_criado;

-- Tenta novamente com outro produto
INSERT INTO itens_pedido (pedido_id, produto_id, quantidade, preco_unit)
VALUES (currval('pedidos_id_seq'), 15, 1, 500);

COMMIT;

6. JOINs no PostgreSQL

JOINs são a operação fundamental para combinar dados de múltiplas tabelas em uma única consulta. O PostgreSQL suporta todos os tipos de JOIN do SQL padrão, com desempenho otimizado por seu sofisticado planejador de consultas.

Estrutura Base das Tabelas de Exemplo

Para todos os exemplos a seguir, considere este esquema:


-- Tabela de clientes
CREATE TABLE clientes (
    id          SERIAL PRIMARY KEY,
    nome        VARCHAR(100) NOT NULL,
    email       VARCHAR(150) UNIQUE,
    cidade      VARCHAR(100),
    data_cad    DATE DEFAULT CURRENT_DATE
);

-- Tabela de pedidos
CREATE TABLE pedidos (
    id          SERIAL PRIMARY KEY,
    cliente_id  INTEGER REFERENCES clientes(id),
    data_pedido DATE DEFAULT CURRENT_DATE,
    valor_total NUMERIC(10,2) NOT NULL,
    status      VARCHAR(20) DEFAULT 'pendente'
);

-- Tabela de itens do pedido
CREATE TABLE itens_pedido (
    id          SERIAL PRIMARY KEY,
    pedido_id   INTEGER REFERENCES pedidos(id),
    produto_id  INTEGER REFERENCES produtos(id),
    quantidade  INTEGER NOT NULL CHECK (quantidade > 0),
    preco_unit  NUMERIC(10,2) NOT NULL
);

-- Tabela de produtos
CREATE TABLE produtos (
    id          SERIAL PRIMARY KEY,
    nome        VARCHAR(100) NOT NULL,
    preco       NUMERIC(10,2) NOT NULL CHECK (preco > 0),
    categoria   VARCHAR(50),
    estoque     INTEGER DEFAULT 0
);

INNER JOIN

O INNER JOIN retorna apenas as linhas que possuem correspondência em ambas as tabelas. É o tipo de JOIN mais comum e o padrão quando se usa apenas JOIN. 


-- INNER JOIN: clientes que possuem pedidos
SELECT c.id, c.nome, p.id AS pedido_id, p.valor_total, p.status
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
ORDER BY c.nome, p.data_pedido;

-- INNER JOIN com múltiplas tabelas (clientes -> pedidos -> itens -> produtos)
SELECT
    c.nome AS cliente,
    p.id AS pedido,
    prod.nome AS produto,
    ip.quantidade,
    ip.preco_unit,
    (ip.quantidade * ip.preco_unit) AS subtotal
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN itens_pedido ip ON p.id = ip.pedido_id
INNER JOIN produtos prod ON ip.produto_id = prod.id
ORDER BY c.nome, p.id;

-- INNER JOIN com filtro adicional
SELECT c.nome, COUNT(p.id) AS total_pedidos, SUM(p.valor_total) AS total_gasto
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
WHERE p.status = 'entregue'
GROUP BY c.id, c.nome
HAVING SUM(p.valor_total) > 500
ORDER BY total_gasto DESC;

-- INNER JOIN usando USING (quando a coluna de junção tem o mesmo nome)
SELECT c.nome, p.valor_total
FROM clientes c
INNER JOIN pedidos p USING (cliente_id);

Resultado esperado: Ana (2 pedidos), Bruno (1 pedido), Carla (1 pedido), Daniel (1 pedido). Eva não aparece porque não possui pedidos.

LEFT JOIN (LEFT OUTER JOIN)

O LEFT JOIN retorna todas as linhas da tabela à esquerda (cláusula FROM), mesmo que não haja correspondência na tabela à direita. As colunas da tabela direita sem correspondência recebem NULL. 


-- LEFT JOIN: todos os clientes, com ou sem pedidos
SELECT c.id, c.nome, p.id AS pedido_id, p.valor_total, p.status
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
ORDER BY c.nome, p.data_pedido;

-- LEFT JOIN para encontrar clientes SEM pedidos (anti-join)
SELECT c.id, c.nome, c.email
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
WHERE p.id IS NULL
ORDER BY c.nome;

-- LEFT JOIN com agregação (total gasto, incluindo quem nunca comprou)
SELECT
    c.nome,
    c.cidade,
    COUNT(p.id) AS total_pedidos,
    COALESCE(SUM(p.valor_total), 0) AS total_gasto
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
GROUP BY c.id, c.nome, c.cidade
ORDER BY total_gasto DESC;

-- LEFT JOIN em cadeia (clientes -> pedidos -> itens)
SELECT
    c.nome,
    COALESCE(SUM(ip.quantidade * ip.preco_unit), 0) AS total_consumido
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
LEFT JOIN itens_pedido ip ON p.id = ip.pedido_id
GROUP BY c.id, c.nome
ORDER BY total_consumido DESC;

O COALESCE é essencial em LEFT JOINs com agregações: ele converte NULL (cliente sem pedidos) em 0, evitando resultados inesperados.

RIGHT JOIN (RIGHT OUTER JOIN)

O RIGHT JOIN é o oposto do LEFT JOIN: retorna todas as linhas da tabela à direita, com NULL nas colunas da tabela esquerda quando não há correspondência. Na prática, muitos desenvolvedores preferem reorganizar a consulta como LEFT JOIN para maior legibilidade — mas é importante entender RIGHT JOIN. 


-- RIGHT JOIN: todos os pedidos, mesmo sem cliente (raro, mas possível)
SELECT c.nome, p.id AS pedido_id, p.valor_total
FROM clientes c
RIGHT JOIN pedidos p ON c.id = p.cliente_id;

-- RIGHT JOIN equivalente ao LEFT JOIN (trocando a ordem das tabelas)
-- Estas duas consultas produzem o mesmo resultado:

-- RIGHT JOIN
SELECT c.nome, p.valor_total
FROM pedidos p
RIGHT JOIN clientes c ON c.id = p.cliente_id;

-- LEFT JOIN equivalente (mais comum)
SELECT c.nome, p.valor_total
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id;

-- RIGHT JOIN para auditoria: pedidos órfãos (sem cliente)
SELECT p.id, p.valor_total, p.data_pedido, c.nome
FROM clientes c
RIGHT JOIN pedidos p ON c.id = p.cliente_id
WHERE c.id IS NULL;

FULL OUTER JOIN

O FULL OUTER JOIN retorna todas as linhas de ambas as tabelas. Quando há correspondência, as colunas são combinadas. Quando não há, as colunas da tabela sem match recebem NULL. É a união do LEFT JOIN com o RIGHT JOIN. 


-- FULL OUTER JOIN: todos os clientes e todos os pedidos
SELECT c.nome AS cliente, p.id AS pedido_id, p.valor_total
FROM clientes c
FULL OUTER JOIN pedidos p ON c.id = p.cliente_id
ORDER BY c.nome NULLS LAST, p.id;

-- FULL OUTER JOIN para auditoria de integridade referencial
-- (encontra clientes sem pedidos E pedidos sem cliente)
SELECT
    CASE WHEN c.id IS NULL THEN 'PEDIDO ÓRFÃO'
         WHEN p.id IS NULL THEN 'CLIENTE SEM PEDIDO'
         ELSE 'OK'
    END AS status_ref,
    COALESCE(c.nome, 'N/D') AS cliente,
    COALESCE(p.id::TEXT, 'N/D') AS pedido,
    COALESCE(p.valor_total::TEXT, 'N/D') AS valor
FROM clientes c
FULL OUTER JOIN pedidos p ON c.id = p.cliente_id
WHERE c.id IS NULL OR p.id IS NULL
ORDER BY status_ref;

-- FULL OUTER JOIN com agregação (visão completa)
SELECT
    COALESCE(c.cidade, 'Sem Cidade') AS cidade,
    COUNT(DISTINCT c.id) AS total_clientes,
    COUNT(DISTINCT p.id) AS total_pedidos,
    COALESCE(SUM(p.valor_total), 0) AS receita_total
FROM clientes c
FULL OUTER JOIN pedidos p ON c.id = p.cliente_id AND c.cidade IS NOT NULL
GROUP BY c.cidade
ORDER BY receita_total DESC;

O FULL OUTER JOIN é menos usado no dia a dia, mas é extremamente útil em relatórios de conciliação e auditoria de dados.

CROSS JOIN

O CROSS JOIN produz o produto cartesiano de duas tabelas: cada linha da primeira tabela é combinada com cada linha da segunda. Para tabelas de tamanho N e M, o resultado tem N × M linhas. Use com cautela — o volume de dados cresce exponencialmente! 


-- CROSS JOIN: todas as combinações de cliente e produto
SELECT c.nome, p.nome AS produto
FROM clientes c
CROSS JOIN produtos p
ORDER BY c.nome, p.nome;

-- Geração de séries: combinar datas com categorias
SELECT d.dia, c.nome AS categoria
FROM generate_series('2025-06-01'::DATE, '2025-06-07'::DATE, '1 day') AS d(dia)
CROSS JOIN (SELECT DISTINCT categoria FROM produtos) AS c(categoria)
ORDER BY d.dia, c.nome;

-- CROSS JOIN para criar matriz de comissões
SELECT v.nome AS vendedor, f.mes, f.fator
FROM (VALUES ('Ana'), ('Bruno'), ('Carla')) AS v(nome)
CROSS JOIN (
    VALUES
        ('Jan', 1.0), ('Fev', 1.05), ('Mar', 1.1),
        ('Abr', 1.15), ('Mai', 1.2), ('Jun', 1.25)
) AS f(mes, fator)
ORDER BY v.nome, f.mes;

-- CROSS JOIN com filtro (se comporta como INNER JOIN, mas menos eficiente)
SELECT c.nome, p.nome AS produto
FROM clientes c
CROSS JOIN produtos p
WHERE c.cidade = 'São Paulo' AND p.categoria = 'Eletrônicos';

Cuidado: Um CROSS JOIN entre duas tabelas de 10.000 linhas cada gera 100.000.000 de resultados. Use apenas quando souber exatamente o que está fazendo ou em tabelas muito pequenas (dimensões, parâmetros).

NATURAL JOIN

O NATURAL JOIN automaticamente faz a junção usando colunas com o mesmo nome em ambas as tabelas. É conveniente, mas pode ser perigoso se as tabelas tiverem colunas com o mesmo nome que não deveriam ser usadas na junção. 


-- NATURAL JOIN (equivale a INNER JOIN ... USING(colunas_comuns))
SELECT * FROM pedidos
NATURAL JOIN clientes;

-- Natural LEFT JOIN
SELECT nome, valor_total FROM clientes
NATURAL LEFT JOIN pedidos;

-- ATENÇÃO: se ambas as tabelas tiverem colunas 'id', 'nome', 'data_cad',
-- o NATURAL JOIN usará TODAS elas — o que provavelmente não é o desejado.
-- Por segurança, muitos DBAs desencorajam o uso de NATURAL JOIN.

SELF JOIN

Um SELF JOIN ocorre quando uma tabela é unida a ela mesma. É útil para comparar linhas dentro da mesma tabela ou representar hierarquias (como funcionário → gerente). 


-- Tabela de funcionários com hierarquia (gerente_id referencia funcionarios.id)
CREATE TABLE funcionarios (
    id          SERIAL PRIMARY KEY,
    nome        VARCHAR(100) NOT NULL,
    cargo       VARCHAR(50),
    gerente_id  INTEGER REFERENCES funcionarios(id)
);

INSERT INTO funcionarios (nome, cargo, gerente_id) VALUES
    ('Carlos', 'CEO', NULL),
    ('Ana', 'Diretora', 1),
    ('Bruno', 'Gerente TI', 2),
    ('Diana', 'Gerente RH', 2),
    ('Eduardo', 'Analista', 3),
    ('Fernanda', 'Analista', 3);

-- SELF JOIN: listar cada funcionário com seu gerente
SELECT
    f.nome AS funcionario,
    f.cargo AS cargo_funcionario,
    g.nome AS gerente,
    g.cargo AS cargo_gerente
FROM funcionarios f
LEFT JOIN funcionarios g ON f.gerente_id = g.id
ORDER BY f.nome;

-- SELF JOIN com recursão: árvore hierárquica completa (WITH RECURSIVE)
WITH RECURSIVE hierarquia AS (
    -- Caso base: CEO (sem gerente)
    SELECT id, nome, cargo, gerente_id, 0 AS nivel,
           nome::TEXT AS caminho
    FROM funcionarios
    WHERE gerente_id IS NULL

    UNION ALL

    -- Caso recursivo: subordinados
    SELECT f.id, f.nome, f.cargo, f.gerente_id, h.nivel + 1,
           h.caminho || ' -> ' || f.nome
    FROM funcionarios f
    INNER JOIN hierarquia h ON f.gerente_id = h.id
)
SELECT nivel, REPEAT('    ', nivel) || nome AS organograma, cargo, caminho
FROM hierarquia
ORDER BY caminho;

7. Comparação PostgreSQL vs Oracle

Embora ambos sejam SGBDs relacionais robustos de nível empresarial, PostgreSQL e Oracle diferem em aspectos importantes. Compreender essas diferenças é essencial para quem transita entre as duas plataformas.

Aspecto PostgreSQL Oracle

Custo Gratuito (sem custos de licenciamento) Alto custo de licenciamento por core/usuário
Arquitetura Processos (um processo por conexão) Multithreaded (threads dentro de um processo)

Linguagem Procedural PL/pgSQL (similar a PL/SQL). Também suporta Python, Perl, Tcl, JavaScript, R, Java, Lua PL/SQL (integrada ao kernel). Suporte limitado a outras linguagens (Java via JVM integrada)
Sequências/Auto-incremento SERIAL, BIGSERIAL, GENERATED AS IDENTITY (padrão SQL) SEQUENCE + TRIGGER ou GENERATED AS IDENTITY (a partir do 12c)
Tipos de Dados Extensível. Tipos nativos: JSONB, JSON, XML, Arrays, hstore, Range, Geometric, Network, UUID, JSONPath Tipos built-in menos extensíveis. XMLType, JSON (a partir do 12c), Spatial, Text. Arrays não são nativos

    quantidade  INTEGER NOT NULL CHECK (quantidade > 0),
    preco_unit  NUMERIC(10,2) NOT NULL CHECK (preco_unit > 0)
);

Passo 4 — Executar consultas JOIN de análise


-- Consulta 1: Clientes e seus pedidos (INNER JOIN)
SELECT
    c.nome AS cliente,
    c.cidade,
    p.id AS pedido,
    p.data_pedido,
    p.valor_total,
    p.status
FROM vendas.clientes c
INNER JOIN vendas.pedidos p ON c.id = p.cliente_id
ORDER BY c.nome, p.data_pedido;

-- Consulta 2: Ticket médio por cliente (incluindo quem não comprou)
SELECT
    c.nome,
    c.cidade,
    COUNT(p.id) AS total_pedidos,
    COALESCE(SUM(p.valor_total), 0) AS total_gasto,
    COALESCE(AVG(p.valor_total), 0) AS ticket_medio
FROM vendas.clientes c
LEFT JOIN vendas.pedidos p ON c.id = p.cliente_id
GROUP BY c.id, c.nome, c.cidade
ORDER BY total_gasto DESC;

-- Consulta 3: Produtos mais vendidos (INNER JOIN de 3 tabelas)
SELECT
    pr.nome AS produto,
    cat.nome AS categoria,
    SUM(ip.quantidade) AS total_vendido,
    SUM(ip.quantidade * ip.preco_unit) AS receita_gerada
FROM vendas.itens_pedido ip
INNER JOIN vendas.produtos pr ON ip.produto_id = pr.id
INNER JOIN vendas.categorias cat ON pr.categoria_id = cat.id
GROUP BY pr.id, pr.nome, cat.nome
ORDER BY receita_gerada DESC;

-- Consulta 4: Clientes que nunca compraram (LEFT JOIN + IS NULL)
SELECT
    c.nome,
    c.email,
    c.cidade,
    c.data_cadastro
FROM vendas.clientes c
LEFT JOIN vendas.pedidos p ON c.id = p.cliente_id
WHERE p.id IS NULL
ORDER BY c.data_cadastro;

-- Consulta 5: Vendas por categoria (CROSS JOIN de categorias com meses)
WITH meses AS (
    SELECT generate_series(
        '2025-05-01'::DATE,
        '2025-06-01'::DATE,
        '1 month'::INTERVAL
    ) AS mes
)
SELECT
    TO_CHAR(m.mes, 'YYYY-MM') AS mes,
    cat.nome AS categoria,
    COALESCE(SUM(ip.quantidade * ip.preco_unit), 0) AS vendas
FROM meses m
CROSS JOIN vendas.categorias cat
LEFT JOIN vendas.produtos pr ON pr.categoria_id = cat.id
LEFT JOIN vendas.itens_pedido ip ON ip.produto_id = pr.id
LEFT JOIN vendas.pedidos p ON ip.pedido_id = p.id
    AND DATE_TRUNC('month', p.data_pedido) = m.mes
GROUP BY m.mes, cat.id, cat.nome
ORDER BY m.mes, vendas DESC;

-- Consulta 6: Status dos pedidos com itens detalhados (FULL OUTER JOIN simulado)
SELECT
    p.id AS pedido,
    p.status,
    c.nome AS cliente,
    pr.nome AS produto,
    ip.quantidade,
    ip.preco_unit
FROM vendas.pedidos p
LEFT JOIN vendas.clientes c ON p.cliente_id = c.id
LEFT JOIN vendas.itens_pedido ip ON p.id = ip.pedido_id
LEFT JOIN vendas.produtos pr ON ip.produto_id = pr.id
ORDER BY p.id, pr.nome;

2. Crie 3 tabelas relacionadas (clientes, pedidos, itens) com tipos apropriados

No banco loja, crie as tabelas clientes, pedidos e itens_pedido com constraints adequadas: PRIMARY KEY (SERIAL), FOREIGN KEY, CHECK, NOT NULL, UNIQUE e valores DEFAULT onde fizer sentido. Use tipos de dados apropriados para cada coluna.

Gabarito: 

-- Tabela clientes
CREATE TABLE clientes (
    id              SERIAL PRIMARY KEY,
    nome            VARCHAR(100) NOT NULL,
    email           VARCHAR(150) UNIQUE NOT NULL,
    telefone        VARCHAR(20),
    cpf             VARCHAR(14) UNIQUE,
    data_nascimento DATE,
    cidade          VARCHAR(100),
    estado          CHAR(2),
    data_cadastro   DATE DEFAULT CURRENT_DATE,
    ativo           BOOLEAN DEFAULT true
);

-- Tabela pedidos
CREATE TABLE pedidos (
    id              SERIAL PRIMARY KEY,
    cliente_id      INTEGER NOT NULL REFERENCES clientes(id),
    data_pedido     TIMESTAMP DEFAULT NOW(),
    valor_total     NUMERIC(10,2) NOT NULL DEFAULT 0 CHECK (valor_total >= 0),
    status          VARCHAR(20) DEFAULT 'pendente'
        CHECK (status IN ('pendente', 'confirmado', 'enviado', 'entregue', 'cancelado'))
);

-- Tabela itens_pedido
CREATE TABLE itens_pedido (
    id              SERIAL PRIMARY KEY,
    pedido_id       INTEGER NOT NULL REFERENCES pedidos(id) ON DELETE CASCADE,
    produto_nome    VARCHAR(100) NOT NULL,
    quantidade      INTEGER NOT NULL CHECK (quantidade > 0),
    preco_unitario  NUMERIC(10,2) NOT NULL CHECK (preco_unitario > 0),
    CONSTRAINT uq_item_pedido UNIQUE (pedido_id, produto_nome)
);

-- Criar índices para melhor performance
CREATE INDEX idx_pedidos_cliente ON pedidos(cliente_id);
CREATE INDEX idx_pedidos_status ON pedidos(status);
CREATE INDEX idx_pedidos_data ON pedidos(data_pedido);
CREATE INDEX idx_itens_pedido ON itens_pedido(pedido_id);

A chave estrangeira ON DELETE CASCADE em itens_pedido garante que deletar um pedido automaticamente remove seus itens. Os índices melhoram a performance das consultas por cliente e por status.

3. Faça uma consulta com LEFT JOIN listando clientes que nunca compraram

Usando as tabelas criadas no exercício anterior (populadas com dados de exemplo), escreva uma consulta que liste nome, email e data de cadastro de todos os clientes que nunca realizaram um pedido. Ordene por data de cadastro (mais antigos primeiro).

Gabarito: 

SELECT
    c.nome,
    c.email,
    c.cidade,
    c.data_cadastro
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
WHERE p.id IS NULL
ORDER BY c.data_cadastro ASC;

Explicação: O LEFT JOIN garante que todos os clientes apareçam. O WHERE p.id IS NULL filtra apenas aqueles que não têm correspondência na tabela pedidos — ou seja, nunca compraram. Se o LEFT JOIN fosse substituído por INNER JOIN, esses clientes simplesmente não apareceriam no resultado.

4. Implemente um UPSERT para cadastro de produtos com ON CONFLICT

Crie a tabela produtos com chave primária em id e constraint UNIQUE em codigo_barras. Em seguida, escreva uma instrução UPSERT que: se o codigo_barras já existir, atualize o preço e o estoque; se não existir, insira o novo produto. Retorne o id, nome, preço e se foi inserido ou atualizado.

Gabarito: 

-- Passo 1: Criar a tabela produtos
CREATE TABLE produtos (
    id              SERIAL PRIMARY KEY,
    codigo_barras   VARCHAR(13) UNIQUE NOT NULL,
    nome            VARCHAR(100) NOT NULL,
    descricao       TEXT,
    preco           NUMERIC(10,2) NOT NULL CHECK (preco > 0),
    estoque         INTEGER NOT NULL DEFAULT 0 CHECK (estoque >= 0),
    categoria       VARCHAR(50),
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_produtos_codigo_barras ON produtos(codigo_barras);
CREATE INDEX idx_produtos_categoria ON produtos(categoria);

-- Passo 2: UPSERT — Inserir ou atualizar por código de barras
INSERT INTO produtos (codigo_barras, nome, preco, estoque, categoria)
VALUES ('7891234567890', 'Notebook Dell Inspiron', 4200.00, 15, 'Informática')
ON CONFLICT (codigo_barras)
DO UPDATE SET
    nome = EXCLUDED.nome,
    preco = EXCLUDED.preco,
    estoque = EXCLUDED.estoque,
    categoria = EXCLUDED.categoria,
    updated_at = NOW()
RETURNING
    id,
    nome,
    preco,
    estoque,
    CASE
        WHEN xmax = 0 THEN 'Inserido'
        ELSE 'Atualizado'
    END AS acao_realizada;

-- Passo 3: Testar inserindo um novo produto
INSERT INTO produtos (codigo_barras, nome, preco, estoque, categoria)
VALUES ('7899876543210', 'Mouse Logitech MX', 350.00, 100, 'Informática')
ON CONFLICT (codigo_barras)
DO UPDATE SET
    preco = EXCLUDED.preco,
    estoque = EXCLUDED.estoque,
    updated_at = NOW()
RETURNING id, nome, preco,
    CASE WHEN xmax = 0 THEN 'Inserido' ELSE 'Atualizado' END AS acao_realizada;

-- Passo 4: Testar atualizando um produto existente (mesmo código_barras)
INSERT INTO produtos (codigo_barras, nome, preco, estoque, categoria)
VALUES ('7899876543210', 'Mouse Logitech MX Master 3S', 450.00, 120, 'Informática')
ON CONFLICT (codigo_barras)
DO UPDATE SET
    nome = EXCLUDED.nome,
    preco = EXCLUDED.preco,
    estoque = EXCLUDED.estoque,
    updated_at = NOW()
RETURNING id, nome, preco,
    CASE WHEN xmax = 0 THEN 'Inserido' ELSE 'Atualizado' END AS acao_realizada;

-- Verificar resultado final
SELECT * FROM produtos ORDER BY id;

Observação: O ON CONFLICT (codigo_barras) funciona porque há uma constraint UNIQUE nessa coluna. Se a constraint fosse composta (ex: UNIQUE(codigo_barras, fornecedor_id)), a cláusula seria ON CONFLICT (codigo_barras, fornecedor_id). O EXCLUDED é uma pseudo-tabela que contém os valores que estavam sendo inseridos — essencial para referenciar os novos valores na cláusula UPDATE.
