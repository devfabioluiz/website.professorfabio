Por que integrar Node.js com PostgreSQL?

Até agora, executamos nossas queries diretamente no psql ou no pgAdmin. Mas em uma aplicação web real, quem dispara as queries é o backend — e o Node.js é um dos runtimes mais populares para isso. A integração permite:

  • APIs REST que leem e gravam dados no PostgreSQL
  • Autenticação de usuários com sessões persistidas no banco
  • Dashboards que consomem queries analíticas em tempo real
  • Sistemas de e-commerce, ERPs, CRMs com dados relacionais
  • Microsserviços que usam PostgreSQL como fonte da verdade

Nesta aula, usaremos o pacote pg (node-postgres), o driver mais maduro e popular para conectar Node.js ao PostgreSQL. Ele implementa o protocolo nativo do PostgreSQL sobre TCP, sem dependências externas.

2. Pool de Conexões — Como o PostgreSQL gerencia conexões no Node.js

Toda vez que um cliente se conecta ao PostgreSQL, o servidor cria um processo backend (no Linux, um fork do processo postmaster). Esse processo consome memória (~5-10 MB por conexão) e uma entrada na tabela pg_stat_activity. Em produção, abrir uma conexão nova a cada requisição HTTP é inviável — gasta tempo, CPU e pode esgotar o limite configurado em max_connections.

O Pool resolve isso mantendo um conjunto fixo de conexões abertas e reutilizáveis:

  1. Quando uma query é solicitada, o pool entrega uma conexão já pronta
  2. Após o uso, a conexão volta ao pool (não é fechada)
  3. Se todas as conexões estiverem ocupadas, a query aguarda na fila (até o timeout)
  4. Conexões ociosas por muito tempo são fechadas automaticamente

Configurando o Pool

Crie um arquivo src/database/pg.js: 

const { Pool } = require("pg");

const pool = new Pool({
  host: process.env.PG_HOST || "localhost",
  port: process.env.PG_PORT || 5432,
  user: process.env.PG_USER || "postgres",
  password: process.env.PG_PASSWORD,
  database: process.env.PG_DATABASE || "meubanco",
  max: 20,
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 5000,
});

pool.on("error", (erro) => {
  console.error("Erro inesperado no pool do PostgreSQL:", erro.message);
});

module.exports = pool;

🤓 Explicando o código 📝

🔹 max: 20

Máximo de conexões simultâneas mantidas no pool. Esse valor NUNCA deve ultrapassar o max_connections do PostgreSQL (padrão 100). Uma boa prática é deixar margem para o pgAdmin, psql e outras ferramentas.

🔹 idleTimeoutMillis: 30000

Se uma conexão ficar 30 segundos ociosa, ela é fechada automaticamente. Isso evita acumular conexões de períodos de baixa demanda.

🔹 connectionTimeoutMillis: 5000

Tempo máximo para estabelecer uma nova conexão. Se o PostgreSQL não responder em 5 segundos, um erro é lançado — evitando que a aplicação trave indefinidamente.

🔹 pool.on("error")

Captura erros de conexões ociosas que quebraram (ex: PostgreSQL reiniciou). Sem esse listener, o erro derruba o processo Node.js.

Arquivo .env — Credenciais fora do código

Nunca coloque host, usuário e senha diretamente no arquivo pg.js. Use variáveis de ambiente:

PG_PORT=5432

PG_PASSWORD=sua_senha_segura

3. Executando queries de forma segura — Parameterized Queries

Um dos ataques mais comuns e perigosos contra bancos de dados é o SQL Injection. Ele ocorre quando concatenamos strings diretamente na query:

❌ ERRADO — Vulnerável a SQL Injection

// NUNCA FAÇA ISSO!
const nome = req.body.nome;

// Se o usuário digitar: '; DROP TABLE produtos; --
await pool.query(`SELECT * FROM produtos WHERE nome = '${nome}'`);

✅ CORRETO — Parameterized Query

O pg oferece suporte nativo a parameterized queries usando placeholders $1, $2, $3... O driver envia os valores separadamente do texto SQL, eliminando qualquer possibilidade de injeção: 

// Forma correta e segura
const nome = req.body.nome;

const resultado = await pool.query(
  "SELECT * FROM produtos WHERE nome = $1",
  [nome]
);

// O pg envia:
//   Parse: SELECT * FROM produtos WHERE nome = $1
//   Bind:  $1 -> 'valor seguro'
//   Execute

🤓 Explicando o código 📝

🔹 Como a proteção funciona internamente

O PostgreSQL usa o Extended Query Protocol. Nesse protocolo, a query e os parâmetros trafegam em mensagens separadas:

  1. Parse: o texto SQL é enviado ao PostgreSQL, que monta a árvore sintática (sem os valores ainda)
  2. Bind: os valores dos parâmetros são enviados já com os tipos definidos — o PostgreSQL sabe que $1 é um VARCHAR, $2 é um INTEGER, etc.
  3. Execute: a query é executada com os valores vinculados — o PostgreSQL nunca interpreta os parâmetros como SQL, apenas como dados

Mesmo que o valor de $1 contenha '; DROP TABLE produtos; --, ele será tratado como uma string literal. O PostgreSQL NUNCA tentará executar comandos que venham como parâmetros.

Múltiplos parâmetros

const resultado = await pool.query(
  `INSERT INTO produtos (nome, preco, estoque)
   VALUES ($1, $2, $3)
   RETURNING *`,
  ["Teclado Mecânico", 350.00, 15]
);

A cláusula RETURNING — Exclusiva do PostgreSQL

O PostgreSQL possui a cláusula RETURNING, que devolve as linhas afetadas por INSERT, UPDATE ou DELETE sem precisar de uma query SELECT extra: 

-- Sem RETURNING: precisa de 2 queries
INSERT INTO produtos (nome, preco) VALUES ('Mouse', 80);
SELECT * FROM produtos WHERE id = currval('produtos_id_seq');

-- Com RETURNING: uma única query resolve
INSERT INTO produtos (nome, preco) VALUES ('Mouse', 80)
RETURNING id, nome, preco, criado_em;

No Node.js, o resultado vem direto no objeto retornado pelo pool: 

const resultado = await pool.query(
  `INSERT INTO produtos (nome, preco) VALUES ($1, $2) RETURNING *`,
  ["Mouse", 80]
);

console.log(resultado.rows[0]);
// { id: 1, nome: 'Mouse', preco: 80, criado_em: 2026-06-29T... }

5. Transações no PostgreSQL via Node.js

Uma transação agrupa múltiplas operações em uma unidade atômica: ou todas são confirmadas juntas (COMMIT) ou todas são revertidas (ROLLBACK). O PostgreSQL é totalmente ACID — as propriedades de Atomicidade, Consistência, Isolamento e Durabilidade são garantidas nativamente.

Usando pool.query() — Desconectado (não recomendado para transações)

Cada chamada a pool.query() pode usar uma conexão diferente do pool. Para transações, você precisa que todas as queries rodem na mesma conexão. Use pool.connect() para obter um Client dedicado:

✅ Transação com Client dedicado

const pool = require("../database/pg");

const transferirEstoque = async (origemId, destinoId, quantidade) => {
  const client = await pool.connect();

  try {
    await client.query("BEGIN");

    const origem = await client.query(
      "SELECT estoque FROM produtos WHERE id = $1 FOR UPDATE",
      [origemId]
    );

    if (origem.rows.length === 0) {
      throw new Error("Produto de origem não encontrado");
    }

    if (origem.rows[0].estoque < quantidade) {
      throw new Error("Estoque insuficiente na origem");
    }

    await client.query(
      "UPDATE produtos SET estoque = estoque - $1 WHERE id = $2",
      [quantidade, origemId]
    );

    await client.query(
      "UPDATE produtos SET estoque = estoque + $1 WHERE id = $2",
      [quantidade, destinoId]
    );

    await client.query("COMMIT");

    return { mensagem: "Transferência realizada com sucesso" };
  } catch (erro) {
    await client.query("ROLLBACK");
    throw erro;
  } finally {
    client.release();
  }
};

🤓 Explicando o código 📝

🔹 pool.connect()

Reserva uma conexão exclusiva do pool. Use para transações ou quando precisar executar múltiplas queries na mesma sessão (ex: SET LOCAL, criar tabelas temporárias).

🔹 client.release() no finally

SEMPRE devolva a conexão ao pool, mesmo se ocorrer erro. Colocar release() no finally garante que a conexão será liberada em qualquer cenário. Se você esquecer, o pool ficará sem conexões disponíveis.

🔹 FOR UPDATE

Bloqueia a linha selecionada até o fim da transação. Isso impede que outra transação altere o estoque simultaneamente (controle de concorrência pessimista).

🔹 SAVEPOINT

Para operações mais complexas, você pode criar pontos de salvamento parciais dentro da transação: 

SAVEPOINT antes_da_validacao;

-- Se a validação falhar...
ROLLBACK TO SAVEPOINT antes_da_validacao;

-- Se tudo estiver certo...
RELEASE SAVEPOINT antes_da_validacao;
COMMIT;

6. JOINs executados a partir do Node.js

Uma das maiores vantagens do PostgreSQL é a capacidade de realizar JOINs complexos. No Node.js, você envia a query completa e recebe os dados já relacionados — o banco faz o trabalho pesado.

Exemplo: Produtos com Categoria (LEFT JOIN)

const listarComCategoria = async () => {
  const resultado = await pool.query(`
    SELECT
      p.id,
      p.nome,
      p.preco,
      p.estoque,
      c.nome AS categoria_nome,
      c.id AS categoria_id

    FROM produtos p

    LEFT JOIN categorias c
      ON p.categoria_id = c.id

    ORDER BY c.nome, p.nome
  `);

  return resultado.rows;
};

Exemplo com parâmetros: Buscar produtos por categoria

const buscarPorCategoria = async (categoriaId) => {
  const resultado = await pool.query(
    `SELECT p.*, c.nome AS categoria_nome
     FROM produtos p
     INNER JOIN categorias c ON p.categoria_id = c.id
     WHERE c.id = $1
     ORDER BY p.nome`,
    [categoriaId]
  );

  return resultado.rows;
};

🤓 Explicando o código 📝

🔹 Por que fazer o JOIN no banco, e não no Node.js?

O PostgreSQL é otimizado para JOINs há décadas. Ele usa índices, hash joins, merge joins e nested loops automaticamente. Fazer o JOIN no Node.js (baixar duas tabelas e cruzar com JavaScript) seria:

  • Mais lento (tráfego de rede desnecessário)
  • Mais memória (todos os dados na RAM do Node.js)
  • Mais código (você reescreveria lógica de banco em JS)

Regra de ouro: quem faz JOIN é o banco. O Node.js recebe o resultado pronto.

const pool = require("./pg");

const criarTabelas = async () => {
  try {
    await pool.query(`
      CREATE TABLE IF NOT EXISTS categorias (
        id SERIAL PRIMARY KEY,
        nome VARCHAR(100) NOT NULL UNIQUE,
        descricao TEXT,
        criado_em TIMESTAMP DEFAULT NOW()
      );
    `);

    await pool.query(`
      CREATE TABLE IF NOT EXISTS produtos (
        id SERIAL PRIMARY KEY,
        nome VARCHAR(150) NOT NULL,
        preco DECIMAL(10, 2) NOT NULL CHECK (preco > 0),
        estoque INTEGER NOT NULL DEFAULT 0 CHECK (estoque >= 0),
        categoria_id INTEGER REFERENCES categorias(id) ON DELETE SET NULL,
        criado_em TIMESTAMP DEFAULT NOW(),
        atualizado_em TIMESTAMP DEFAULT NOW()
      );
    `);

    console.log("Tabelas criadas com sucesso!");

    await pool.end();
    process.exit(0);
  } catch (erro) {
    console.error("Erro ao criar tabelas:", erro.message);
    await pool.end();
    process.exit(1);
  }
};

const pool = require("../database/pg");

    `INSERT INTO categorias (nome, descricao)

2. Transferência de Estoque com Transação

Crie uma rota POST /pg/transferencia que:

  • Receba no body: origemId, destinoId, quantidade
  • Use pool.connect() com BEGIN/COMMIT/ROLLBACK
  • Use SELECT ... FOR UPDATE na consulta de origem
  • Valide se o produto de origem tem estoque suficiente
  • Atualize o estoque dos dois produtos
  • Retorne os dois produtos atualizados em caso de sucesso
Gabarito: 
const pool = require("../database/pg");

const transferirEstoque = async (req, res) => {
  const { origemId, destinoId, quantidade } = req.body;

  if (!origemId || !destinoId || !quantidade) {
    return res.status(400).json({
      erro: "origemId, destinoId e quantidade são obrigatórios",
    });
  }

  const client = await pool.connect();

  try {
    await client.query("BEGIN");

    const origem = await client.query(
      "SELECT * FROM produtos WHERE id = $1 FOR UPDATE",
      [origemId]
    );

    if (origem.rows.length === 0) {
      throw new Error("Produto de origem não encontrado");
    }

    if (origem.rows[0].estoque < quantidade) {
      throw new Error(
        `Estoque insuficiente. Disponível: ${origem.rows[0].estoque}`
      );
    }

    const produtoOrigem = await client.query(
      `UPDATE produtos
       SET estoque = estoque - $1, atualizado_em = NOW()
       WHERE id = $2
       RETURNING *`,
      [quantidade, origemId]
    );

    const produtoDestino = await client.query(
      `UPDATE produtos
       SET estoque = estoque + $1, atualizado_em = NOW()
       WHERE id = $2
       RETURNING *`,
      [quantidade, destinoId]
    );

    await client.query("COMMIT");

    res.json({
      mensagem: "Transferência realizada com sucesso",
      origem: produtoOrigem.rows[0],
      destino: produtoDestino.rows[0],
    });
  } catch (erro) {
    await client.query("ROLLBACK");
    res.status(400).json({ erro: erro.message });
  } finally {
    client.release();
  }
};

module.exports = { transferirEstoque };

Rotas (adicione no produtoPgRoutes.js):

3. Query com JOIN e Filtro — Produtos por Preço Mínimo

Crie uma rota GET /pg/produtos-filtro?precoMin=100&categoria=1 que:

  • Use parâmetros de query string (req.query)
  • Filtre produtos com preço maior ou igual ao informado (opcional)
  • Filtre por categoria (opcional)
  • Faça LEFT JOIN com categorias para retornar o nome da categoria
  • Ordene por preço DESC
  • Use parameterized queries para todos os filtros
Gabarito: 
const listarComFiltro = async (req, res) => {
  try {
    const { precoMin, categoria } = req.query;

    let query = `
      SELECT p.*, c.nome AS categoria_nome
      FROM produtos p
      LEFT JOIN categorias c ON p.categoria_id = c.id
      WHERE 1=1
    `;

    const params = [];
    let paramIndex = 1;

    if (precoMin) {
      query += ` AND p.preco >= $${paramIndex}`;
      params.push(Number(precoMin));
      paramIndex++;
    }

    if (categoria) {
      query += ` AND p.categoria_id = $${paramIndex}`;
      params.push(Number(categoria));
      paramIndex++;
    }

    query += " ORDER BY p.preco DESC";

    const resultado = await pool.query(query, params);

    res.json(resultado.rows);
  } catch (erro) {
    res.status(500).json({ erro: "Erro ao buscar produtos" });
  }
};
