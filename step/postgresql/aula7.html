Módulo 7: Administração do Oracle DBMS

Tópicos do nosso plano de curso abordados nessa aula: Modelo de segurança Oracle (usuários, roles, GRANT/REVOKE, perfis), Backup e restauração (RMAN, Data Pump - expdp/impdp, backups lógicos e físicos).

1. Modelo de Segurança Oracle

O Oracle DBMS implementa um dos modelos de segurança mais robustos entre os SGBDs corporativos. A segurança é baseada em quatro pilares fundamentais:

  • Usuários (Users): contas individuais que se autenticam no banco de dados. Cada usuário possui um schema próprio com o mesmo nome (por padrão).
  • Roles: agrupamentos nomeados de privilégios que podem ser concedidos a usuários ou a outras roles, simplificando a gestão de permissões.
  • Privilégios (Privileges): permissões específicas para executar operações. Dividem-se em privilégios de sistema (ações globais) e privilégios de objeto (ações sobre objetos específicos).
  • Perfis (Profiles): conjuntos de limites de recursos e políticas de senha aplicáveis a usuários, controlando consumo e segurança das contas.

Fluxo típico de configuração de segurança: DBA cria usuários → cria roles com privilégios adequados → concede roles aos usuários → aplica perfis para controle de recursos e senhas.

2. Gerenciando Usuários

No Oracle, o gerenciamento de usuários é feito através de comandos DDL (Data Definition Language). O DBA (ou um usuário com privilégios administrativos) é responsável por criar, alterar e remover contas.

Criando um Usuário Básico

O comando mais simples para criar um usuário exige apenas nome e senha:


CREATE USER joao IDENTIFIED BY senha123;

O usuário é criado, mas não pode fazer login porque não possui o privilégio CREATE SESSION. Para concedê-lo:


GRANT CREATE SESSION TO joao;

Alterando Senha do Usuário

Para modificar a senha de um usuário existente:


ALTER USER joao IDENTIFIED BY nova_senha;

Desbloqueando Conta

Contas podem ser bloqueadas após múltiplas tentativas de login falhas. Para desbloquear:


ALTER USER joao ACCOUNT UNLOCK;

Para bloquear manualmente:


ALTER USER joao ACCOUNT LOCK;

Gerenciando Quota (Espaço em Tablespace)

Controla quanto espaço o usuário pode consumir em cada tablespace:


-- Conceder 100 MB no tablespace USERS
ALTER USER joao QUOTA 100M ON users;

-- Conceder quota ilimitada
ALTER USER joao QUOTA UNLIMITED ON users;

-- Revogar quota (impede criação de objetos)
ALTER USER joao QUOTA 0 ON users;

Criando Usuário Completo

Um comando mais completo, definindo tablespace padrão, tablespace temporário e quotas:


CREATE USER maria
IDENTIFIED BY senha456
DEFAULT TABLESPACE users
TEMPORARY TABLESPACE temp
QUOTA UNLIMITED ON users
QUOTA 50M ON tools;

Removendo Usuário

Para excluir um usuário e todos os seus objetos:


-- Remove apenas o usuário (falha se houver objetos)
DROP USER joao;

-- Remove usuário e todos os seus objetos (tabelas, views, etc.)
DROP USER joao CASCADE;

Atenção: O CASCADE é irreversível — todas as tabelas, views, procedures e demais objetos do schema do usuário são permanentemente excluídos.

Consultando Usuários do Banco


-- Listar todos os usuários
SELECT username, account_status, default_tablespace, created
FROM dba_users
ORDER BY username;

-- Ver quotas de um usuário específico
SELECT tablespace_name, username, max_bytes
FROM dba_ts_quotas
WHERE username = 'JOAO';

3. Roles no Oracle

Uma role é um conjunto nomeado de privilégios que pode ser concedido a usuários ou a outras roles. Roles simplificam drasticamente a administração de segurança em ambientes com muitos usuários.

Roles Predefinidas (Predefined Roles)

O Oracle vem com várias roles padrão. As três mais importantes são:

Role Privilégios Incluídos Uso Típico
CONNECT CREATE SESSION, CREATE TABLE, CREATE VIEW, CREATE SEQUENCE, CREATE SYNONYM, CREATE CLUSTER, CREATE DATABASE LINK, CREATE MATERIALIZED VIEW Desenvolvedores que precisam criar objetos básicos
RESOURCE CREATE PROCEDURE, CREATE TRIGGER, CREATE TYPE, CREATE OPERATOR, CREATE INDEXTYPE (além dos privilégios incluídos) Desenvolvedores que trabalham com PL/SQL e objetos avançados
DBA Praticamente todos os privilégios de sistema, incluindo ADMIN OPTION Administradores do banco de dados 

-- Conceder roles predefinidas a um usuário
GRANT CONNECT, RESOURCE TO joao;
GRANT DBA TO maria;

Criando Roles Personalizadas

Roles customizadas permitem agrupar exatamente os privilégios necessários para cada função na organização:


-- Criar uma role
CREATE ROLE desenvolvedor;

-- Conceder privilégios de sistema à role
GRANT CREATE SESSION TO desenvolvedor;
GRANT CREATE TABLE TO desenvolvedor;
GRANT CREATE VIEW TO desenvolvedor;
GRANT CREATE SEQUENCE TO desenvolvedor;
GRANT CREATE PROCEDURE TO desenvolvedor;
GRANT CREATE TRIGGER TO desenvolvedor;

-- Conceder a role ao usuário
GRANT desenvolvedor TO joao;

Ativando e Configurando Roles

Um usuário pode ter múltiplas roles, mas nem todas ficam ativas por padrão:


-- Ativar uma role específica na sessão
SET ROLE desenvolvedor;

-- Ativar todas as roles concedidas ao usuário
SET ROLE ALL;

-- Ativar todas exceto uma específica
SET ROLE ALL EXCEPT dba;

-- Definir role padrão (ativa automaticamente no login)
ALTER USER joao DEFAULT ROLE desenvolvedor;

-- Definir todas as roles como padrão
ALTER USER joao DEFAULT ROLE ALL;

-- Definir todas exceto uma
ALTER USER joao DEFAULT ROLE ALL EXCEPT dba;

Hierarquia de Roles

Roles podem ser concedidas a outras roles, criando uma hierarquia de permissões:


-- Criar hierarquia de roles
CREATE ROLE estagiario;
CREATE ROLE desenvolvedor;
CREATE ROLE senior;
CREATE ROLE tech_lead;

-- Conceder hierarquia (cada nível herda o anterior)
GRANT CREATE SESSION, CREATE TABLE, CREATE VIEW TO estagiario;
GRANT estagiario TO desenvolvedor;
GRANT CREATE PROCEDURE, CREATE TRIGGER TO desenvolvedor;
GRANT desenvolvedor TO senior;
GRANT CREATE ANY TABLE, DROP ANY TABLE TO senior;
GRANT senior TO tech_lead;
GRANT ALTER DATABASE, CREATE USER TO tech_lead;

-- tech_lead herda todos os privilégios abaixo na cadeia

Removendo Roles


-- Revogar role de um usuário
REVOKE desenvolvedor FROM joao;

-- Excluir a role do banco
DROP ROLE estagiario;

4. Privilégios de Sistema e Objeto

O Oracle divide os privilégios em duas categorias: privilégios de sistema (ações globais no banco) e privilégios de objeto (ações sobre objetos específicos como tabelas, views e procedures).

Privilégios de Sistema Mais Comuns

Privilégio Permite
CREATE SESSIONConectar ao banco de dados
CREATE TABLECriar tabelas no próprio schema
CREATE ANY TABLECriar tabelas em qualquer schema
CREATE VIEWCriar views no próprio schema
CREATE PROCEDURECriar procedures e functions
CREATE TRIGGERCriar triggers
CREATE SEQUENCECriar sequences
CREATE USERCriar usuários (privilégio de DBA)
DROP USERRemover usuários
ALTER DATABASEAlterar configurações do banco
UNLIMITED TABLESPACEUsar espaço ilimitado em qualquer tablespace
SELECT ANY TABLEConsultar qualquer tabela do banco
BACKUP ANY TABLEFazer backup de qualquer tabela
 

-- Conceder privilégios de sistema
GRANT CREATE SESSION TO desenvolvedor;
GRANT CREATE TABLE TO desenvolvedor;
GRANT UNLIMITED TABLESPACE TO desenvolvedor;
GRANT SELECT ANY TABLE TO auditor;

-- Com ADMIN OPTION (permite repassar o privilégio)
GRANT CREATE SESSION TO rh_admin WITH ADMIN OPTION;

-- Revogar privilégios de sistema
REVOKE CREATE TABLE FROM desenvolvedor;
REVOKE UNLIMITED TABLESPACE FROM desenvolvedor;

Privilégios de Objeto

Controlam o acesso a objetos específicos dentro de schemas:

Privilégio Permite em Tabelas Permite em Views Permite em Procedures
SELECTConsultar dadosConsultar a view—
INSERTInserir linhasInserir via view—
UPDATEAtualizar linhasAtualizar via view—
DELETEExcluir linhasExcluir via view—
REFERENCESCriar FK referenciando——
ALTERAlterar estruturaAlterar definição—
INDEXCriar índices——
EXECUTE——Executar a procedure
 

-- Conceder privilégios de objeto (schema dono executa)
GRANT SELECT ON funcionarios TO desenvolvedor;
GRANT SELECT, INSERT, UPDATE ON funcionarios TO rh_admin;
GRANT EXECUTE ON calcular_salario TO rh_admin;

-- Conceder múltiplos privilégios de uma vez
GRANT SELECT, INSERT, UPDATE, DELETE ON departamentos TO desenvolvedor;

-- Conceder em todas as tabelas de um schema
GRANT SELECT ON hr.departamentos TO desenvolvedor;
GRANT SELECT ON hr.funcionarios TO desenvolvedor;

Privilégios em Nível de Coluna

É possível restringir privilégios a colunas específicas de uma tabela:


-- Permitir UPDATE apenas na coluna salario
GRANT UPDATE (salario) ON funcionarios TO rh_admin;

-- Permitir INSERT apenas em colunas específicas
GRANT INSERT (nome, cargo, departamento_id) ON funcionarios TO rh_admin;

-- Permitir REFERENCES em coluna específica
GRANT REFERENCES (departamento_id) ON departamentos TO desenvolvedor;

WITH GRANT OPTION

Permite que o recebedor do privilégio o repasse para outros usuários. Equivalente ao WITH ADMIN OPTION dos privilégios de sistema, mas aplicável a privilégios de objeto: 


-- Com WITH GRANT OPTION, o RH pode conceder SELECT a terceiros
GRANT SELECT ON funcionarios TO rh_admin WITH GRANT OPTION;

-- rh_admin agora pode repassar:
-- rh_admin executa:
GRANT SELECT ON hr.funcionarios TO estagiario;

Revogando Privilégios de Objeto


-- Revogar privilégios específicos
REVOKE INSERT ON funcionarios FROM estagiario;

-- Revogar todos os privilégios concedidos a um usuário
REVOKE ALL ON funcionarios FROM estagiario;

-- Revogar um privilégio que foi repassado via GRANT OPTION
-- CASCADE remove também de quem recebeu indiretamente
REVOKE SELECT ON funcionarios FROM rh_admin CASCADE;

Verificando Privilégios Concedidos


-- Privilégios de sistema concedidos a um usuário
SELECT * FROM dba_sys_privs
WHERE grantee = 'JOAO';

-- Privilégios de sistema concedidos a roles
SELECT * FROM dba_sys_privs
WHERE grantee = 'DESENVOLVEDOR';

-- Privilégios de objeto concedidos
SELECT grantee, owner, table_name, privilege
FROM dba_tab_privs
WHERE grantee IN ('JOAO', 'DESENVOLVEDOR');

-- Roles concedidas a um usuário
SELECT * FROM dba_role_privs
WHERE grantee = 'JOAO';

5. Perfis (Profiles)

Um profile é um conjunto nomeado de limites de recursos e políticas de senha que pode ser atribuído a usuários. Profiles permitem ao DBA controlar o consumo de recursos do banco e impor políticas de segurança nas senhas.

Parâmetros de Limite de Recursos (Kernel)

Parâmetro Descrição
SESSIONS_PER_USERNúmero máximo de sessões simultâneas por usuário
CPU_PER_SESSIONTempo máximo de CPU por sessão (em centésimos de segundo)
CPU_PER_CALLTempo máximo de CPU por chamada SQL
CONNECT_TIMETempo máximo de conexão (em minutos)
IDLE_TIMETempo máximo de inatividade antes de desconectar (em minutos)
LOGICAL_READS_PER_SESSIONMáximo de blocos lidos por sessão
LOGICAL_READS_PER_CALLMáximo de blocos lidos por chamada SQL
COMPOSITE_LIMITLimite composto baseado em custo total
PRIVATE_SGAMemória privada máxima na SGA por sessão

Parâmetros de Política de Senha

Parâmetro Descrição
FAILED_LOGIN_ATTEMPTSTentativas de login falhas antes do bloqueio da conta
PASSWORD_LOCK_TIMEDias que a conta fica bloqueada após atingir o limite de falhas
PASSWORD_LIFE_TIMEDias de vida útil da senha antes de expirar
PASSWORD_GRACE_TIMEDias de "período de graça" após expiração (ainda pode logar, mas recebe aviso)
PASSWORD_REUSE_TIMEDias antes de poder reutilizar uma senha antiga
PASSWORD_REUSE_MAXNúmero de alterações antes de poder reutilizar uma senha
PASSWORD_VERIFY_FUNCTIONFunção PL/SQL para validação de complexidade da senha

Criando e Aplicando Profiles


-- Criar um profile com limites de recursos e políticas de senha
CREATE PROFILE perfil_padrao LIMIT
    SESSIONS_PER_USER       3
    IDLE_TIME               30
    FAILED_LOGIN_ATTEMPTS   5
    PASSWORD_LOCK_TIME      1
    PASSWORD_LIFE_TIME      90
    PASSWORD_GRACE_TIME     7
    PASSWORD_REUSE_TIME     365
    PASSWORD_REUSE_MAX      10
    CPU_PER_SESSION         UNLIMITED
    CONNECT_TIME            120;

-- Aplicar o profile a um usuário
ALTER USER joao PROFILE perfil_padrao;

-- Aplicar a múltiplos usuários
ALTER USER maria PROFILE perfil_padrao;
ALTER USER pedro PROFILE perfil_padrao;

Profile Específico para Aplicações


CREATE PROFILE perfil_app LIMIT
    SESSIONS_PER_USER       10
    IDLE_TIME               60
    CONNECT_TIME            480
    CPU_PER_SESSION         10000
    FAILED_LOGIN_ATTEMPTS   3
    PASSWORD_LOCK_TIME      1
    PASSWORD_LIFE_TIME      60;

-- Aplicar ao usuário da aplicação
ALTER USER app_user PROFILE perfil_app;

Consultando Profiles


-- Listar todos os profiles e seus limites
SELECT * FROM dba_profiles
ORDER BY profile, resource_name;

-- Ver qual profile um usuário está usando
SELECT username, profile
FROM dba_users
WHERE username = 'JOAO';

-- Ver usuários com um profile específico
SELECT username, account_status, profile
FROM dba_users
WHERE profile = 'PERFIL_PADRAO';

Alterando e Removendo Profiles


-- Alterar limites de um profile existente
ALTER PROFILE perfil_padrao LIMIT
    SESSIONS_PER_USER       5
    PASSWORD_LIFE_TIME      120
    IDLE_TIME               45;

-- Remover um profile (usuários voltam ao DEFAULT)
DROP PROFILE perfil_app;

-- Remover com CASCADE (desassocia de todos os usuários)
DROP PROFILE perfil_app CASCADE;

Importante: Para que os limites de recursos (CPU, leituras lógicas, tempo de conexão) sejam efetivamente impostos, o parâmetro de inicialização RESOURCE_LIMIT deve estar configurado como TRUE: 


-- Verificar se RESOURCE_LIMIT está ativo
SHOW PARAMETER resource_limit;

-- Ativar (requer privilégios de DBA)
ALTER SYSTEM SET resource_limit = TRUE SCOPE=BOTH;

6. Backup e Restauração - Visão Geral

A proteção dos dados é uma das responsabilidades mais críticas de um DBA. O Oracle oferece múltiplas ferramentas e estratégias de backup, cada uma adequada a diferentes cenários. Os backups se classificam em duas grandes categorias:

Backup Físico

  • Cópia dos arquivos físicos do banco (datafiles, control files, redo logs, archive logs)
  • Pode ser feito com o banco online (hot backup) ou offline (cold backup)
  • Ferramenta principal: RMAN (Recovery Manager)
  • Permite recuperação point-in-time (PITR)
  • Mais rápido para bancos grandes
  • Backup e restore em nível de bloco

Backup Lógico

  • Extrai a definição lógica e os dados dos objetos (tabelas, views, procedures, etc.)
  • Gera arquivos dump portáteis (podem ser importados em versões diferentes)
  • Ferramenta principal: Data Pump (expdp/impdp)
  • Mais granular: permite exportar schemas, tabelas ou tablespaces específicos
  • Útil para migração entre versões, cópia seletiva de dados, refreshes de ambientes
  • Mais lento que RMAN para bancos grandes

Estratégia ideal: Combinar backups físicos regulares (RMAN) para recuperação completa com backups lógicos pontuais (Data Pump) para granularidade e portabilidade.

7. RMAN (Recovery Manager)

O RMAN é a ferramenta nativa do Oracle para backup e recuperação física. Ele opera em nível de bloco e é integrado ao kernel do Oracle, permitindo backups online (com o banco em operação) e recuperação point-in-time.

Conectando ao RMAN


# Conectar como SYSDBA (autenticação OS)
rman target /

# Conectar com usuário e senha
rman target system/senha

# Conectar a um banco remoto (via TNS)
rman target system/senha@ORCL

# Conectar ao banco target e ao catálogo de recuperação
rman target / catalog rman_cat_user/senha@catalog_db

Verificando Configuração RMAN


# Mostrar todas as configurações do RMAN
RMAN> SHOW ALL;

# Configurar retenção de backup (7 dias)
RMAN> CONFIGURE RETENTION POLICY TO RECOVERY WINDOW OF 7 DAYS;

# Configurar dispositivo padrão para disco
RMAN> CONFIGURE DEFAULT DEVICE TYPE TO DISK;

# Configurar paralelismo (usar 2 canais simultâneos)
RMAN> CONFIGURE DEVICE TYPE DISK PARALLELISM 2;

# Configurar local de backup
RMAN> CONFIGURE CHANNEL DEVICE TYPE DISK FORMAT '/backup/%d_%T_%s_%p.bkp';

# Configurar política de redundância (manter 2 cópias)
RMAN> CONFIGURE RETENTION POLICY TO REDUNDANCY 2;

Backup Full Database


# Backup completo do banco (inclui todos os datafiles, controlfile e spfile)
RMAN> BACKUP DATABASE;

# Backup completo com tag para identificação
RMAN> BACKUP DATABASE TAG 'backup_semanal';

# Backup completo incluindo archive logs e deletando os já armazenados
RMAN> BACKUP DATABASE PLUS ARCHIVELOG DELETE INPUT;

# Backup completo com formato de arquivo customizado
RMAN> BACKUP DATABASE FORMAT '/backup/full_%d_%T_%s.bkp';

Backup de Tablespace Específico


# Backup de um tablespace específico
RMAN> BACKUP TABLESPACE users;

# Backup de múltiplos tablespaces
RMAN> BACKUP TABLESPACE users, tools, data_ts;

# Backup de tablespace incluindo archive logs
RMAN> BACKUP TABLESPACE users PLUS ARCHIVELOG;

Backup Incremental

Backups incrementais economizam tempo e espaço, copiando apenas os blocos que mudaram desde o último backup:


# Backup incremental nível 0 (base para incrementais futuros - cópia completa)
RMAN> BACKUP INCREMENTAL LEVEL 0 DATABASE;

# Backup incremental nível 1 (apenas blocos alterados desde o nível 0)
RMAN> BACKUP INCREMENTAL LEVEL 1 DATABASE;

# Backup incremental cumulativo nível 1 (desde o último nível 0)
RMAN> BACKUP INCREMENTAL LEVEL 1 CUMULATIVE DATABASE;

# Backup incremental de tablespace
RMAN> BACKUP INCREMENTAL LEVEL 1 TABLESPACE users;

Backup de Archive Logs


# Backup de todos os archive logs
RMAN> BACKUP ARCHIVELOG ALL;

# Backup de archive logs e deletar os originais (libera espaço)
RMAN> BACKUP ARCHIVELOG ALL DELETE INPUT;

# Backup de archive logs a partir de uma data
RMAN> BACKUP ARCHIVELOG FROM TIME "SYSDATE-7";

# Backup de archive logs por sequência
RMAN> BACKUP ARCHIVELOG FROM SEQUENCE 100 UNTIL SEQUENCE 150;

Backup de Control File e SPFile


# Backup do control file atual
RMAN> BACKUP CURRENT CONTROLFILE;

# Backup do spfile (server parameter file)
RMAN> BACKUP SPFILE;

# Configurar backup automático de controlfile e spfile
RMAN> CONFIGURE CONTROLFILE AUTOBACKUP ON;
RMAN> CONFIGURE CONTROLFILE AUTOBACKUP FORMAT FOR DEVICE TYPE DISK TO '/backup/cf_%d_%F.bkp';

Listando Backups


# Listar todos os backups
RMAN> LIST BACKUP;

# Listar backups do banco inteiro
RMAN> LIST BACKUP OF DATABASE;

# Listar backups de um tablespace
RMAN> LIST BACKUP OF TABLESPACE users;

# Listar backups por tag
RMAN> LIST BACKUP TAG 'backup_semanal';

# Listar resumo compacto dos backups
RMAN> LIST BACKUP SUMMARY;

# Listar backups que expiraram (de acordo com a política de retenção)
RMAN> LIST EXPIRED BACKUP;

Validando Backups


# Verificar se um backup é válido (sem restaurar)
RMAN> RESTORE DATABASE VALIDATE;

# Validar um backup específico
RMAN> RESTORE TABLESPACE users VALIDATE;

# Verificar integridade física dos arquivos de backup
RMAN> CROSSCHECK BACKUP;

# Verificar backups obsoletos
RMAN> REPORT OBSOLETE;

Restauração e Recuperação (Restore & Recover)


# Restaurar e recuperar o banco inteiro
RMAN> RESTORE DATABASE;
RMAN> RECOVER DATABASE;

# Restaurar um tablespace específico
RMAN> RESTORE TABLESPACE users;
RMAN> RECOVER TABLESPACE users;

# Restaurar um datafile específico
RMAN> RESTORE DATAFILE '/u01/oradata/ORCL/users01.dbf';
RMAN> RECOVER DATAFILE '/u01/oradata/ORCL/users01.dbf';

# Restaurar control file do autobackup
RMAN> RESTORE CONTROLFILE FROM AUTOBACKUP;

Recuperação Point-in-Time (PITR)

A recuperação point-in-time permite restaurar o banco para um estado exato em uma data/hora específica no passado:


# Recuperar até uma data/hora específica
RMAN> RUN {
    SET UNTIL TIME "TO_DATE('2025-06-20 10:00:00', 'YYYY-MM-DD HH24:MI:SS')";
    RESTORE DATABASE;
    RECOVER DATABASE;
}

# Recuperar até um SCN (System Change Number) específico
RMAN> RUN {
    SET UNTIL SCN 1234567;
    RESTORE DATABASE;
    RECOVER DATABASE;
}

# Recuperar até um sequence number de archive log
RMAN> RUN {
    SET UNTIL SEQUENCE 150;
    RESTORE DATABASE;
    RECOVER DATABASE;
}

# Abrir o banco com RESETLOGS após PITR
SQL> ALTER DATABASE OPEN RESETLOGS;

Recuperação de Tablespace Específico (TSPITR)


# Recuperar tablespace users para 2 dias atrás
RMAN> RECOVER TABLESPACE users
    UNTIL TIME "SYSDATE-2"
    AUXILIARY DESTINATION '/tmp/aux';

Removendo Backups Antigos


# Deletar backups obsoletos (fora da política de retenção)
RMAN> DELETE OBSOLETE;

# Deletar backups expirados
RMAN> DELETE EXPIRED BACKUP;

# Deletar um backup específico
RMAN> DELETE BACKUP TAG 'backup_semanal';

# Deletar backups mais antigos que 30 dias
RMAN> DELETE BACKUP COMPLETED BEFORE "SYSDATE-30";

# Forçar deleção (ignorar erros de arquivo não encontrado)
RMAN> DELETE EXPIRED BACKUP FORCE;

8. Data Pump (expdp/impdp)

O Oracle Data Pump é a ferramenta de backup lógico do Oracle, sucessora do antigo exp/imp. Ela exporta e importa metadados e dados em formato proprietário (.dmp), oferecendo alta performance, paralelismo e filtros avançados.

Pré-requisito: Criando um Directory Object

Antes de usar o Data Pump, é necessário criar um directory object que aponte para um diretório físico no servidor:


-- Criar diretório físico no sistema operacional (como usuário oracle)
-- mkdir -p /backup/datapump

-- Criar o directory object no Oracle
CREATE OR REPLACE DIRECTORY dpump_dir AS '/backup/datapump';

-- Conceder permissão de leitura/escrita no diretório
GRANT READ, WRITE ON DIRECTORY dpump_dir TO system;
GRANT READ, WRITE ON DIRECTORY dpump_dir TO joao;

-- Listar todos os directory objects
SELECT * FROM dba_directories;

Export de Schema (Usuário)


# Exportar um schema completo
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=backup_rh.dmp LOGFILE=export_rh.log SCHEMAS=rh

# Exportar múltiplos schemas
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=backup_schemas.dmp LOGFILE=export_schemas.log SCHEMAS=rh,financeiro,vendas

# Exportar schema com paralelismo (2 workers)
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=backup_rh_%U.dmp LOGFILE=export_rh.log SCHEMAS=rh PARALLEL=2

# Exportar schema sem dados (apenas metadados - estrutura)
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=estrutura_rh.dmp SCHEMAS=rh CONTENT=METADATA_ONLY

Export Full Database


# Export completo de todo o banco
expdp system/senha FULL=Y DIRECTORY=dpump_dir DUMPFILE=full_backup.dmp LOGFILE=export_full.log

# Export completo com paralelismo
expdp system/senha FULL=Y DIRECTORY=dpump_dir DUMPFILE=full_%U.dmp PARALLEL=4

# Export completo apenas metadados
expdp system/senha FULL=Y DIRECTORY=dpump_dir DUMPFILE=full_metadata.dmp CONTENT=METADATA_ONLY

Export de Tabelas Específicas


# Exportar tabelas específicas de um schema
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=tabelas_rh.dmp LOGFILE=export_tabelas.log TABLES=rh.funcionarios,rh.departamentos

# Exportar com filtro WHERE (apenas linhas que atendem condição)
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=func_ativo.dmp TABLES=rh.funcionarios QUERY='WHERE status=''ATIVO'''

# Exportar com filtro de partição
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=tab_part.dmp TABLES=rh.vendas PARTITION=p_2025

Export de Tablespace


# Exportar todos os objetos de um tablespace
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=tbs_users.dmp TABLESPACES=users

# Exportar múltiplos tablespaces
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=tbs_backup.dmp TABLESPACES=users,data_ts

Opções de Conteúdo (CONTENT)


# Apenas metadados (DDL - estrutura sem dados)
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=metadata.dmp SCHEMAS=rh CONTENT=METADATA_ONLY

# Apenas dados (sem estrutura - requer que tabelas já existam no destino)
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=dados.dmp SCHEMAS=rh CONTENT=DATA_ONLY

# Ambos (padrão) - metadados + dados
expdp system/senha DIRECTORY=dpump_dir DUMPFILE=completo.dmp SCHEMAS=rh CONTENT=ALL

Import de Schema


# Importar schema completo (cria objetos e insere dados)
impdp system/senha DIRECTORY=dpump_dir DUMPFILE=backup_rh.dmp LOGFILE=import_rh.log SCHEMAS=rh

# Importar schema com remapeamento (schema origem -> schema destino)
impdp system/senha DIRECTORY=dpump_dir DUMPFILE=backup_rh.dmp REMAP_SCHEMA=rh:rh_teste

# Importar schema com remapeamento de tablespace
impdp system/senha DIRECTORY=dpump_dir DUMPFILE=backup_rh.dmp REMAP_TABLESPACE=users:tbs_novo

# Importar schema com paralelismo
impdp system/senha DIRECTORY=dpump_dir DUMPFILE=backup_rh_%U.dmp SCHEMAS=rh PARALLEL=2

Import Full Database


# Import completo (restauração total)
impdp system/senha FULL=Y DIRECTORY=dpump_dir DUMPFILE=full_backup.dmp LOGFILE=import_full.log

# Import full com transformação (não importar storage clauses)
impdp system/senha FULL=Y DIRECTORY=dpump_dir DUMPFILE=full_backup.dmp TRANSFORM=STORAGE:N

Import de Tabelas Específicas


# Importar tabelas específicas de um dump
impdp system/senha DIRECTORY=dpump_dir DUMPFILE=tabelas_rh.dmp TABLES=rh.funcionarios

# Importar apenas dados (sem recriar a tabela - TABLE_EXISTS_ACTION=APPEND)
impdp system/senha DIRECTORY=dpump_dir DUMPFILE=dados.dmp SCHEMAS=rh CONTENT=DATA_ONLY TABLE_EXISTS_ACTION=APPEND

# Importar truncando a tabela antes (TABLE_EXISTS_ACTION=TRUNCATE)
impdp system/senha DIRECTORY=dpump_dir DUMPFILE=tabelas_rh.dmp TABLES=rh.funcionarios TABLE_EXISTS_ACTION=TRUNCATE

Opções TABLE_EXISTS_ACTION

9. Comparação RMAN vs Data Pump

Característica RMAN Data Pump
Tipo de Backup Físico (blocos de dados) Lógico (DDL + dados)
Velocidade Muito rápido (nível de bloco) Moderado a rápido (depende de paralelismo)
Granularidade Banco inteiro, tablespace, datafile, archive log Banco inteiro, schema, tabela, partição
Recuperação Point-in-Time Sim (completo, via archive logs) Não (restaura o estado do momento da exportação)
Backup Online Sim (requer ARCHIVELOG mode) Sim (sempre online)
Portabilidade entre versões Limitada (mesma plataforma e versão) Alta (versões diferentes, plataformas diferentes)
Incremental Sim (nível 0 e nível 1) Não
Compressão Sim (built-in, vários algoritmos) Sim (COMPRESSION=ALL)
Criptografia Sim (ENCRYPTION) Sim (ENCRYPTION_PASSWORD)
Paralelismo Sim (canais paralelos) Sim (PARALLEL parameter, workers)
Filtragem de dados Não Sim (QUERY, SAMPLE, INCLUDE/EXCLUDE)
Migração cross-platform Limitada (requer RMAN CONVERT) Sim (total)
Uso principal Recuperação de desastres, backup diário Migração, cópia parcial, refreshes, exportação seletiva

Vamos configurar um sistema de backup automatizado que executa exports lógicos diários usando Data Pump, agendados via DBMS_SCHEDULER.

Passo 3 — Criar uma procedure de backup


CREATE OR REPLACE PROCEDURE prc_backup_diario AS
    v_job_name   VARCHAR2(200);
    v_job_action VARCHAR2(4000);
BEGIN
    v_job_name := 'DATAPUMP_DAILY_' || TO_CHAR(SYSDATE, 'YYYYMMDD_HH24MISS');

    v_job_action := 'expdp system/senha' ||
        ' DIRECTORY=dpump_daily' ||
        ' DUMPFILE=backup_diario_' || TO_CHAR(SYSDATE, 'YYYYMMDD') || '.dmp' ||
        ' LOGFILE=backup_diario_' || TO_CHAR(SYSDATE, 'YYYYMMDD') || '.log' ||
        ' SCHEMAS=rh,financeiro,vendas' ||
        ' COMPRESSION=ALL' ||
        ' PARALLEL=2';

    DBMS_SCHEDULER.CREATE_JOB(
        job_name   => v_job_name,
        job_type   => 'EXECUTABLE',
        job_action => '/bin/sh',
        number_of_arguments => 2,
        enabled    => FALSE
    );

    DBMS_SCHEDULER.SET_JOB_ARGUMENT_VALUE(v_job_name, 1, '-c');
    DBMS_SCHEDULER.SET_JOB_ARGUMENT_VALUE(v_job_name, 2, v_job_action);
    DBMS_SCHEDULER.ENABLE(v_job_name);

    DBMS_OUTPUT.PUT_LINE('Job Data Pump criado: ' || v_job_name);
END;
/

Dever de Casa

1. Crie uma role "relatorios" com permissão SELECT em todas as tabelas e atribua a um usuário

Crie a role relatorios, conceda o privilégio SELECT ANY TABLE a ela, crie um usuário analista com autenticação por senha, conceda a role ao usuário e configure a role como padrão. Além disso, conceda CREATE SESSION à role.

Gabarito: 

-- Passo 1: Criar a role
CREATE ROLE relatorios;

-- Passo 2: Conceder privilégios à role
GRANT CREATE SESSION TO relatorios;
GRANT SELECT ANY TABLE TO relatorios;

-- Passo 3: Criar o usuário
CREATE USER analista
IDENTIFIED BY senha_segura
DEFAULT TABLESPACE users
TEMPORARY TABLESPACE temp
QUOTA UNLIMITED ON users;

-- Passo 4: Conceder a role ao usuário
GRANT relatorios TO analista;

-- Passo 5: Configurar role padrão
ALTER USER analista DEFAULT ROLE relatorios;

Observação: O privilégio SELECT ANY TABLE é muito poderoso — permite consultar qualquer tabela de qualquer schema, incluindo tabelas do SYS. Em ambientes de produção, prefira conceder SELECT em tabelas específicas.

2. Crie um backup full com RMAN e verifique com LIST BACKUP

Conecte ao RMAN como target, execute um backup completo do banco com a tag 'aula7_backup', incluindo archive logs e deletando os logs já arquivados. Em seguida, liste os backups para confirmar que o backup foi criado com sucesso.

Gabarito: 

# Passo 1: Conectar ao RMAN
rman target /

# Passo 2: Executar backup completo
RMAN> BACKUP DATABASE PLUS ARCHIVELOG DELETE INPUT TAG 'aula7_backup';

# Passo 3: Verificar o backup criado
RMAN> LIST BACKUP TAG 'aula7_backup';

# Passo 4: Ver resumo de todos os backups
RMAN> LIST BACKUP SUMMARY;

# Passo 5: Validar integridade do backup
RMAN> RESTORE DATABASE VALIDATE;

A saída do LIST BACKUP deve mostrar os datafiles, o control file, o spfile e os archive logs incluídos no backup. Verifique se não há erros na coluna STATUS.

3. Exporte um schema específico com Data Pump e depois faça a importação com remapeamento

Exporte o schema rh usando Data Pump. Em seguida, faça a importação do mesmo dump para um novo schema chamado rh_teste usando a opção REMAP_SCHEMA. Use o directory object dpump_dir.

Gabarito: 

# Passo 1: Criar o diretório (se não existir)
mkdir -p /backup/datapump

# Passo 2: Criar o directory object no Oracle
sqlplus system/senha
SQL> CREATE OR REPLACE DIRECTORY dpump_dir AS '/backup/datapump';
SQL> GRANT READ, WRITE ON DIRECTORY dpump_dir TO system;

# Passo 3: Exportar o schema rh
expdp system/senha \
    DIRECTORY=dpump_dir \
    DUMPFILE=rh_backup.dmp \
    LOGFILE=rh_export.log \
    SCHEMAS=rh

# Passo 4: Criar schema de destino (se não existir)
sqlplus system/senha
SQL> CREATE USER rh_teste IDENTIFIED BY senha123
     DEFAULT TABLESPACE users TEMPORARY TABLESPACE temp
     QUOTA UNLIMITED ON users;
SQL> GRANT CONNECT, RESOURCE TO rh_teste;

# Passo 5: Importar com remapeamento de schema
impdp system/senha \
    DIRECTORY=dpump_dir \
    DUMPFILE=rh_backup.dmp \
    LOGFILE=rh_import.log \
    REMAP_SCHEMA=rh:rh_teste

# Passo 6: Verificar objetos importados
sqlplus system/senha
SQL> SELECT object_type, COUNT(*) FROM dba_objects
     WHERE owner = 'RH_TESTE'
     GROUP BY object_type
     ORDER BY object_type;

A opção REMAP_SCHEMA é extremamente útil para clonar ambientes (ex: produzir uma cópia de produção em testes). Todos os objetos do schema origem são recriados no schema destino, preservando constraints, índices e grants.
