História do Banco de Dados Oracle

O Oracle Database é um dos SGBDs (Sistema de Gerenciamento de Banco de Dados) mais antigos e influentes do mercado. Sua história começa com uma ideia visionária que revolucionou a forma como armazenamos e consultamos dados.

Fundação e Primeiros Anos

Em 1977, Larry Ellison, Bob Miner e Ed Oates fundaram a Software Development Laboratories (SDL), que mais tarde se tornaria a Oracle Corporation. A inspiração veio de um artigo científico de Edgar F. Codd, publicado em 1970, que descrevia o modelo relacional de banco de dados.

Em 1979, a empresa lançou o Oracle V2 (não houve versão 1 por questões de marketing), tornando-se o primeiro SGBD relacional comercial do mundo. Escrito em Assembly, o Oracle V2 já implementava SQL básico e transações.

Arquitetura do Oracle DBMS

A arquitetura do Oracle DBMS é composta por dois componentes principais: a instância e o banco de dados. Entender essa distinção é fundamental para compreender como o Oracle funciona.

Instância vs Banco de Dados

Banco de dados: conjunto de arquivos físicos no disco (data files, control files, redo logs). É a estrutura física que armazena os dados de forma persistente.

Instância: conjunto de estruturas de memória (SGA) e processos em background que acessam e gerenciam o banco de dados. A instância é iniciada quando o Oracle é aberto (startup) e pode existir sem um banco de dados associado (modo NOMOUNT).

Em uma configuração RAC (Real Application Clusters), múltiplas instâncias podem acessar o mesmo banco de dados físico simultaneamente.

SGA — System Global Area

A SGA é uma região de memória compartilhada que contém dados e informações de controle da instância. Principais componentes:

  • Database Buffer Cache: armazena blocos de dados lidos dos data files em memória para acesso rápido
  • Redo Log Buffer: armazena registros de alterações (redo entries) antes de serem gravados nos redo logs físicos
  • Shared Pool: subdividido em Library Cache (armazena SQL parseado e planos de execução) e Data Dictionary Cache (armazena informações do dicionário de dados)
  • Large Pool: memória opcional para operações de backup, processos de servidor compartilhados (shared server) e UGA
  • Java Pool: memória para execução de código Java armazenado no banco
  • Streams Pool: memória para Oracle Streams (replicação)

PGA — Program Global Area

A PGA é uma região de memória privada de cada processo servidor. Contém:

  • Sort Area: usada para operações de ordenação (ORDER BY, GROUP BY)
  • Hash Area: usada para hash joins
  • Bitmap Merge Area: usada para operações de bitmap
  • Session Memory: informações de login e variáveis de sessão

O tamanho total da PGA é controlado pelo parâmetro PGA_AGGREGATE_TARGET.

Processos Background

O Oracle mantém diversos processos em segundo plano essenciais para o funcionamento do SGBD:

Processo Sigla Função
Database Writer DBWn Escreve blocos alterados (dirty buffers) do buffer cache para os data files
Log Writer LGWR Escreve entradas do redo log buffer para os redo log files
Checkpoint CKPT Sinaliza os checkpoints, atualiza cabeçalhos dos data files e control files
System Monitor SMON Recuperação de instância após crash, libera segmentos temporários, coalesce espaço livre
Process Monitor PMON Limpa processos que falharam, libera locks, faz rollback de transações não commitadas
Archiver ARCn Copia redo logs preenchidos para arquivos de archive log (modo ARCHIVELOG)
Recoverer RECO Resolve transações distribuídas pendentes (two-phase commit)
Listener Registration LREG Registra serviços no listener Oracle
Manageability Monitor MMON Coleta estatísticas AWR (Automatic Workload Repository), dispara alertas

Estrutura Física do Banco de Dados

  • Data Files: arquivos que armazenam os dados de tabelas, índices, e todos os segmentos do banco. Extensão comum: .dbf
  • Control Files: arquivos binários que contêm metadados sobre a estrutura física do banco (nome do banco, localização dos data files e redo logs, timestamp de criação, SCN atual). Extensão comum: .ctl
  • Redo Log Files: registram todas as alterações feitas nos dados. Essenciais para recuperação em caso de falha. Extensão comum: .log
  • Archive Log Files: cópias offline dos redo logs preenchidos, usados para recuperação completa (point-in-time recovery)
  • Parameter File: arquivo de inicialização (SPFILE ou PFILE) com parâmetros de configuração da instância
  • Password File: armazena senhas de usuários com privilégios SYSDBA e SYSOPER

Tablespaces — Estrutura Lógica

Um tablespace é um contêiner lógico que agrupa segmentos (tabelas, índices). Cada tablespace é composto por um ou mais data files. Principais tablespaces:

  • SYSTEM: armazena o dicionário de dados (metadados do banco)
  • SYSAUX: auxiliar do SYSTEM, armazena metadados de ferramentas como AWR, EM, etc.
  • UNDO (UNDOTBS): armazena dados de desfazer (rollback), consistência de leitura e recuperação
  • TEMP: armazena segmentos temporários para ordenação e operações de hash
  • USERS: tablespace padrão para objetos de usuários comuns

Esquema Lógico Completo

A hierarquia lógica de armazenamento do Oracle é:

  • Database → contém um ou mais Tablespaces
  • Tablespace → composto por um ou mais Data Files
  • Tablespace → contém Segments (tabelas, índices, etc.)
  • Segment → composto por Extents (unidade de alocação)
  • Extent → conjunto contíguo de Oracle Blocks
  • Oracle Block → menor unidade de I/O (padrão 8 KB)

Versões do Oracle e Suas Principais Características

Oracle 8i (1999)

O "i" significa Internet. Foi a primeira versão orientada à web, com suporte nativo a Java (JVM integrada no banco) e XML DB para armazenamento de documentos XML. Introduziu o Oracle Internet File System (iFS).

Oracle 9i (2001)

Grande avanço em alta disponibilidade com o Real Application Clusters (RAC), permitindo múltiplas instâncias acessarem o mesmo banco. Introduziu o Flashback Query (consultar dados como estavam em um momento passado) e o Oracle Data Guard para disaster recovery.

Oracle 10g (2004)

O "g" significa Grid Computing. Foco em gerenciamento automático: Automatic Storage Management (ASM), Automatic Database Diagnostic Monitor (ADDM), Automatic Workload Repository (AWR). Introduziu o Flashback Database completo e o Oracle Enterprise Manager Database Control.

Oracle 11g (2007)

Versão madura com melhorias incrementais poderosas: Real Application Testing (captura e replay de workloads), Advanced Compression (economia de espaço), Total Recall (Flashback Data Archive para compliance), SecureFiles (evolução dos LOBs), e Result Cache (cache de consultas e funções).

Oracle 12c (2013)

O "c" significa Cloud. Revolucionou a arquitetura com o modelo Multitenant: um Container Database (CDB) pode abrigar múltiplos Pluggable Databases (PDB). Isso permite consolidação de bancos com isolamento e gerenciamento centralizado. Introduziu também o In-Memory Column Store para analytics acelerado. 


-- Criando um Pluggable Database (PDB) no modelo Multitenant 12c+
CREATE PLUGGABLE DATABASE pdb_treinamento
  ADMIN USER pdb_admin IDENTIFIED BY senha_admin
  ROLES = (DBA)
  FILE_NAME_CONVERT = ('/pdbseed/', '/pdb_treinamento/');

-- Abrindo o PDB
ALTER PLUGGABLE DATABASE pdb_treinamento OPEN;

Oracle 18c e 19c (2018-2019)

A versão 18c introduziu o conceito de Autonomous Database (banco autônomo que se auto-gerencia, auto-protege e auto-repara usando machine learning). A versão 19c é uma Long Term Support (LTS) — a mais estável e recomendada para ambientes de produção. Inclui JSON nativo com pesquisa por índices e Active Data Guard DML (escrita em standby).

Oracle 21c (2021)

Introduziu Blockchain Tables (tabelas imutáveis, resistentes a fraudes), SQL Macros (reutilização de lógica SQL parametrizada), Native JSON Binary (formato OSON para performance), e AutoML integrado. 


-- Exemplo de Blockchain Table no Oracle 21c+
CREATE BLOCKCHAIN TABLE auditoria_transacoes (
  id         NUMBER GENERATED BY DEFAULT AS IDENTITY,
  transacao  VARCHAR2(200),
  valor      NUMBER,
  data_hora  DATE DEFAULT SYSDATE,
  hash_prev  RAW(32)
)
NO DROP UNTIL 30 DAYS IDLE
NO DELETE LOCKED
HASHING USING "SHA2_512" VERSION "V1";

Oracle 23ai (2023)

Versão mais recente com foco em Inteligência Artificial embutida. Principais novidades: AI Vector Search (busca semântica com embeddings), JSON Relational Duality (acessar os mesmos dados como JSON ou relacional), Operational Property Graph (grafos nativos), e melhorias no Autonomous Database. 


-- Oracle 23ai: JSON Relational Duality
-- Mesmos dados disponíveis como relacional E como JSON
CREATE JSON RELATIONAL DUALITY VIEW v_cliente_duality AS
  SELECT JSON {
    'id'   : c.id,
    'nome' : c.nome,
    'pedidos' : [
      SELECT JSON {
        'id_pedido' : p.id,
        'data'      : p.data_pedido,
        'total'     : p.total
      }
      FROM pedidos p WITH INSERT UPDATE
      WHERE p.cliente_id = c.id
    ]
  }
  FROM clientes c WITH INSERT UPDATE DELETE;

Utilitários do Oracle

O Oracle oferece um conjunto de utilitários de linha de comando e interfaces gráficas para administração e interação com o banco de dados.

1. SQL*Plus

O SQL*Plus é o cliente de linha de comando tradicional do Oracle. Ele permite conectar-se ao banco, executar comandos SQL e PL/SQL, formatar saídas e executar scripts. Está disponível em todas as plataformas e é a ferramenta essencial para DBAs Oracle.

Comandos principais:


# Conectar ao Oracle usando SQL*Plus
sqlplus / as sysdba

# Conectar com usuário específico
sqlplus usuario/senha@host:porta/SID

# Executar um arquivo de script SQL
sqlplus usuario/senha @meuscript.sql

# Spool (salvar saída em arquivo)
SQL> spool resultado.txt
SQL> SELECT * FROM funcionarios;
SQL> spool off
          

-- Exemplos de comandos SQL*Plus úteis
-- Mostrar a estrutura de uma tabela
DESCRIBE funcionarios;

-- Listar todas as tabelas do usuário atual
SELECT table_name FROM user_tables;

-- Mostrar o plano de execução de uma consulta
EXPLAIN PLAN FOR
  SELECT * FROM funcionarios WHERE departamento_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- Configurar o formato de saída
SET LINESIZE 200
SET PAGESIZE 50
COLUMN nome FORMAT A30

2. Database Configuration Assistant (DBCA)

O DBCA é uma ferramenta gráfica (GUI) que auxilia na criação, configuração e exclusão de bancos de dados Oracle. É a forma mais amigável para criar um novo banco, especialmente para iniciantes.

Funcionalidades do DBCA:

  • Criar um banco de dados com assistente passo a passo
  • Escolher entre templates pré-configurados (Data Warehouse, General Purpose, Custom)
  • Configurar o SID, memória, character set, tablespaces
  • Gerenciar templates existentes (criar, editar, excluir)
  • Remover bancos de dados
  • Criar Pluggable Databases (PDBs) no modelo Multitenant

3. Administration Assistant for Windows

Utilitário específico para ambientes Windows que permite gerenciar serviços Oracle, usuários do sistema operacional, e configurações de segurança integradas com o Windows (Windows Authentication). É acessado via Painel de Controle ou Menu Iniciar do Oracle.

Principais tarefas:

  • Gerenciar serviços Windows do Oracle (iniciar, parar, configurar tipo de inicialização)
  • Adicionar usuários do Windows a grupos Oracle (ORA_DBA, ORA_OPER)
  • Configurar permissões de diretórios Oracle Home
  • Gerenciar autenticação integrada Windows (NTS Authentication)

4. Network Configuration Assistant (NetCA)

O NetCA configura os componentes de rede do Oracle, incluindo listeners e nomes de serviço (tnsnames.ora). Sem a configuração de rede, o cliente não consegue se conectar ao banco.

Conceitos importantes:

  • Listener: processo que escuta requisições de conexão na porta configurada (padrão 1521)
  • tnsnames.ora: arquivo de configuração do cliente que mapeia nomes de serviço (aliases) para endereços de conexão (host, porta, SID/service name)
  • Service Name: identificador lógico do serviço de banco de dados (não confundir com SID)

Exemplo de configuração — Conteúdo do tnsnames.ora: 


ORCL =
  (DESCRIPTION =
    (ADDRESS = (PROTOCOL = TCP)(HOST = localhost)(PORT = 1521))
    (CONNECT_DATA =
      (SERVER = DEDICATED)
      (SERVICE_NAME = orcl)
    )
  )

TREINAMENTO =
  (DESCRIPTION =
    (ADDRESS = (PROTOCOL = TCP)(HOST = 192.168.1.100)(PORT = 1521))
    (CONNECT_DATA =
      (SERVER = DEDICATED)
      (SERVICE_NAME = treinamento.local)
    )
  )

Comandos para gerenciar o Listener:


# Verificar status do listener
lsnrctl status

# Iniciar o listener
lsnrctl start

# Parar o listener
lsnrctl stop

# Recarregar configuração do listener
lsnrctl reload

# Listar serviços registrados no listener
lsnrctl services

Passo a passo: Instalando o Oracle DBMS (Oracle XE)

O Oracle Database XE (Express Edition) é a versão gratuita do Oracle, ideal para aprendizado e desenvolvimento. Possui limitações (até 12 GB de dados, 2 GB de RAM, 2 vCPUs), mas contém todas as funcionalidades essenciais.

Passo 1 — Baixar o Oracle Database XE

  • Acesse https://www.oracle.com/database/technologies/xe-downloads.html
  • Escolha a versão para Windows (arquivo .zip)
  • Extraia o conteúdo do arquivo .zip para uma pasta (ex: C:\oracle_xe_install)

Passo 2 — Executar o instalador

  • Navegue até a pasta extraída e execute setup.exe como Administrador
  • Na tela de boas-vindas, clique em Next
  • Aceite os termos de licença
  • Escolha o diretório de instalação (padrão: C:\app\oracle\product\21c)

Passo 3 — Configurar a senha do SYS e SYSTEM

  • Defina uma senha forte para os usuários SYS e SYSTEM
  • Anote essa senha — ela é essencial para administração do banco

Passo 4 — Concluir a instalação

  • O instalador criará o banco de dados inicial automaticamente
  • O SID padrão é XE
  • A porta padrão do listener é 1521
  • O Oracle Enterprise Manager Express estará disponível em https://localhost:5500/em

Passo 5 — Verificar a instalação com SQL*Plus


# Abrir o SQL*Plus e conectar como SYSDBA
sqlplus / as sysdba
          

-- Verificar a versão do Oracle
SELECT banner FROM v$version;

-- Verificar o nome e status do banco
SELECT name, open_mode, created FROM v$database;

-- Verificar o tamanho da SGA
SELECT component, current_size/1024/1024 AS tamanho_mb
FROM v$sga_dynamic_components;

-- Listar os tablespaces existentes
SELECT tablespace_name, status, contents
FROM dba_tablespaces;

Passo 6 — Criar um usuário para desenvolvimento


-- Criar usuário de desenvolvimento
CREATE USER dev_user IDENTIFIED BY dev_senha123;

-- Conceder privilégios básicos
GRANT CONNECT, RESOURCE, UNLIMITED TABLESPACE TO dev_user;

-- Verificar se foi criado
SELECT username, account_status FROM dba_users WHERE username = 'DEV_USER';

-- Conectar com o novo usuário
CONNECT dev_user/dev_senha123@XE

Passo a passo: Criando banco com DBCA

O Database Configuration Assistant é a maneira mais simples e visual de criar um banco de dados Oracle. Vamos criar um banco chamado "treinamento".

Passo 1 — Iniciar o DBCA

  • No menu Iniciar, procure por "Database Configuration Assistant"
  • Execute como Administrador

Passo 2 — Selecionar a operação

  • Na tela inicial, selecione "Create a database"
  • Clique em Next

Passo 3 — Escolher o modo de criação

  • Typical configuration: criação rápida com poucas opções (recomendado para iniciantes)
  • Advanced configuration: controle total sobre cada parâmetro (recomendado para produção)
  • Selecione Advanced configuration e clique em Next

Passo 4 — Template do banco

  • Data Warehouse: otimizado para consultas analíticas e grandes volumes
  • General Purpose or Transaction Processing: equilíbrio entre OLTP e analytics (recomendado)
  • Custom Database: começa com estrutura mínima
  • Selecione General Purpose or Transaction Processing

Passo 5 — Identificação do banco

  • Global Database Name: treinamento.local
  • SID: TREIN (identificador único do banco, máximo 12 caracteres)
  • Marque a opção "Create as Container database" se quiser usar Multitenant (recomendado 12c+)

Passo 6 — Configuração de armazenamento

  • Escolha o local dos data files (padrão via Oracle Flexible Architecture — OFA)
  • Ative o ARCHIVELOG mode se precisar de recuperação completa

Passo 7 — Configuração de memória

  • Automatic Memory Management (AMM): o Oracle gerencia SGA e PGA automaticamente
  • Defina o MEMORY_TARGET (ex: 1024 MB)

Passo 8 — Revisão e criação

  • Revise todas as configurações na tela de sumário
  • Clique em Finish
  • O DBCA gerará scripts SQL equivalentes — você pode salvá-los para referência
  • Clique em OK para iniciar a criação do banco

Passo a passo: Criando banco via arquivo de configuração (CREATE DATABASE)

O método manual usando CREATE DATABASE oferece controle absoluto sobre cada aspecto da criação do banco. É essencial para DBAs avançados.

Pré-requisitos

  • Instância Oracle iniciada em modo NOMOUNT
  • PFILE ou SPFILE configurado com os parâmetros mínimos
  • Diretórios para os data files criados no sistema operacional

Passo 1 — Criar o arquivo de parâmetros (PFILE)


# initTREIN.ora — arquivo de parâmetros mínimo
db_name='TREIN'
memory_target=1024M
control_files=('/u01/app/oracle/oradata/TREIN/control01.ctl',
               '/u01/app/oracle/oradata/TREIN/control02.ctl')
db_block_size=8192
open_cursors=300
processes=150

Passo 2 — Iniciar a instância em modo NOMOUNT


# Conectar como SYSDBA
sqlplus / as sysdba

# Definir o PFILE e iniciar em modo NOMOUNT
SQL> STARTUP NOMOUNT PFILE='/u01/app/oracle/product/19c/dbs/initTREIN.ora';

Passo 3 — Criar o banco de dados


CREATE DATABASE meubanco
  USER SYS IDENTIFIED BY senha_sys
  USER SYSTEM IDENTIFIED BY senha_system
  LOGFILE
    GROUP 1 ('/u01/app/oracle/oradata/redo01.log') SIZE 100M,
    GROUP 2 ('/u01/app/oracle/oradata/redo02.log') SIZE 100M,
    GROUP 3 ('/u01/app/oracle/oradata/redo03.log') SIZE 100M
  MAXLOGFILES 5
  MAXLOGMEMBERS 5
  MAXLOGHISTORY 1
  MAXDATAFILES 100
  CHARACTER SET AL32UTF8
  NATIONAL CHARACTER SET AL16UTF16
  DATAFILE '/u01/app/oracle/oradata/system01.dbf'
    SIZE 500M
    AUTOEXTEND ON NEXT 50M MAXSIZE UNLIMITED
  SYSAUX DATAFILE '/u01/app/oracle/oradata/sysaux01.dbf'
    SIZE 200M
    AUTOEXTEND ON NEXT 50M MAXSIZE UNLIMITED
  DEFAULT TABLESPACE users
    DATAFILE '/u01/app/oracle/oradata/users01.dbf'
    SIZE 100M
    AUTOEXTEND ON NEXT 10M MAXSIZE UNLIMITED
  DEFAULT TEMPORARY TABLESPACE temp
    TEMPFILE '/u01/app/oracle/oradata/temp01.dbf'
    SIZE 100M
    AUTOEXTEND ON NEXT 25M MAXSIZE UNLIMITED
  UNDO TABLESPACE undotbs
    DATAFILE '/u01/app/oracle/oradata/undotbs01.dbf'
    SIZE 100M
    AUTOEXTEND ON NEXT 25M MAXSIZE UNLIMITED;

Passo 4 — Executar scripts do dicionário de dados

Após o CREATE DATABASE, é necessário executar scripts que criam as views do dicionário de dados e pacotes PL/SQL essenciais: 


# Executar os scripts do dicionário de dados (no SQL*Plus)
SQL> @?/rdbms/admin/catalog.sql
SQL> @?/rdbms/admin/catproc.sql
SQL> @?/rdbms/admin/catblock.sql

# Opcional: criar o tablespace EXAMPLE
SQL> @?/demo/schema/mk_dir.sql
SQL> @?/demo/schema/mk_schema.sql

Passo 5 — Verificar o banco criado


-- Verificar informações do banco
SELECT name, open_mode, log_mode, created, db_unique_name
FROM v$database;

-- Listar tablespaces
SELECT tablespace_name, status, contents, bigfile
FROM dba_tablespaces;

-- Listar data files

FROM dba_data_files;

-- Listar redo log groups
SELECT group#, sequence#, bytes/1024/1024 AS tamanho_mb, members, status
FROM v$log;

Comparação Oracle vs PostgreSQL vs MySQL

Embora todos sejam SGBDs relacionais com suporte a SQL, existem diferenças significativas de arquitetura, licenciamento e recursos.

Característica Oracle PostgreSQL MySQL
Licença Proprietária (cara). XE é gratuito com limitações Open Source (PostgreSQL License) Open Source (GPL). Edição Enterprise paga
Arquitetura Instância + Banco de Dados. Multitenant CDB/PDB a partir do 12c Cluster de bancos (postmaster + backends) Single process + thread pool
Plataformas Windows, Linux, Unix (AIX, Solaris, HP-UX) Windows, Linux, Unix, macOS Windows, Linux, Unix, macOS
Linguagem Procedural PL/SQL (com packages, procedures, functions, triggers) PL/pgSQL, PL/Python, PL/Perl, PL/Tcl, PL/Java Stored Procedures (SQL/PSM). Sem packages
Auto Increment SEQUENCE + NEXTVAL (não tem SERIAL) SERIAL / BIGSERIAL (cria sequence automaticamente) AUTO_INCREMENT na coluna
Tabela temporária Global Temporary Table (dados privados por sessão) CREATE TEMP TABLE (dados privados por sessão) CREATE TEMPORARY TABLE
Limite de resultados ROWNUM ou FETCH FIRST (12c+) LIMIT / OFFSET LIMIT / OFFSET
Outer Join antigo Operador (+) Somente LEFT/RIGHT JOIN padrão Somente LEFT/RIGHT JOIN padrão
DUAL table Sim — SELECT SYSDATE FROM DUAL; Opcional — SELECT CURRENT_DATE; (sem FROM) Sim — SELECT NOW(); (sem FROM também funciona)
Rollback de transação UNDO tablespace gerenciado automaticamente MVCC (Multi-Version Concurrency Control) InnoDB: UNDO logs (MVCC)
Backup RMAN (Recovery Manager) + Data Pump pg_dump / pg_restore + WAL archiving mysqldump / mysqlpump + binlog
Extensibilidade Tipagem Oracle específica (objetos, varrays, nested tables) Extensões, tipos customizados, index methods Plugins, storage engines

1. Instalar o Oracle XE e conectar via SQL*Plus

Instale o Oracle Database XE em seu computador e conecte-se ao banco usando SQL*Plus. Após conectar, execute uma consulta simples para verificar a versão do Oracle.

Gabarito:

Comandos de conexão:


# Conexão como SYSDBA
sqlplus / as sysdba

3. Pesquisar e listar 5 processos background do Oracle e suas funções

Pesquise na documentação oficial ou em fontes confiáveis e descreva a função de 5 processos background do Oracle.

Gabarito:
  1. DBWn (Database Writer): escreve blocos de dados alterados (dirty buffers) do buffer cache para os data files em disco. É acionado quando o buffer cache atinge um limite ou ocorre checkpoint.
  2. LGWR (Log Writer): escreve as entradas do redo log buffer (registros de alterações) para os redo log files online. É acionado a cada COMMIT, quando o buffer está 1/3 cheio, ou a cada 3 segundos.
  3. CKPT (Checkpoint): sinaliza ao DBWn que todos os dirty buffers até um determinado SCN (System Change Number) devem ser gravados em disco. Atualiza os cabeçalhos dos data files e control files com informações de checkpoint.
  4. SMON (System Monitor): realiza recuperação de instância após crash (roll forward + roll back de transações não commitadas), libera segmentos temporários que não estão mais em uso, e coalesce (une) extents livres adjacentes em tablespaces gerenciados por dicionário.
  5. PMON (Process Monitor): limpa processos de servidor que falharam, libera locks mantidos por sessões mortas, faz rollback de transações não commitadas de processos que morreram, e restaura o estado da sessão.
