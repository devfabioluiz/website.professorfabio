2. Projeto Final: Sistema de Gestão Empresarial

O projeto final consiste em implementar um Sistema de Gestão Empresarial em dois SGBDs diferentes (Oracle e PostgreSQL), documentando as diferenças encontradas. Este projeto integra todos os conhecimentos adquiridos ao longo do curso.

2.1 Modelagem do Banco de Dados

O sistema gerencia empresas, seus departamentos, funcionários, projetos e a alocação de funcionários em projetos. O DER deve ter no mínimo 5 tabelas normalizadas (3FN): 


-- DER Conceitual (notação textual simplificada):

-- EMPRESAS (id, nome, cnpj, endereco, data_fundacao)
--   |
--   | 1:N
--   |
-- DEPARTAMENTOS (id, nome, empresa_id FK, gerente_id FK)
--   |
--   | 1:N
--   |
-- FUNCIONARIOS (id, nome, cpf, cargo, salario, data_admissao, departamento_id FK)
--   |
--   | N:M (via ALOCOES)
--   |
-- PROJETOS (id, nome, descricao, data_inicio, data_fim, status, departamento_id FK)
--
-- ALOCOES (id, funcionario_id FK, projeto_id FK, horas_semanais, data_inicio, data_fim)

2.2 Parte 1: Implementação no Oracle

Requisitos Oracle:

  1. Criar a estrutura completa com sequences e triggers de auditoria
  2. Criar um package com procedures para CRUD das 5 tabelas
  3. Criar views materializadas para relatórios gerenciais
  4. Configurar backup com RMAN

2.2.1 Criação das Tabelas, Sequences e Triggers de Auditoria (Oracle) 


-- Oracle: Criação das tabelas
CREATE TABLE empresas (
    id             NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    nome           VARCHAR2(200) NOT NULL,
    cnpj           VARCHAR2(18) UNIQUE NOT NULL,
    endereco       VARCHAR2(300),
    data_fundacao  DATE,
    criado_em      DATE DEFAULT SYSDATE,
    atualizado_em  DATE DEFAULT SYSDATE
);

CREATE TABLE departamentos (
    id          NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    nome        VARCHAR2(100) NOT NULL,
    empresa_id  NUMBER NOT NULL REFERENCES empresas(id),
    gerente_id  NUMBER,
    criado_em   DATE DEFAULT SYSDATE,
    atualizado_em DATE DEFAULT SYSDATE
);

CREATE TABLE funcionarios (
    id               NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    nome             VARCHAR2(200) NOT NULL,
    cpf              VARCHAR2(14) UNIQUE NOT NULL,
    cargo            VARCHAR2(100),
    salario          NUMBER(12,2) NOT NULL,
    data_admissao    DATE NOT NULL,
    departamento_id  NUMBER NOT NULL REFERENCES departamentos(id),
    criado_em        DATE DEFAULT SYSDATE,
    atualizado_em    DATE DEFAULT SYSDATE
);

CREATE TABLE projetos (
    id               NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    nome             VARCHAR2(200) NOT NULL,
    descricao        VARCHAR2(1000),
    data_inicio      DATE NOT NULL,
    data_fim         DATE,
    status           VARCHAR2(20) DEFAULT 'planejado',
    departamento_id  NUMBER NOT NULL REFERENCES departamentos(id),
    criado_em        DATE DEFAULT SYSDATE,
    atualizado_em    DATE DEFAULT SYSDATE
);

CREATE TABLE alocacoes (
    id              NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    funcionario_id  NUMBER NOT NULL REFERENCES funcionarios(id),
    projeto_id      NUMBER NOT NULL REFERENCES projetos(id),
    horas_semanais  NUMBER(4,1) NOT NULL,
    data_inicio     DATE NOT NULL,
    data_fim        DATE,
    criado_em       DATE DEFAULT SYSDATE,
    atualizado_em   DATE DEFAULT SYSDATE,
    CONSTRAINT uk_func_proj UNIQUE (funcionario_id, projeto_id, data_inicio)
);

-- Tabela de auditoria (Oracle)
CREATE TABLE auditoria_log (
    id           NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    tabela       VARCHAR2(50) NOT NULL,
    operacao     VARCHAR2(10) NOT NULL,
    registro_id  NUMBER,
    usuario      VARCHAR2(100) DEFAULT USER,
    data_hora    DATE DEFAULT SYSDATE,
    dados_antigos VARCHAR2(4000),
    dados_novos   VARCHAR2(4000)
);

-- Trigger de auditoria para EMPRESAS (Oracle)
CREATE OR REPLACE TRIGGER trg_audit_empresas
    AFTER INSERT OR UPDATE OR DELETE ON empresas
    FOR EACH ROW
DECLARE
    v_operacao VARCHAR2(10);
BEGIN
    IF INSERTING THEN
        v_operacao := 'INSERT';
    ELSIF UPDATING THEN
        v_operacao := 'UPDATE';
    ELSIF DELETING THEN
        v_operacao := 'DELETE';
    END IF;

    INSERT INTO auditoria_log (tabela, operacao, registro_id, dados_antigos, dados_novos)
    VALUES (
        'EMPRESAS',
        v_operacao,
        COALESCE(:NEW.id, :OLD.id),
        CASE WHEN :OLD.id IS NOT NULL THEN 'id=' || :OLD.id || ',nome=' || :OLD.nome END,
        CASE WHEN :NEW.id IS NOT NULL THEN 'id=' || :NEW.id || ',nome=' || :NEW.nome END
    );
END;

2.2.2 Package com Procedures CRUD (Oracle)


-- Oracle: Package specification
CREATE OR REPLACE PACKAGE pkg_gestao_empresarial AS

    -- CRUD Empresas
    PROCEDURE sp_inserir_empresa(
        p_nome          IN VARCHAR2,
        p_cnpj          IN VARCHAR2,
        p_endereco      IN VARCHAR2 DEFAULT NULL,
        p_data_fundacao IN DATE DEFAULT NULL,
        p_id            OUT NUMBER
    );

    PROCEDURE sp_atualizar_empresa(
        p_id       IN NUMBER,
        p_nome     IN VARCHAR2,
        p_endereco IN VARCHAR2,
        p_cnpj     IN VARCHAR2
    );

    PROCEDURE sp_excluir_empresa(p_id IN NUMBER);

    -- CRUD Funcionarios
    PROCEDURE sp_inserir_funcionario(
        p_nome            IN VARCHAR2,
        p_cpf             IN VARCHAR2,
        p_cargo           IN VARCHAR2,
        p_salario         IN NUMBER,
        p_data_admissao   IN DATE,
        p_departamento_id IN NUMBER,
        p_id              OUT NUMBER
    );

    -- Relatórios
    FUNCTION fn_total_folha_departamento(p_depto_id IN NUMBER)
        RETURN NUMBER;

    FUNCTION fn_funcionarios_por_projeto(p_projeto_id IN NUMBER)
        RETURN SYS_REFCURSOR;

END pkg_gestao_empresarial;
/

-- Oracle: Package body
CREATE OR REPLACE PACKAGE BODY pkg_gestao_empresarial AS

    PROCEDURE sp_inserir_empresa(
        p_nome          IN VARCHAR2,
        p_cnpj          IN VARCHAR2,
        p_endereco      IN VARCHAR2 DEFAULT NULL,
        p_data_fundacao IN DATE DEFAULT NULL,
        p_id            OUT NUMBER
    ) AS
    BEGIN
        INSERT INTO empresas (nome, cnpj, endereco, data_fundacao)
        VALUES (p_nome, p_cnpj, p_endereco, p_data_fundacao)
        RETURNING id INTO p_id;
        COMMIT;
    EXCEPTION
        WHEN DUP_VAL_ON_INDEX THEN
            RAISE_APPLICATION_ERROR(-20001, 'CNPJ já cadastrado');
        WHEN OTHERS THEN
            ROLLBACK;
            RAISE;
    END;

    PROCEDURE sp_atualizar_empresa(
        p_id       IN NUMBER,
        p_nome     IN VARCHAR2,
        p_endereco IN VARCHAR2,
        p_cnpj     IN VARCHAR2
    ) AS
    BEGIN
        UPDATE empresas
        SET nome = p_nome,
            endereco = p_endereco,
            cnpj = p_cnpj,
            atualizado_em = SYSDATE
        WHERE id = p_id;

        IF SQL%ROWCOUNT = 0 THEN
            RAISE_APPLICATION_ERROR(-20002, 'Empresa não encontrada');
        END IF;
        COMMIT;
    END;

    PROCEDURE sp_excluir_empresa(p_id IN NUMBER) AS
    BEGIN
        DELETE FROM empresas WHERE id = p_id;
        IF SQL%ROWCOUNT = 0 THEN
            RAISE_APPLICATION_ERROR(-20003, 'Empresa não encontrada');
        END IF;
        COMMIT;
    END;

    PROCEDURE sp_inserir_funcionario(
        p_nome            IN VARCHAR2,
        p_cpf             IN VARCHAR2,
        p_cargo           IN VARCHAR2,
        p_salario         IN NUMBER,
        p_data_admissao   IN DATE,
        p_departamento_id IN NUMBER,
        p_id              OUT NUMBER
    ) AS
    BEGIN
        INSERT INTO funcionarios (nome, cpf, cargo, salario, data_admissao, departamento_id)
        VALUES (p_nome, p_cpf, p_cargo, p_salario, p_data_admissao, p_departamento_id)
        RETURNING id INTO p_id;
        COMMIT;
    END;

    FUNCTION fn_total_folha_departamento(p_depto_id IN NUMBER)
        RETURN NUMBER
    AS
        v_total NUMBER;
    BEGIN
        SELECT COALESCE(SUM(salario), 0) INTO v_total
        FROM funcionarios
        WHERE departamento_id = p_depto_id;
        RETURN v_total;
    END;

    FUNCTION fn_funcionarios_por_projeto(p_projeto_id IN NUMBER)
        RETURN SYS_REFCURSOR
    AS
        v_cursor SYS_REFCURSOR;
    BEGIN
        OPEN v_cursor FOR
            SELECT f.nome, f.cargo, a.horas_semanais, a.data_inicio
            FROM funcionarios f
            JOIN alocacoes a ON f.id = a.funcionario_id
            WHERE a.projeto_id = p_projeto_id;
        RETURN v_cursor;
    END;

END pkg_gestao_empresarial;
/

2.2.3 Views Materializadas para Relatórios (Oracle)


-- Oracle: View materializada - Resumo de alocações por projeto
CREATE MATERIALIZED VIEW mv_resumo_projetos
BUILD IMMEDIATE
REFRESH COMPLETE ON DEMAND
AS
SELECT
    p.id          AS projeto_id,
    p.nome        AS projeto_nome,
    p.status,
    COUNT(a.funcionario_id) AS total_alocados,
    SUM(a.horas_semanais)   AS total_horas_semanais,
    AVG(a.horas_semanais)   AS media_horas_alocado
FROM projetos p
LEFT JOIN alocacoes a ON p.id = a.projeto_id
GROUP BY p.id, p.nome, p.status;

-- Oracle: View materializada - Folha salarial por departamento/empresa
CREATE MATERIALIZED VIEW mv_folha_empresarial
BUILD IMMEDIATE
REFRESH COMPLETE ON DEMAND
AS
SELECT
    e.nome    AS empresa,
    d.nome    AS departamento,
    COUNT(f.id)  AS total_funcionarios,
    SUM(f.salario) AS total_folha,
    AVG(f.salario) AS salario_medio,
    MIN(f.salario) AS menor_salario,
    MAX(f.salario) AS maior_salario
FROM empresas e
JOIN departamentos d ON e.id = d.empresa_id
LEFT JOIN funcionarios f ON d.id = f.departamento_id
GROUP BY e.nome, d.nome
ORDER BY e.nome, d.nome;

-- Oracle: View materializada com log para FAST REFRESH
CREATE MATERIALIZED VIEW LOG ON funcionarios
    WITH ROWID, SEQUENCE (id, nome, salario, departamento_id)
    INCLUDING NEW VALUES;

-- Forçar refresh manual
EXEC DBMS_MVIEW.REFRESH('MV_FOLHA_EMPRESARIAL', 'C');
EXEC DBMS_MVIEW.REFRESH('MV_RESUMO_PROJETOS', 'C');

2.2.4 Configuração de Backup com RMAN (Oracle)


# Oracle: Configurar banco em modo ARCHIVELOG
sqlplus / as sysdba
SQL> SHUTDOWN IMMEDIATE;
SQL> STARTUP MOUNT;
SQL> ALTER DATABASE ARCHIVELOG;
SQL> ALTER DATABASE OPEN;

# Verificar modo de archive
SQL> SELECT log_mode FROM v$database;

# Configurar RMAN e criar backup completo
rman target /

RMAN> CONFIGURE RETENTION POLICY TO RECOVERY WINDOW OF 7 DAYS;
RMAN> CONFIGURE BACKUP OPTIMIZATION ON;
RMAN> CONFIGURE DEFAULT DEVICE TYPE TO DISK;
RMAN> CONFIGURE DEVICE TYPE DISK PARALLELISM 2;
RMAN> CONFIGURE CHANNEL DEVICE TYPE DISK FORMAT '/backup/%d_%T_%s_%p.bkp';

# Backup completo do banco
RMAN> BACKUP DATABASE PLUS ARCHIVELOG DELETE INPUT;

# Backup incremental nível 1 (diário)
RMAN> BACKUP INCREMENTAL LEVEL 1 DATABASE;

# Listar backups
RMAN> LIST BACKUP SUMMARY;

# Restaurar banco completo
RMAN> SHUTDOWN IMMEDIATE;
RMAN> STARTUP MOUNT;
RMAN> RESTORE DATABASE;
RMAN> RECOVER DATABASE;
RMAN> ALTER DATABASE OPEN;

2.3.2 Procedures e Functions Equivalentes (PostgreSQL)


-- PostgreSQL: Procedures e functions (equivalente ao package Oracle)
-- Como PostgreSQL não tem packages, usamos schema e naming convention

-- Procedure: Inserir empresa
CREATE OR REPLACE PROCEDURE sp_inserir_empresa(
    IN  p_nome          VARCHAR,
    IN  p_cnpj          VARCHAR,
    IN  p_endereco      VARCHAR DEFAULT NULL,
    IN  p_data_fundacao DATE DEFAULT NULL,
    OUT p_id            INT
)
LANGUAGE plpgsql
AS $$
BEGIN
    INSERT INTO empresas (nome, cnpj, endereco, data_fundacao)
    VALUES (p_nome, p_cnpj, p_endereco, p_data_fundacao)
    RETURNING id INTO p_id;
    COMMIT;
EXCEPTION
    WHEN unique_violation THEN
        RAISE EXCEPTION 'CNPJ já cadastrado: %', p_cnpj;
    WHEN OTHERS THEN
        ROLLBACK;
        RAISE;
END;
$$;

-- Procedure: Atualizar empresa
CREATE OR REPLACE PROCEDURE sp_atualizar_empresa(
    p_id       INT,
    p_nome     VARCHAR,
    p_endereco VARCHAR,
    p_cnpj     VARCHAR
)
LANGUAGE plpgsql
AS $$
BEGIN
    UPDATE empresas
    SET nome = p_nome,
        endereco = p_endereco,
        cnpj = p_cnpj,
        atualizado_em = CURRENT_TIMESTAMP
    WHERE id = p_id;

    IF NOT FOUND THEN
        RAISE EXCEPTION 'Empresa ID % não encontrada', p_id;
    END IF;
    COMMIT;
END;
$$;

-- Procedure: Excluir empresa
CREATE OR REPLACE PROCEDURE sp_excluir_empresa(p_id INT)
LANGUAGE plpgsql
AS $$
BEGIN
    DELETE FROM empresas WHERE id = p_id;
    IF NOT FOUND THEN
        RAISE EXCEPTION 'Empresa ID % não encontrada', p_id;
    END IF;
    COMMIT;
END;
$$;

-- Procedure: Inserir funcionário
CREATE OR REPLACE PROCEDURE sp_inserir_funcionario(
    IN  p_nome            VARCHAR,
    IN  p_cpf             VARCHAR,
    IN  p_cargo           VARCHAR,
    IN  p_salario         NUMERIC,
    IN  p_data_admissao   DATE,
    IN  p_departamento_id INT,
    OUT p_id              INT
)
LANGUAGE plpgsql
AS $$
BEGIN
    INSERT INTO funcionarios (nome, cpf, cargo, salario, data_admissao, departamento_id)
    VALUES (p_nome, p_cpf, p_cargo, p_salario, p_data_admissao, p_departamento_id)
    RETURNING id INTO p_id;
    COMMIT;
END;
$$;

-- Function: Total folha do departamento
CREATE OR REPLACE FUNCTION fn_total_folha_departamento(p_depto_id INT)
RETURNS NUMERIC
LANGUAGE plpgsql
STABLE
AS $$
DECLARE
    v_total NUMERIC;
BEGIN
    SELECT COALESCE(SUM(salario), 0) INTO v_total
    FROM funcionarios
    WHERE departamento_id = p_depto_id;
    RETURN v_total;
END;
$$;

-- Function: Funcionários por projeto (RETURNS TABLE)
CREATE OR REPLACE FUNCTION fn_funcionarios_por_projeto(p_projeto_id INT)
RETURNS TABLE(
    nome           VARCHAR,
    cargo          VARCHAR,
    horas_semanais NUMERIC,
    data_inicio    DATE
)
LANGUAGE plpgsql
STABLE
AS $$
BEGIN

$$;

2.3.3 Triggers de Validação (PostgreSQL)


-- PostgreSQL: Trigger de validação - impedir alocação com horas > 44h semanais
CREATE OR REPLACE FUNCTION trg_valida_horas_alocacao()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
DECLARE
    v_total_horas NUMERIC;
BEGIN
    -- Soma todas as horas do funcionário em todas as alocações ativas
    SELECT COALESCE(SUM(horas_semanais), 0)
    INTO v_total_horas
    FROM alocacoes
    WHERE funcionario_id = NEW.funcionario_id
      AND (data_fim IS NULL OR data_fim >= CURRENT_DATE);

    IF (v_total_horas + NEW.horas_semanais) > 44 THEN
        RAISE EXCEPTION 'Funcionário ID % excederia 44h semanais (total: %, nova: %)',
            NEW.funcionario_id, v_total_horas, NEW.horas_semanais;
    END IF;

    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_alocacao_horas
    BEFORE INSERT OR UPDATE ON alocacoes
    FOR EACH ROW
    EXECUTE FUNCTION trg_valida_horas_alocacao();

-- Trigger: atualizar atualizado_em automaticamente
CREATE OR REPLACE FUNCTION trg_atualiza_timestamp()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    NEW.atualizado_em := CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_empresas_timestamp
    BEFORE UPDATE ON empresas
    FOR EACH ROW
    EXECUTE FUNCTION trg_atualiza_timestamp();

CREATE TRIGGER trg_departamentos_timestamp
    BEFORE UPDATE ON departamentos
    FOR EACH ROW
    EXECUTE FUNCTION trg_atualiza_timestamp();

CREATE TRIGGER trg_funcionarios_timestamp
    BEFORE UPDATE ON funcionarios
    FOR EACH ROW
    EXECUTE FUNCTION trg_atualiza_timestamp();

-- Trigger: validar data de admissão não pode ser futura
CREATE OR REPLACE FUNCTION trg_valida_data_admissao()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    IF NEW.data_admissao > CURRENT_DATE THEN
        RAISE EXCEPTION 'Data de admissão (%) não pode ser futura', NEW.data_admissao;
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_func_data_admissao
    BEFORE INSERT OR UPDATE ON funcionarios
    FOR EACH ROW
    EXECUTE FUNCTION trg_valida_data_admissao();

-- Trigger: auditoria de alterações (nível STATEMENT)
CREATE TABLE auditoria_log (
    id           SERIAL PRIMARY KEY,
    tabela       VARCHAR(50) NOT NULL,
    operacao     VARCHAR(10) NOT NULL,
    registro_id  INT,
    usuario      VARCHAR(100) DEFAULT CURRENT_USER,
    data_hora    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    dados_antigos TEXT,
    dados_novos   TEXT
);

CREATE OR REPLACE FUNCTION trg_audit_empresas()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    INSERT INTO auditoria_log (tabela, operacao, registro_id, dados_antigos, dados_novos)
    VALUES (
        'EMPRESAS',
        TG_OP,
        COALESCE(NEW.id, OLD.id),
        CASE WHEN OLD.id IS NOT NULL THEN 'id=' || OLD.id || ',nome=' || OLD.nome END,
        CASE WHEN NEW.id IS NOT NULL THEN 'id=' || NEW.id || ',nome=' || NEW.nome END
    );
    RETURN COALESCE(NEW, OLD);
END;
$$;

CREATE TRIGGER trg_audit_empresas
    AFTER INSERT OR UPDATE OR DELETE ON empresas
    FOR EACH ROW
    EXECUTE FUNCTION trg_audit_empresas();

2.3.4 Rules para Logging (PostgreSQL)


-- PostgreSQL: Rule system para logging de operações sensíveis

-- Tabela de log
CREATE TABLE log_operacoes (
    id        SERIAL PRIMARY KEY,
    operacao  VARCHAR(20) NOT NULL,
    tabela    VARCHAR(50) NOT NULL,
    dados     TEXT,
    data_hora TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Rule: log de INSERT em funcionários (DO ALSO - executa ação adicional)
CREATE OR REPLACE RULE log_insert_funcionario AS
    ON INSERT TO funcionarios
    DO ALSO
    INSERT INTO log_operacoes (operacao, tabela, dados)
    VALUES ('INSERT', 'funcionarios',
            'nome=' || NEW.nome || ',cargo=' || NEW.cargo || ',salario=' || NEW.salario);

-- Rule: log de UPDATE em salário de funcionários (condicional)
CREATE OR REPLACE RULE log_update_salario AS
    ON UPDATE TO funcionarios
    WHERE OLD.salario IS DISTINCT FROM NEW.salario
    DO ALSO
    INSERT INTO log_operacoes (operacao, tabela, dados)
    VALUES ('UPDATE_SALARIO', 'funcionarios',
            'id=' || NEW.id || ',de=' || OLD.salario || ',para=' || NEW.salario);

-- Rule: log de DELETE em projetos (DO ALSO)
CREATE OR REPLACE RULE log_delete_projeto AS
    ON DELETE TO projetos
    DO ALSO
    INSERT INTO log_operacoes (operacao, tabela, dados)
    VALUES ('DELETE', 'projetos', 'id=' || OLD.id || ',nome=' || OLD.nome);

-- Rule: impedir INSERT em funcionários sem departamento (DO INSTEAD)
CREATE OR REPLACE RULE bloqueia_funcionario_sem_depto AS
    ON INSERT TO funcionarios
    WHERE NEW.departamento_id IS NULL
    DO INSTEAD NOTHING;

2.3.5 Views Materializadas para Relatórios (PostgreSQL)


-- PostgreSQL: View materializada - Resumo de projetos
CREATE MATERIALIZED VIEW mv_resumo_projetos AS
SELECT
    p.id                        AS projeto_id,
    p.nome                      AS projeto_nome,
    p.status,
    COUNT(a.funcionario_id)     AS total_alocados,
    COALESCE(SUM(a.horas_semanais), 0) AS total_horas_semanais,
    ROUND(AVG(a.horas_semanais), 1)    AS media_horas_alocado
FROM projetos p
LEFT JOIN alocacoes a ON p.id = a.projeto_id
GROUP BY p.id, p.nome, p.status;

-- PostgreSQL: View materializada - Folha empresarial
CREATE MATERIALIZED VIEW mv_folha_empresarial AS
SELECT
    e.nome    AS empresa,
    d.nome    AS departamento,
    COUNT(f.id)  AS total_funcionarios,
    COALESCE(SUM(f.salario), 0) AS total_folha,
    ROUND(AVG(f.salario), 2)    AS salario_medio,
    MIN(f.salario) AS menor_salario,
    MAX(f.salario) AS maior_salario
FROM empresas e
JOIN departamentos d ON e.id = d.empresa_id
LEFT JOIN funcionarios f ON d.id = f.departamento_id
GROUP BY e.nome, d.nome
ORDER BY e.nome, d.nome;

-- Refresh das views materializadas
REFRESH MATERIALIZED VIEW mv_resumo_projetos;
REFRESH MATERIALIZED VIEW mv_folha_empresarial;

-- Criar índice na view materializada
CREATE UNIQUE INDEX idx_mv_projetos_id ON mv_resumo_projetos (projeto_id);
CREATE INDEX idx_mv_folha_empresa ON mv_folha_empresarial (empresa);

2.4 Documentação: Diferenças Oracle vs PostgreSQL

Parte essencial do projeto final é documentar as diferenças encontradas entre as duas implementações. Abaixo, um modelo de documentação comparativa:

Aspecto Oracle PostgreSQL Impacto
Auto-incremento IDENTITY columns (12c+) ou SEQUENCE + trigger SERIAL / BIGSERIAL (cria sequence implicitamente) Sintaxe mais simples no PostgreSQL
Tipagem de strings VARCHAR2 (recomendado pela Oracle) VARCHAR (padrão SQL) VARCHAR2 não existe no PostgreSQL
Packages CREATE PACKAGE / PACKAGE BODY Não possui. Usa-se schema + naming convention Organização de código diferente
Procedures com COMMIT Permitido naturalmente Permitido apenas em PROCEDURE (PG 11+), não em FUNCTION Funções PostgreSQL são "puras" (sem efeitos colaterais transacionais)
RETURNS tipo RETURN tipo (after IS, before BEGIN) RETURNS tipo (after parameters, before LANGUAGE) Posição sintática diferente
Delimitador de corpo Não usa delimitador especial Usa $$ (dollar quoting) Código PostgreSQL mais verboso visualmente
Cursores de retorno SYS_REFCURSOR RETURNS TABLE com RETURN QUERY ou RETURNS SETOF refcursor Mais simples e declarativo no PostgreSQL
Triggers Sintaxe: CREATE OR REPLACE TRIGGER ... FOR EACH ROW. OLD/NEW sem prefixo. Sintaxe: CREATE TRIGGER ... FOR EACH ROW EXECUTE FUNCTION. OLD/NEW como variáveis. PostgreSQL separa trigger da função; Oracle une em um só objeto
Views materializadas REFRESH FAST/COMPLETE/FORCE, materialized view logs REFRESH MATERIALIZED VIEW [CONCURRENTLY] Oracle tem mais opções de refresh incremental
Rules Não possui rule system CREATE RULE com DO INSTEAD / DO ALSO Recurso exclusivo do PostgreSQL
Backup RMAN (binário, incremental, block-level) pg_dump / pg_restore (lógico, SQL) + WAL archiving (físico) Filosofias diferentes de backup
Exceções Nomeadas no standard package ou RAISE_APPLICATION_ERROR RAISE EXCEPTION com mensagem customizada; EXCEPTION WHEN ... Sintaxe mais flexível no PostgreSQL
Sobrecarga de funções Apenas dentro de packages Nativa no schema (mesmo nome, parâmetros diferentes) Mais natural no PostgreSQL

Responda as 12 questões abaixo com base no conteúdo estudado ao longo do curso (Oracle e PostgreSQL). Cada questão possui apenas uma alternativa correta.

Questão 1 — Arquitetura Oracle

Na arquitetura Oracle, a SGA (System Global Area) é uma região de memória compartilhada que contém diversos componentes. Qual dos seguintes NÃO faz parte da SGA?

  1. Database Buffer Cache
  2. Shared Pool
  3. Sort Area
  4. Redo Log Buffer

Questão 2 — Tipos de Dados Oracle vs PostgreSQL

Qual das seguintes afirmações sobre tipos de dados é correta?

  1. Oracle utiliza VARCHAR e PostgreSQL utiliza VARCHAR2 como tipo de caractere variável
  2. Tanto Oracle quanto PostgreSQL suportam o tipo SERIAL para colunas auto-incremento
  3. O tipo NUMBER no Oracle aceita precisão e escala (NUMBER(10,2)); no PostgreSQL o equivalente é NUMERIC(10,2)
  4. PostgreSQL não suporta o tipo DATE; apenas TIMESTAMP

Questão 3 — Cursores PL/SQL

No Oracle PL/SQL, após executar um FETCH em um cursor explícito, qual atributo indica se a última operação de fetch retornou uma linha?

  1. %ISOPEN
  2. %ROWCOUNT
  3. %NOTFOUND
  4. %FOUND

Questão 4 — Sintaxe Oracle vs PostgreSQL

Qual comando SQL produz resultados equivalentes em ambos os bancos?

  1. SELECT * FROM tabela WHERE ROWNUM <= 10;
  2. SELECT * FROM tabela LIMIT 10;
  3. SELECT SYSDATE FROM DUAL;
  4. SELECT * FROM tabela FETCH FIRST 10 ROWS ONLY;

Questão 5 — Rules vs Triggers

Sobre o sistema de regras (Rule System) do PostgreSQL, é correto afirmar:

  1. Rules são equivalentes a triggers AFTER em todos os aspectos
  2. Uma rule com DO INSTEAD pode reescrever completamente a operação original
  3. Rules podem conter lógica condicional complexa com IF/ELSE como em triggers
  4. O Oracle também possui um sistema de rules semelhante ao PostgreSQL

Questão 6 — Views Materializadas

Qual a principal diferença entre views materializadas no Oracle e no PostgreSQL?

  1. Oracle suporta FAST REFRESH com materialized view logs; PostgreSQL não possui equivalente
  2. PostgreSQL não suporta views materializadas
  3. No Oracle, views materializadas são sempre atualizadas automaticamente; no PostgreSQL, exigem REFRESH manual
  4. PostgreSQL suporta REFRESH CONCURRENTLY, que permite leitura durante o refresh sem bloquear a view — Oracle não oferece esse modo

Questão 7 — Backup

Qual ferramenta é utilizada para backup e restore no PostgreSQL?

  1. RMAN
  2. Data Pump
  3. pg_dump / pg_restore
  4. mysqldump

Questão 8 — Estrutura de Bloco PL/pgSQL

Qual é a estrutura correta de um bloco PL/pgSQL no PostgreSQL?

  1. DECLARE ... BEGIN ... EXCEPTION ... END;
  2. BEGIN ... DECLARE ... EXCEPTION ... END;
  3. DECLARE ... EXCEPTION ... BEGIN ... END;
  4. CREATE ... DECLARE ... BEGIN ... END;

Questão 9 — Triggers: OLD e NEW

Qual a diferença fundamental entre OLD/NEW em triggers Oracle vs PostgreSQL?

  1. No Oracle, OLD e NEW são prefixados com dois-pontos (:OLD, :NEW); no PostgreSQL, são usados sem prefixo
  2. O PostgreSQL não suporta OLD e NEW em triggers
  3. No Oracle, OLD/NEW só existem em triggers BEFORE; no PostgreSQL, só em triggers AFTER
  4. Ambos usam a mesma sintaxe: OLD e NEW sem qualquer prefixo

Questão 10 — Sobrecarga de Funções

Sobre sobrecarga de funções no PostgreSQL, é correto afirmar:

  1. PostgreSQL não suporta sobrecarga; cada função deve ter nome único
  2. PostgreSQL suporta sobrecarga apenas para funções em linguagens diferentes (ex: uma SQL e uma PL/pgSQL)
  3. PostgreSQL suporta sobrecarga de funções com mesmo nome, desde que tenham número ou tipo de parâmetros diferentes
  4. A sobrecarga no PostgreSQL só funciona com funções IMMUTABLE

Questão 11 — Controle Transacional

Sobre controle de transações em procedures no PostgreSQL, qual afirmação é verdadeira?

  1. Funções PostgreSQL podem executar COMMIT e ROLLBACK livremente
  2. Apenas procedures (CREATE PROCEDURE) podem executar COMMIT e ROLLBACK; funções não podem
  3. Nem procedures nem funções podem controlar transações no PostgreSQL
  4. O controle transacional no PostgreSQL é idêntico ao Oracle em todos os aspectos

Questão 12 — Modelos de Segurança

Comparando os modelos de segurança do Oracle e PostgreSQL, qual diferença é correta?

  1. Oracle usa SECURITY DEFINER; PostgreSQL não possui esse conceito
  2. Ambos suportam roles (funções/papéis) para agrupar privilégios, mas apenas Oracle possui o conceito de privilégios de sistema (SYSDBA, SYSOPER)
  3. PostgreSQL suporta row-level security (RLS); Oracle não
  4. Apenas Oracle suporta GRANT e REVOKE; PostgreSQL usa um sistema diferente

4. Gabarito do Exame

Gabarito — Questão 1

Resposta: C — Sort Area

A Sort Area faz parte da PGA (Program Global Area), que é memória privada de cada processo servidor, não da SGA (compartilhada). A SGA contém: Database Buffer Cache, Redo Log Buffer, Shared Pool (Library Cache + Data Dictionary Cache), Large Pool, Java Pool e Streams Pool.

Gabarito — Questão 2

Resposta: C

NUMBER(10,2) no Oracle e NUMERIC(10,2) no PostgreSQL são equivalentes. A alternativa (a) está invertida (Oracle usa VARCHAR2, PostgreSQL usa VARCHAR). A alternativa (b) é falsa porque SERIAL é exclusivo do PostgreSQL (Oracle usa IDENTITY ou SEQUENCE). A alternativa (d) é falsa pois PostgreSQL suporta DATE plenamente.

Gabarito — Questão 3

Resposta: D — %FOUND

%FOUND retorna TRUE se a última operação FETCH obteve uma linha com sucesso. %NOTFOUND retorna TRUE se o FETCH não encontrou mais linhas. %ROWCOUNT retorna o número de linhas fetchadas até o momento. %ISOPEN indica se o cursor está aberto.

Gabarito — Questão 4

Resposta: D

FETCH FIRST 10 ROWS ONLY é SQL padrão (SQL:2008) e funciona em ambos os bancos (Oracle 12c+ e PostgreSQL). ROWNUM é exclusivo Oracle. LIMIT é exclusivo PostgreSQL. SYSDATE FROM DUAL é exclusivo Oracle (PostgreSQL usa CURRENT_DATE sem FROM).

Gabarito — Questão 5

Resposta: B

Uma rule com DO INSTEAD reescreve a operação original (ex: um INSERT pode ser convertido em UPDATE). Rules são diferentes de triggers: rules operam no nível de reescrita de query (antes da execução), enquanto triggers operam durante a execução. Rules não contêm lógica procedimental como IF/ELSE — para isso usa-se triggers. Oracle não possui rule system.

Gabarito — Questão 6

Resposta: D

O PostgreSQL suporta REFRESH MATERIALIZED VIEW CONCURRENTLY, que permite consultas simultâneas durante o refresh (requer índice único na view). O Oracle também oferece refresh online em certas condições, mas a diferença arquitetural mais marcante é que o Oracle tem opções FAST/FORCE/COMPLETE com materialized view logs, enquanto o PostgreSQL tem CONCURRENTLY (que evita bloqueios). A alternativa (a) está parcialmente correta em dizer que o PostgreSQL não tem FAST REFRESH com logs — isso é verdade, mas a alternativa (d) descreve melhor a diferença prática.

Esclarecimento: Tanto a alternativa (a) quanto a (d) contêm verdades. A resposta oficial é D por destacar uma funcionalidade diferenciada do PostgreSQL (CONCURRENTLY) que o Oracle não tem nessa forma exata. O Oracle tem refresh online via REFRESH ON COMMIT e DBMS_MVIEW.REFRESH com opção de atomicidade.

Gabarito — Questão 7

Resposta: C — pg_dump / pg_restore

pg_dump gera backup lógico (arquivo SQL ou formato customizado) e pg_restore restaura. RMAN e Data Pump são ferramentas Oracle. mysqldump é do MySQL.

Gabarito — Questão 8

Resposta: A

A estrutura correta de um bloco PL/pgSQL é [DECLARE ...] BEGIN ... [EXCEPTION ...] END;. A seção DECLARE é opcional (para declarar variáveis), BEGIN é obrigatório (código executável), e EXCEPTION é opcional (tratamento de erros).

Gabarito — Questão 9

Resposta: A

No Oracle PL/SQL, os valores OLD e NEW em triggers de linha são referenciados com dois-pontos: :OLD.coluna e :NEW.coluna. No PostgreSQL PL/pgSQL, são referenciados sem prefixo: OLD.coluna e NEW.coluna (variáveis implícitas do tipo RECORD).

Gabarito — Questão 10

Resposta: C

PostgreSQL suporta sobrecarga de funções nativamente: múltiplas funções com o mesmo nome podem coexistir no mesmo schema, desde que difiram no número de parâmetros ou nos tipos de parâmetros. O PostgreSQL resolve qual função chamar com base na melhor correspondência de tipos dos argumentos. Oracle também suporta, mas apenas dentro de packages.

Gabarito — Questão 11

Resposta: B

No PostgreSQL, apenas procedures (CREATE PROCEDURE, introduzido no PG 11) podem conter COMMIT e ROLLBACK. Funções (CREATE FUNCTION) não podem executar controle transacional — qualquer tentativa resulta em erro. Esta é uma diferença importante em relação ao Oracle, onde tanto procedures quanto funções podem controlar transações.

Gabarito — Questão 12

Resposta: B

Tanto Oracle quanto PostgreSQL suportam roles para agrupar privilégios, e ambos usam GRANT/REVOKE. A principal diferença é que o Oracle possui privilégios de sistema administrativos (SYSDBA, SYSOPER, SYSBACKUP) que não existem no PostgreSQL. Ambos suportam row-level security (Oracle via VPD/FGAC, PostgreSQL via RLS policies). Ambos suportam SECURITY DEFINER (PostgreSQL) / DEFINER rights (Oracle).
