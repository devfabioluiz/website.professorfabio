O que são Regras (Rules)?

O Rule System (sistema de regras) do PostgreSQL é um mecanismo de reescrita de queries (query rewriting). Quando você define uma regra em uma tabela ou view, o PostgreSQL intercepta comandos SQL (SELECT, INSERT, UPDATE, DELETE) e os reescreve — transformando-os em outros comandos — antes mesmo da execução.

Como as Regras funcionam

O processo funciona em três etapas:

  1. Parser: o comando SQL original é analisado e transformado em uma árvore de consulta (query tree)
  2. Rewriter: o sistema de regras inspeciona a query tree e aplica as regras definidas, modificando a árvore conforme as ações especificadas (DO INSTEAD ou DO ALSO)
  3. Planner/Executor: a query tree modificada é planejada e executada normalmente
Diferença fundamental entre Rules e Triggers: Regras atuam na reescrita da query (antes do planejamento), enquanto Triggers atuam durante a execução da query (em cada linha ou statement). Isso tem implicações profundas de performance e comportamento.

Sintaxe CREATE RULE


CREATE [ OR REPLACE ] RULE nome_regra
AS ON evento
   [ TO tabela_ou_view ]
   [ WHERE condicao ]
DO [ ALSO | INSTEAD ]
   { NOTHING | comando; ( comandos... ) }

Componentes da sintaxe:

  • evento: SELECT, INSERT, UPDATE ou DELETE (um por regra)
  • tabela_ou_view: a tabela ou view alvo da regra
  • WHERE (opcional): condição para a regra ser aplicada. NEW e OLD podem ser referenciados
  • DO ALSO: executa o comando da regra além do comando original (ação adicional)
  • DO INSTEAD: executa o comando da regra no lugar do comando original (substituição)
  • NOTHING: não executa nada (usado com DO INSTEAD para suprimir a ação)

Criando Regras — Exemplos Práticos

Exemplo 1: DO ALSO — Log de auditoria em INSERT

Cenário: queremos registrar cada INSERT na tabela clientes em uma tabela de log automaticamente, sem alterar a aplicação. 


-- Criar a tabela de clientes
CREATE TABLE clientes (
    id SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    email VARCHAR(100),
    status VARCHAR(20) DEFAULT 'ativo',
    criado_em TIMESTAMP DEFAULT now()
);

-- Criar a tabela de log (auditoria)
CREATE TABLE log_auditoria (
    id SERIAL PRIMARY KEY,
    operacao VARCHAR(20),
    tabela VARCHAR(100),
    registro_id INTEGER,
    data_operacao TIMESTAMP DEFAULT now(),
    usuario_sessao VARCHAR(100) DEFAULT current_user
);

-- Criar a regra: para cada INSERT em clientes, faça TAMBÉM um INSERT no log
CREATE RULE log_insert_clientes
AS ON INSERT TO clientes
DO ALSO
    INSERT INTO log_auditoria (operacao, tabela, registro_id, data_operacao)
    VALUES ('INSERT', 'clientes', NEW.id, now());

Ao executar INSERT INTO clientes (nome, email) VALUES ('Ana', 'ana@email.com');, o PostgreSQL automaticamente insere um registro em log_auditoria. 


-- Testar: inserir um cliente
INSERT INTO clientes (nome, email) VALUES ('Ana Silva', 'ana@email.com');
INSERT INTO clientes (nome, email) VALUES ('João Souza', 'joao@email.com');

-- Verificar o log gerado automaticamente
SELECT * FROM log_auditoria;

Exemplo 2: DO INSTEAD — Redirecionar INSERT em view para tabela base

Quando criamos uma view, podemos usar DO INSTEAD para tornar a view "updatable" — ou seja, que aceite comandos de INSERT, UPDATE e DELETE que são redirecionados para a tabela base. 


-- Criar uma view de clientes ativos
CREATE VIEW view_clientes_ativos AS
    SELECT id, nome, email, status, criado_em
    FROM clientes
    WHERE status = 'ativo';

-- Criar regra DO INSTEAD para INSERT na view
CREATE RULE redirect_insert_view
AS ON INSERT TO view_clientes_ativos
DO INSTEAD
    INSERT INTO clientes (nome, email, status)
    VALUES (NEW.nome, NEW.email, 'ativo');

-- Agora podemos inserir via view!
INSERT INTO view_clientes_ativos (nome, email) VALUES ('Carlos Lima', 'carlos@email.com');

-- Verificar: o cliente aparece na tabela base e na view
SELECT * FROM clientes;
SELECT * FROM view_clientes_ativos;

Exemplo 3: DO INSTEAD — Simulando UPDATE via view


-- Regra para UPDATE na view
CREATE RULE redirect_update_view
AS ON UPDATE TO view_clientes_ativos
DO INSTEAD
    UPDATE clientes
    SET nome = NEW.nome,
        email = NEW.email
    WHERE id = OLD.id;

-- Atualizar via view
UPDATE view_clientes_ativos
SET email = 'carlos.novo@email.com'
WHERE id = 3;

-- Verificar atualização
SELECT * FROM view_clientes_ativos WHERE id = 3;

Exemplo 4: DO INSTEAD com DELETE — Remoção lógica via view


-- Regra para DELETE na view: ao invés de deletar, marca como inativo
CREATE RULE redirect_delete_view
AS ON DELETE TO view_clientes_ativos
DO INSTEAD
    UPDATE clientes
    SET status = 'inativo'
    WHERE id = OLD.id;

-- "Deletar" via view (na verdade, soft delete)
DELETE FROM view_clientes_ativos WHERE id = 3;

-- O registro não aparece mais na view, mas ainda existe na tabela base
SELECT * FROM clientes WHERE id = 3;       -- status = 'inativo'
SELECT * FROM view_clientes_ativos;         -- id 3 sumiu!

Exemplo 5: Regra com condição WHERE — Proteção contra DELETE indevido 


-- Regra que bloqueia DELETE em clientes com status 'ativo'
-- Se o status for 'inativo', permite o DELETE normalmente
-- Se o status for 'ativo', DO INSTEAD NOTHING (não faz nada)

CREATE RULE protege_delete_ativos
AS ON DELETE TO clientes
WHERE OLD.status = 'ativo'
DO INSTEAD NOTHING;

-- Testar: tentar deletar cliente ativo
DELETE FROM clientes WHERE id = 1;  -- cliente ativo → NÃO deletado!

-- Verificar
SELECT * FROM clientes WHERE id = 1;  -- ainda existe!

-- Atualizar para inativo e tentar novamente
UPDATE clientes SET status = 'inativo' WHERE id = 1;
DELETE FROM clientes WHERE id = 1;  -- cliente inativo → DELETADO!

Exemplo 6: Regra para SELECT — Redirecionamento de consulta

Regras de SELECT permitem interceptar consultas e modificá-las. Útil para particionamento transparente ou filtros automáticos. 


-- Suponha uma tabela particionada manualmente por ano
CREATE TABLE vendas_2024 (
    id SERIAL PRIMARY KEY,
    produto VARCHAR(100),
    valor NUMERIC,
    data_venda DATE DEFAULT CURRENT_DATE,
    CONSTRAINT vendas_2024_ano_check CHECK (EXTRACT(YEAR FROM data_venda) = 2024)
);

CREATE TABLE vendas_2025 (
    id SERIAL PRIMARY KEY,
    produto VARCHAR(100),
    valor NUMERIC,
    data_venda DATE DEFAULT CURRENT_DATE,
    CONSTRAINT vendas_2025_ano_check CHECK (EXTRACT(YEAR FROM data_venda) = 2025)
);

-- View que une as partições
CREATE VIEW vendas AS
    SELECT * FROM vendas_2024
    UNION ALL
    SELECT * FROM vendas_2025;

-- Regra que redireciona INSERTs para a partição correta
CREATE RULE vendas_insert_2024
AS ON INSERT TO vendas
WHERE EXTRACT(YEAR FROM NEW.data_venda) = 2024
DO INSTEAD
    INSERT INTO vendas_2024 (produto, valor, data_venda)
    VALUES (NEW.produto, NEW.valor, NEW.data_venda);

CREATE RULE vendas_insert_2025
AS ON INSERT TO vendas
WHERE EXTRACT(YEAR FROM NEW.data_venda) = 2025
DO INSTEAD
    INSERT INTO vendas_2025 (produto, valor, data_venda)
    VALUES (NEW.produto, NEW.valor, NEW.data_venda);

-- Agora o INSERT na view é automaticamente redirecionado
INSERT INTO vendas (produto, valor, data_venda) VALUES ('Notebook', 4500, '2024-06-15');
INSERT INTO vendas (produto, valor, data_venda) VALUES ('Mouse', 120, '2025-02-10');

SELECT * FROM vendas_2024;
SELECT * FROM vendas_2025;

Exemplo 7: Múltiplas ações em uma regra


-- Regra que loga DELETE + registra no log de auditoria
CREATE RULE log_e_registra_delete
AS ON DELETE TO clientes
DO ALSO (
    INSERT INTO log_auditoria (operacao, tabela, registro_id, data_operacao)
    VALUES ('DELETE', 'clientes', OLD.id, now());

    INSERT INTO clientes_historico (cliente_id, nome, email, status, deletado_em)
    VALUES (OLD.id, OLD.nome, OLD.email, OLD.status, now());
);

Metas e Objetivos das Regras — Casos de Uso

Caso de Uso Descrição Tipo de Regra
Auditoria / Logging Registrar automaticamente operações DML em tabelas de log DO ALSO + INSERT
Views atualizáveis (Updatable Views) Permitir INSERT/UPDATE/DELETE em views complexas, redirecionando para tabelas base DO INSTEAD
Soft Delete Transformar DELETE em UPDATE de status (marcar como inativo ao invés de remover) DO INSTEAD
Particionamento de dados Redirecionar INSERTs para a partição correta baseado em condição (data, região, etc.) DO INSTEAD + WHERE
Segurança / Bloqueio Prevenir operações perigosas em registros específicos (ex: bloquear DELETE em registros ativos) DO INSTEAD NOTHING + WHERE
Denormalização automática Atualizar automaticamente colunas derivadas em outras tabelas quando dados mudam DO ALSO + UPDATE
Migração transparente Durante migração de schema, redirecionar operações da tabela antiga para a nova DO INSTEAD

Comparação: Regras vs Triggers

Embora ambos possam ser usados para automatizar ações no banco, regras e triggers operam em momentos diferentes e têm características distintas. A escolha errada pode levar a bugs sutis ou problemas de performance.

Característica Regras (Rules) Triggers
Momento de atuação Antes da execução (reescrita da query) Durante/após a execução (BEFORE/AFTER)
Nível de operação Sempre a nível de statement (comando inteiro) FOR EACH ROW ou FOR EACH STATEMENT
Acesso a OLD/NEW Sim, via OLD e NEW na cláusula WHERE e comandos Sim, via variáveis OLD e NEW (FOR EACH ROW)
Performance Mais rápido para operações em lote, pois reescreve a query inteira uma vez Mais lento para operações em massa (executado por linha)
Suporte a SELECT Sim (regras de SELECT são únicas das regras) Não (triggers não interceptam SELECT)
Múltiplas regras Múltiplas regras para o mesmo evento são possíveis, mas a ordem de aplicação pode ser complexa Triggers têm ordem definida (alfabética por nome)
RETURNING Pode não funcionar corretamente com regras complexas (DO INSTEAD quebra o vínculo) Funciona normalmente com triggers
Debuggability Difícil de debugar — a query original é reescrita antes de chegar ao log Mais fácil — triggers aparecem nos logs e no EXPLAIN
Recomendação Casos específicos: views atualizáveis, particionamento, log simples Caso geral: validação, auditoria complexa, lógica de negócio
Atenção: A documentação oficial do PostgreSQL recomenda: "For the things that can be implemented by both, which is best depends on the usage of the database. A trigger is fired for any row affected once. A rule manipulates the query or generates an additional query. So if many rows are affected in one statement, a rule issuing one extra command is likely to be faster than a trigger that is called for every single row and must execute its operations many times."

Views no PostgreSQL — Revisão e Tópicos Avançados

CRUD básico de Views


-- Criar view
CREATE VIEW clientes_ativos AS
    SELECT id, nome, email, criado_em
    FROM clientes
    WHERE status = 'ativo';

-- Substituir view (sem perder permissões e dependências)
CREATE OR REPLACE VIEW clientes_ativos AS
    SELECT id, nome, email, status, criado_em
    FROM clientes
    WHERE status = 'ativo' AND email IS NOT NULL;

-- Listar views do schema atual
SELECT table_schema, table_name
FROM information_schema.views
WHERE table_schema NOT IN ('pg_catalog', 'information_schema');

-- Remover view
DROP VIEW IF EXISTS clientes_ativos;

-- Remover view e todas as views dependentes (CASCADE)
DROP VIEW IF EXISTS clientes_ativos CASCADE;

-- Renomear view
ALTER VIEW clientes_ativos RENAME TO v_clientes_ativos;

Views que são automaticamente atualizáveis

O PostgreSQL suporta views simples que são automaticamente atualizáveis (auto-updatable) — ou seja, você pode fazer INSERT, UPDATE e DELETE diretamente, sem precisar de regras ou triggers, desde que:

  • A view referencia apenas uma tabela (ou uma view atualizável)
  • A view não contém: DISTINCT, GROUP BY, HAVING, LIMIT/OFFSET, UNION/INTERSECT/EXCEPT, funções de agregação, funções de janela
  • Todas as colunas da view que não têm valor default na tabela base precisam estar presentes na view para INSERT 

-- Esta view é automaticamente atualizável (single table, no aggregation)
CREATE VIEW v_clientes_nordeste AS
    SELECT id, nome, email, estado, cidade
    FROM clientes
    WHERE estado IN ('BA', 'PE', 'CE', 'MA', 'PI', 'RN', 'PB', 'SE', 'AL');

-- INSERT funciona diretamente!
INSERT INTO v_clientes_nordeste (nome, email, estado, cidade)
VALUES ('Maria Oliveira', 'maria@email.com', 'BA', 'Salvador');

-- UPDATE funciona diretamente!
UPDATE v_clientes_nordeste
SET email = 'maria.nova@email.com'
WHERE id = 10;

-- DELETE funciona diretamente! (remove da tabela base)
DELETE FROM v_clientes_nordeste WHERE id = 10;

WITH CHECK OPTION — Protegendo a integridade da view

O WITH CHECK OPTION impede que operações de INSERT ou UPDATE criem/alterem linhas que depois não seriam visíveis na própria view. Isso garante a consistência lógica da view. 


-- Criar view COM CHECK OPTION
CREATE VIEW clientes_nordeste_check AS
    SELECT id, nome, email, estado, cidade
    FROM clientes
    WHERE estado IN ('BA', 'PE', 'CE', 'MA', 'PI', 'RN', 'PB', 'SE', 'AL')
WITH CHECK OPTION;

-- Isto FUNCIONA: insere cliente no Nordeste
INSERT INTO clientes_nordeste_check (nome, email, estado, cidade)
VALUES ('Pedro Santos', 'pedro@email.com', 'PE', 'Recife');

-- Isto FALHA: tenta inserir cliente de São Paulo (fora do Nordeste)
-- ERRO: new row violates check option for view "clientes_nordeste_check"
INSERT INTO clientes_nordeste_check (nome, email, estado, cidade)
VALUES ('Paulo Souza', 'paulo@email.com', 'SP', 'São Paulo');

-- Isto FALHA: tenta atualizar cliente do Nordeste para estado de SP
UPDATE clientes_nordeste_check
SET estado = 'SP'
WHERE id = 10;

WITH LOCAL CHECK OPTION vs WITH CASCADED CHECK OPTION

Quando você tem views aninhadas (uma view que referencia outra view), a diferença entre LOCAL e CASCADED se torna importante: 


-- View base (sem check option)
CREATE VIEW v_clientes_ativos AS
    SELECT * FROM clientes
    WHERE status = 'ativo';

-- View derivada LOCAL: verifica APENAS a condição DESTA view (email NOT NULL)
-- Não propaga a verificação para a view base
CREATE VIEW v_clientes_contato_local AS
    SELECT * FROM v_clientes_ativos
    WHERE email IS NOT NULL
WITH LOCAL CHECK OPTION;

-- View derivada CASCADED (padrão): verifica esta view E TODAS as views base na hierarquia
CREATE VIEW v_clientes_contato_cascaded AS
    SELECT * FROM v_clientes_ativos
    WHERE email IS NOT NULL
WITH CASCADED CHECK OPTION;
Resumo: WITH LOCAL CHECK OPTION verifica apenas a condição da view atual. WITH CASCADED CHECK OPTION (padrão quando se usa apenas WITH CHECK OPTION) verifica a condição da view atual e de todas as views base na hierarquia.

Views com JOIN — Não são automaticamente atualizáveis

Views que envolvem JOIN entre múltiplas tabelas não são automaticamente atualizáveis. Para permitir INSERT/UPDATE/DELETE, você precisa criar regras ou triggers DO INSTEAD. 


-- View com JOIN (não é automaticamente atualizável)
CREATE VIEW v_pedidos_clientes AS
    SELECT p.id AS pedido_id,
           p.data_pedido,
           p.valor_total,
           c.nome AS cliente_nome,
           c.email AS cliente_email
    FROM pedidos p
    JOIN clientes c ON p.cliente_id = c.id;

-- Para tornar atualizável, criamos regras
CREATE RULE insert_v_pedidos_clientes
AS ON INSERT TO v_pedidos_clientes
DO INSTEAD (
    -- Primeiro garante que o cliente existe (ou insere)
    INSERT INTO clientes (nome, email)
    VALUES (NEW.cliente_nome, NEW.cliente_email)
    ON CONFLICT (email) DO NOTHING;

    -- Depois insere o pedido
    INSERT INTO pedidos (cliente_id, data_pedido, valor_total)
    SELECT id, NEW.data_pedido, NEW.valor_total
    FROM clientes
    WHERE email = NEW.cliente_email;
);

Views Materializadas no PostgreSQL

Uma Materialized View é uma view cujo resultado é armazenado fisicamente em disco. Diferente de uma view comum (que é apenas uma query salva, executada sob demanda), a materialized view contém uma cópia dos dados no momento do último refresh. Isso oferece performance de leitura muito superior para consultas complexas, ao custo de os dados poderem ficar desatualizados (stale).

Criando uma Materialized View


-- Criar tabelas de exemplo para demonstração
CREATE TABLE produtos (
    id SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    categoria VARCHAR(50),
    preco NUMERIC(10,2),
    estoque INTEGER DEFAULT 0,
    data_cadastro DATE DEFAULT CURRENT_DATE
);

-- Inserir dados de exemplo
INSERT INTO produtos (nome, categoria, preco, estoque) VALUES
    ('Notebook Dell', 'Informática', 4500.00, 15),
    ('Mouse Logitech', 'Informática', 150.00, 200),
    ('Teclado Mecânico', 'Informática', 350.00, 80),
    ('Cadeira Gamer', 'Móveis', 1800.00, 25),
    ('Mesa Escritório', 'Móveis', 750.00, 40),
    ('Monitor 27"', 'Informática', 2200.00, 30),
    ('Headset Bluetooth', 'Áudio', 400.00, 55),
    ('Caixa de Som', 'Áudio', 600.00, 35),
    ('Webcam HD', 'Informática', 250.00, 90),
    ('Hub USB', 'Informática', 80.00, 150);

-- Criar materialized view com agregação
CREATE MATERIALIZED VIEW mv_resumo_produtos AS
    SELECT
        categoria,
        COUNT(*) AS total_produtos,
        SUM(estoque) AS estoque_total,
        ROUND(AVG(preco), 2) AS preco_medio,
        MIN(preco) AS preco_minimo,
        MAX(preco) AS preco_maximo,
        SUM(preco * estoque) AS valor_total_estoque
    FROM produtos
    GROUP BY categoria
    ORDER BY total_produtos DESC;

-- Consultar a materialized view (muito rápido!)
SELECT * FROM mv_resumo_produtos;

REFRESH MATERIALIZED VIEW

Como os dados da materialized view são uma fotografia (snapshot) do momento da criação/refresh, você precisa atualizá-la manualmente quando os dados base mudarem. 


-- Inserir novos produtos
INSERT INTO produtos (nome, categoria, preco, estoque) VALUES
    ('Suporte Monitor', 'Informática', 120.00, 60),
    ('Fone de Ouvido', 'Áudio', 200.00, 45);

-- Neste momento, mv_resumo_produtos NÃO reflete os novos dados!
SELECT * FROM mv_resumo_produtos;  -- dados desatualizados!

-- FORMA 1: Refresh simples (bloqueia leituras durante a atualização)
REFRESH MATERIALIZED VIEW mv_resumo_produtos;

-- Agora os dados estão atualizados
SELECT * FROM mv_resumo_produtos;

REFRESH MATERIALIZED VIEW CONCURRENTLY

O REFRESH CONCURRENTLY permite atualizar a materialized view sem bloquear leituras simultâneas. Para isso funcionar, a view precisa ter um índice único. 


-- Criar índice único (obrigatório para CONCURRENTLY)
CREATE UNIQUE INDEX idx_mv_resumo_categoria
    ON mv_resumo_produtos (categoria);

-- FORMA 2: Refresh concorrente (NÃO bloqueia leituras)
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_resumo_produtos;

-- Verificar dados atualizados
SELECT * FROM mv_resumo_produtos;

Materialized Views vs Views — Comparação

Característica View Regular Materialized View
Armazenamento Não armazena dados (apenas a definição da query) Armazena fisicamente os dados em disco
Velocidade de leitura Query executada toda vez que a view é consultada Muito rápida — dados pré-calculados
Atualização dos dados Sempre atualizados (reflete tabelas base em tempo real) Ficam desatualizados até o próximo REFRESH
Uso de disco Zero (não armazena dados) Usa espaço em disco proporcional ao resultado
Índices Não permite criar índices na view Permite criar índices na materialized view
Escrita (INSERT/UPDATE/DELETE) Possível com regras ou triggers DO INSTEAD Não suporta operações de escrita
Casos de uso ideais Consultas simples, segurança (row-level security), abstração de schema Dashboards, relatórios, agregações pesadas, data warehousing

Gerenciando Materialized Views


-- Listar todas as materialized views
SELECT schemaname, matviewname, hasindexes, ispopulated
FROM pg_matviews;

-- Verificar tamanho em disco
SELECT
    schemaname,
    matviewname,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || matviewname)) AS tamanho
FROM pg_matviews;

-- Renomear
ALTER MATERIALIZED VIEW mv_resumo_produtos RENAME TO mv_produtos_resumo;

-- Alterar dono
ALTER MATERIALIZED VIEW mv_produtos_resumo OWNER TO outro_usuario;

-- Remover
DROP MATERIALIZED VIEW IF EXISTS mv_produtos_resumo;

Agendando REFRESH automático (pg_cron ou pgAgent)

Para ambientes de produção, você pode agendar o refresh periódico usando extensões como pg_cron: 


-- Instalar extensão pg_cron (requer superuser)
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- Agendar refresh a cada 30 minutos
SELECT cron.schedule(
    'refresh-mv-resumo',     -- nome do job
    '*/30 * * * *',          -- cron: a cada 30 minutos
    'REFRESH MATERIALIZED VIEW CONCURRENTLY mv_resumo_produtos;'
);

-- Listar jobs agendados
SELECT * FROM cron.job;

-- Remover agendamento
SELECT cron.unschedule('refresh-mv-resumo');
          

# Alternativa: agendar via crontab do Linux (sem pg_cron)
# Executar refresh a cada 30 minutos
*/30 * * * * psql -U postgres -d meu_banco -c "REFRESH MATERIALIZED VIEW CONCURRENTLY mv_resumo_produtos;"

Passo a passo: Sistema de logging com Rules + Views materializadas

Vamos construir um sistema completo de auditoria usando Rules para logging automático e uma Materialized View para o dashboard de relatórios.

Passo 1 — Criar tabelas base e tabela de auditoria


-- Tabela de clientes
CREATE TABLE clientes (
    id SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE,
    telefone VARCHAR(20),
    cidade VARCHAR(50),
    estado CHAR(2),
    status VARCHAR(20) DEFAULT 'ativo',
    saldo NUMERIC(12,2) DEFAULT 0,
    criado_em TIMESTAMP DEFAULT now(),
    atualizado_em TIMESTAMP DEFAULT now()
);

-- Tabela de pedidos
CREATE TABLE pedidos (
    id SERIAL PRIMARY KEY,
    cliente_id INTEGER REFERENCES clientes(id),
    descricao VARCHAR(200),
    valor NUMERIC(12,2) NOT NULL,
    status_pedido VARCHAR(20) DEFAULT 'pendente',
    data_pedido DATE DEFAULT CURRENT_DATE
);

-- Tabela de auditoria centralizada
CREATE TABLE auditoria (
    id SERIAL PRIMARY KEY,
    operacao VARCHAR(10) NOT NULL,       -- INSERT, UPDATE, DELETE
    tabela VARCHAR(50) NOT NULL,
    registro_id INTEGER,
    dados_antigos JSONB,                 -- snapshot antes da mudança (UPDATE/DELETE)
    dados_novos JSONB,                   -- snapshot depois da mudança (INSERT/UPDATE)
    usuario_executor VARCHAR(100) DEFAULT current_user,
    data_operacao TIMESTAMP DEFAULT now()
);

-- Criar índice para consultas de auditoria
CREATE INDEX idx_auditoria_tabela ON auditoria (tabela);
CREATE INDEX idx_auditoria_data ON auditoria (data_operacao);
CREATE INDEX idx_auditoria_operacao ON auditoria (operacao);

Passo 2 — Criar regras de auditoria para clientes


-- Regra: logar INSERT em clientes
CREATE RULE audit_insert_clientes
AS ON INSERT TO clientes
DO ALSO
    INSERT INTO auditoria (operacao, tabela, registro_id, dados_novos)
    VALUES ('INSERT', 'clientes', NEW.id,
            jsonb_build_object(
                'nome', NEW.nome, 'email', NEW.email,
                'telefone', NEW.telefone, 'cidade', NEW.cidade,
                'estado', NEW.estado, 'saldo', NEW.saldo
            ));

-- Regra: logar UPDATE em clientes
CREATE RULE audit_update_clientes
AS ON UPDATE TO clientes
DO ALSO
    INSERT INTO auditoria (operacao, tabela, registro_id, dados_antigos, dados_novos)

-- Criar tabela com soft delete
CREATE TABLE produtos (
    id SERIAL PRIMARY KEY,
    nome VARCHAR(100),
    preco NUMERIC(10,2),
    deletado BOOLEAN DEFAULT FALSE
);

-- View que oculta produtos deletados
CREATE VIEW v_produtos_ativos AS
    SELECT id, nome, preco
    FROM produtos
    WHERE deletado = FALSE;

-- Regra que transforma DELETE em soft delete
CREATE RULE soft_delete_produto
AS ON DELETE TO v_produtos_ativos
DO INSTEAD
    UPDATE produtos
    SET deletado = TRUE
    WHERE id = OLD.id;

-- Inserir dados
INSERT INTO produtos (nome, preco) VALUES ('Produto A', 100);
INSERT INTO produtos (nome, preco) VALUES ('Produto B', 200);
INSERT INTO produtos (nome, preco) VALUES ('Produto C', 300);

-- Soft delete via view
DELETE FROM v_produtos_ativos WHERE id = 2;

-- View não mostra mais o produto 2
SELECT * FROM v_produtos_ativos;       -- Apenas produtos A e C

-- Mas na tabela ele ainda existe com deletado = TRUE
SELECT * FROM produtos;                 -- Todos os 3 produtos

2. Criar view com JOIN e WITH CHECK OPTION

Crie duas tabelas: categorias e produtos. Crie uma view chamada v_produtos_eletronicos que faz JOIN entre as tabelas e mostra apenas produtos da categoria "Eletrônicos". Adicione WITH CHECK OPTION para garantir que inserções via view respeitem a condição da categoria.

Gabarito: 

-- Criar tabela categorias
CREATE TABLE categorias (
    id SERIAL PRIMARY KEY,
    nome VARCHAR(50) UNIQUE NOT NULL
);

-- Criar tabela produtos
CREATE TABLE produtos (
    id SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    preco NUMERIC(10,2),
    categoria_id INTEGER REFERENCES categorias(id)
);

-- Inserir categorias
INSERT INTO categorias (nome) VALUES ('Eletrônicos'), ('Móveis'), ('Vestuário');

-- Criar view com JOIN e CHECK OPTION
CREATE VIEW v_produtos_eletronicos AS
    SELECT p.id, p.nome, p.preco, c.nome AS categoria
    FROM produtos p
    JOIN categorias c ON p.categoria_id = c.id
    WHERE c.nome = 'Eletrônicos'
WITH CHECK OPTION;

-- Como a view tem JOIN, ela NÃO é automaticamente atualizável
-- Precisamos criar regras DO INSTEAD para INSERT/UPDATE/DELETE

-- Regra para INSERT
CREATE RULE insert_v_produtos_eletronicos
AS ON INSERT TO v_produtos_eletronicos
DO INSTEAD
    INSERT INTO produtos (nome, preco, categoria_id)
    SELECT NEW.nome, NEW.preco, c.id
    FROM categorias c
    WHERE c.nome = 'Eletrônicos';

-- Regra para UPDATE
CREATE RULE update_v_produtos_eletronicos
AS ON UPDATE TO v_produtos_eletronicos
DO INSTEAD
    UPDATE produtos
    SET nome = NEW.nome, preco = NEW.preco
    WHERE id = OLD.id
      AND categoria_id = (SELECT id FROM categorias WHERE nome = 'Eletrônicos');

-- Regra para DELETE
CREATE RULE delete_v_produtos_eletronicos
AS ON DELETE TO v_produtos_eletronicos
DO INSTEAD
    DELETE FROM produtos
    WHERE id = OLD.id
      AND categoria_id = (SELECT id FROM categorias WHERE nome = 'Eletrônicos');

3. Criar materialized view com agregação e configurar refresh periódico

Crie uma materialized view que sumarize vendas por mês e por categoria de produto. A view deve conter: mês, categoria, total de vendas, valor total vendido, ticket médio. Configure um refresh periódico usando pg_cron (ou documente como faria via crontab).

Gabarito: 

-- Criar tabelas base para o exercício
CREATE TABLE categorias (
    id SERIAL PRIMARY KEY,
    nome VARCHAR(50) UNIQUE NOT NULL
);

CREATE TABLE produtos (
    id SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    preco NUMERIC(10,2),
    categoria_id INTEGER REFERENCES categorias(id)
);

CREATE TABLE vendas (
    id SERIAL PRIMARY KEY,
    produto_id INTEGER REFERENCES produtos(id),
    quantidade INTEGER NOT NULL,
    valor_unitario NUMERIC(10,2) NOT NULL,
    data_venda DATE DEFAULT CURRENT_DATE
);

    (3, 3, 750, '2025-04-10');

-- Criar materialized view de resumo mensal
CREATE MATERIALIZED VIEW mv_vendas_mensais AS
    SELECT
        TO_CHAR(v.data_venda, 'YYYY-MM') AS mes,
        c.nome AS categoria,
        COUNT(*) AS total_vendas,
        SUM(v.quantidade * v.valor_unitario) AS valor_total,
        ROUND(AVG(v.quantidade * v.valor_unitario), 2) AS ticket_medio,
        SUM(v.quantidade) AS total_itens_vendidos
    FROM vendas v
    JOIN produtos p ON v.produto_id = p.id
    JOIN categorias c ON p.categoria_id = c.id
    GROUP BY TO_CHAR(v.data_venda, 'YYYY-MM'), c.nome
    ORDER BY mes DESC, valor_total DESC;

-- Criar índice único para CONCURRENTLY
CREATE UNIQUE INDEX idx_mv_vendas_mes_cat
    ON mv_vendas_mensais (mes, categoria);

-- Consultar o resumo
SELECT * FROM mv_vendas_mensais;

-- Refresh concorrente
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_vendas_mensais;

-- Configurar refresh automático com pg_cron
CREATE EXTENSION IF NOT EXISTS pg_cron;

SELECT cron.schedule(
    'refresh-vendas-mensais',
    '0 */6 * * *',              -- a cada 6 horas
    'REFRESH MATERIALIZED VIEW CONCURRENTLY mv_vendas_mensais;'
);
