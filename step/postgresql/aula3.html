Revisão de Views

Uma view (visão) é uma tabela virtual baseada no resultado de uma consulta SQL. Ela não armazena dados fisicamente — em vez disso, a consulta que a define é executada toda vez que a view é acessada. Pense nela como uma "janela" para os dados de uma ou mais tabelas.

Benefícios das Views

  • Segurança: ocultar colunas sensíveis (ex: salários) e expor apenas os dados necessários para determinados usuários. Você pode conceder acesso à view sem dar acesso direto às tabelas base.
  • Simplicidade: encapsular consultas complexas com JOINs, agregações e subconsultas em um único objeto reutilizável. O usuário faz SELECT * FROM resumo_vendas em vez de escrever uma query de 20 linhas.
  • Abstração de dados: isolar aplicações de mudanças na estrutura das tabelas. Se uma tabela for alterada, você só precisa atualizar a view, não todos os sistemas que a consomem.
  • Consistência: garantir que todos os usuários e aplicações vejam os dados da mesma forma, seguindo as mesmas regras de negócio definidas na view.

No Oracle, views são objetos de esquema armazenados no dicionário de dados. Diferente de alguns SGBDs, o Oracle oferece recursos avançados como FORCE VIEW, WITH READ ONLY, WITH CHECK OPTION e, principalmente, Materialized Views, que abordaremos em detalhes nesta aula.

Criando Views no Oracle

Sintaxe Básica — CREATE VIEW

A sintaxe fundamental para criar uma view é:


CREATE [OR REPLACE] [FORCE | NOFORCE] VIEW nome_view [(alias1, alias2, ...)]
AS
  consulta_select
[WITH READ ONLY]
[WITH CHECK OPTION [CONSTRAINT nome_constraint]];

Cada cláusula opcional tem um propósito específico:

  • OR REPLACE: recria a view se ela já existir, sem precisar de DROP primeiro
  • FORCE: cria a view mesmo que as tabelas referenciadas não existam ou você não tenha privilégios sobre elas
  • NOFORCE: (padrão) só cria se as tabelas base existirem
  • WITH READ ONLY: impede operações DML (INSERT, UPDATE, DELETE) na view
  • WITH CHECK OPTION: garante que dados inseridos/atualizados pela view respeitem a cláusula WHERE da view
Exemplo 1 — View Simples

Filtrar funcionários ativos de uma tabela:


CREATE VIEW funcionarios_ativos AS
SELECT id, nome, cargo, departamento, data_admissao
FROM funcionarios
WHERE status = 'A';

Agora, ao executar SELECT * FROM funcionarios_ativos;, você vê apenas os funcionários com status 'A' (Ativo), sem precisar repetir o filtro toda vez.

Exemplo 2 — View com JOIN

Encapsular relacionamentos entre tabelas:


CREATE VIEW resumo_pedidos AS
SELECT
  c.nome AS cliente,
  c.email,
  p.data_pedido,
  p.valor_total,
  p.status AS status_pedido
FROM clientes c
JOIN pedidos p ON c.id = p.cliente_id;

Esta view combina dados de clientes e pedidos em um único lugar. Qualquer consulta futura pode simplesmente usar SELECT * FROM resumo_pedidos WHERE status_pedido = 'Pendente';

Exemplo 3 — View com Agregação

CREATE VIEW vendas_por_categoria AS
SELECT
  c.nome_categoria,
  COUNT(v.id) AS total_vendas,
  SUM(v.valor) AS receita_total,
  AVG(v.valor) AS ticket_medio
FROM vendas v
JOIN categorias c ON v.categoria_id = c.id
GROUP BY c.nome_categoria;

FORCE VIEW — Criando Views com Dependências Inexistentes

O Oracle permite criar uma view mesmo que o objeto base (tabela ou outra view) ainda não exista. Isso é útil durante desenvolvimento, quando você está construindo objetos em uma ordem específica ou quando executa scripts de migração.

A view criada com FORCE ficará com status INVALID até que o objeto referenciado seja criado. Depois disso, o Oracle a recompila automaticamente no primeiro uso (ou você pode forçar a compilação manualmente). 


CREATE FORCE VIEW visao_teste AS
SELECT id, descricao, valor
FROM tabela_inexistente
WHERE ativo = 1;

Verificando o status da view:


SELECT object_name, status
FROM user_objects
WHERE object_type = 'VIEW'
  AND object_name = 'VISAO_TESTE';

WITH READ ONLY — Views Somente Leitura

Por padrão, views simples (baseadas em uma única tabela, sem agregações ou DISTINCT) permitem operações DML. Para impedir isso, use WITH READ ONLY: 


CREATE VIEW salarios_rh AS
SELECT id, nome, cargo, salario
FROM funcionarios
WHERE departamento = 'RH'
WITH READ ONLY;

Qualquer tentativa de INSERT, UPDATE ou DELETE em salarios_rh resultará em erro: ORA-42399: cannot perform a DML operation on a read-only view.

WITH CHECK OPTION — Restringindo DML à Condição da View

O WITH CHECK OPTION garante que qualquer linha inserida ou atualizada através da view satisfaça a condição WHERE da view. Isso evita que você insira dados que depois "desapareceriam" da view. 


CREATE VIEW produtos_caros AS
SELECT id, nome, preco, categoria
FROM produtos
WHERE preco > 1000
WITH CHECK OPTION;

Agora:

  • INSERT INTO produtos_caros VALUES (1, 'Notebook', 3500, 'TI'); — funciona (preço > 1000)
  • INSERT INTO produtos_caros VALUES (2, 'Mouse', 50, 'TI'); — falha (preço < 1000, viola CHECK OPTION)
  • UPDATE produtos_caros SET preco = 500 WHERE id = 1; — falha (alteraria para preço fora da condição)
Exemplo com CONSTRAINT nomeada:

CREATE VIEW funcionarios_ti AS
SELECT id, nome, salario
FROM funcionarios
WHERE departamento = 'TI'
WITH CHECK OPTION CONSTRAINT ck_func_ti_dept;

Modificando Views

CREATE OR REPLACE VIEW

No Oracle, não existe ALTER VIEW ... AS SELECT .... Para modificar a definição de uma view, usa-se CREATE OR REPLACE VIEW. Isso preserva as permissões (grants) já concedidas sobre a view. 


CREATE OR REPLACE VIEW funcionarios_ativos AS
SELECT id, nome, cargo, departamento, data_admissao, salario
FROM funcionarios
WHERE status = 'A'
WITH READ ONLY;

Neste exemplo, adicionamos a coluna salario e a cláusula WITH READ ONLY a uma view que já existia. Os privilégios concedidos anteriormente (GRANT SELECT, etc.) são mantidos.

ALTER VIEW ... COMPILE — Recompilando Views Inválidas

Quando a estrutura de uma tabela base muda (ex: coluna é removida ou renomeada), a view pode ficar com status INVALID. O Oracle recompila automaticamente no primeiro uso, mas você pode forçar a recompilação manualmente: 


ALTER VIEW resumo_pedidos COMPILE;

Para recompilar todas as views inválidas de uma vez:


BEGIN
  FOR r IN (SELECT object_name FROM user_objects
            WHERE object_type = 'VIEW' AND status = 'INVALID')
  LOOP
    EXECUTE IMMEDIATE 'ALTER VIEW ' || r.object_name || ' COMPILE';
  END LOOP;
END;
/

Excluindo Views

DROP VIEW

Para remover uma view do banco de dados:


DROP VIEW nome_view;

DROP VIEW ... CASCADE CONSTRAINTS

Se outras views ou constraints referenciam a view que você está excluindo, será necessário usar CASCADE CONSTRAINTS. Isso remove as constraints dependentes (como FOREIGN KEYs que referenciam a view): 


DROP VIEW produtos_caros CASCADE CONSTRAINTS;
          

-- Excluir view apenas se existir (Oracle 23c+)
DROP VIEW IF EXISTS visao_teste;

-- Ou use bloco PL/SQL para evitar erro:
BEGIN
  EXECUTE IMMEDIATE 'DROP VIEW visao_teste';
EXCEPTION
  WHEN OTHERS THEN
    IF SQLCODE != -942 THEN
      RAISE;
    END IF;
END;
/

Consultando Metadados de Views no Oracle

O Oracle armazena informações sobre todas as views no dicionário de dados. Existem três níveis de visão do dicionário:

Visão Escopo Descrição
USER_VIEWS Esquema atual Views que pertencem ao usuário logado
ALL_VIEWS Esquemas acessíveis Views que o usuário tem permissão para acessar
DBA_VIEWS Todo o banco Todas as views do banco (requer privilégios DBA)
Consultar o texto SQL de uma view:

SELECT view_name, text_length, text
FROM user_views
WHERE view_name = 'FUNCIONARIOS_ATIVOS';
Listar todas as views do esquema atual:

SELECT object_name, status, created, last_ddl_time
FROM user_objects
WHERE object_type = 'VIEW'
ORDER BY object_name;
Listar colunas de uma view:

SELECT column_name, data_type, data_length
FROM user_tab_columns
WHERE table_name = 'RESUMO_PEDIDOS'
ORDER BY column_id;
Verificar dependências de uma view:

SELECT referenced_owner, referenced_name, referenced_type
FROM user_dependencies
WHERE name = 'RESUMO_PEDIDOS'
  AND type = 'VIEW';
Ver quais objetos dependem de uma view:

SELECT owner, name, type, status
FROM all_dependencies
WHERE referenced_name = 'FUNCIONARIOS_ATIVOS'
  AND referenced_type = 'VIEW';

Views Materializadas

Uma Materialized View (MV) é um objeto de banco de dados que armazena fisicamente o resultado de uma consulta, diferentemente de uma view comum que é apenas uma consulta armazenada executada sob demanda. As MVs são extremamente úteis para:

  • Data Warehousing: pré-agregar grandes volumes de dados para consultas analíticas rápidas
  • Replicação: manter cópias de dados entre bancos de dados distribuídos
  • Resumos e relatórios: armazenar resultados de consultas pesadas que não precisam ser em tempo real
  • Query Rewrite: o otimizador do Oracle pode redirecionar automaticamente consultas às tabelas base para usar a MV, melhorando drasticamente a performance

Sintaxe CREATE MATERIALIZED VIEW


CREATE MATERIALIZED VIEW nome_mv
[BUILD IMMEDIATE | BUILD DEFERRED]
[REFRESH [FAST | COMPLETE | FORCE]
  [ON DEMAND | ON COMMIT]
  [START WITH data_hora NEXT intervalo]]
[ENABLE QUERY REWRITE | DISABLE QUERY REWRITE]
AS
  consulta_select;
Opção Descrição
BUILD IMMEDIATE Preenche a MV com dados no momento da criação (padrão)
BUILD DEFERRED Cria a MV vazia; os dados são carregados no primeiro refresh
REFRESH COMPLETE Reexecuta a consulta inteira e substitui todos os dados
REFRESH FAST Aplica apenas as mudanças incrementais (requer MV Log)
REFRESH FORCE Tenta FAST; se não for possível, faz COMPLETE (padrão)
ON DEMAND Refresh manual ou agendado (padrão)
ON COMMIT Refresh automático a cada COMMIT nas tabelas base
ENABLE QUERY REWRITE Permite que o otimizador use esta MV para responder consultas às tabelas base
Exemplo 1 — MV com Agregação (Data Warehouse)

CREATE MATERIALIZED VIEW mv_vendas_resumo
BUILD IMMEDIATE
REFRESH COMPLETE ON DEMAND
AS
SELECT
  regiao,
  EXTRACT(YEAR FROM data_venda) AS ano,
  EXTRACT(MONTH FROM data_venda) AS mes,
  COUNT(*) AS total_vendas,
  SUM(valor) AS receita_total,
  AVG(valor) AS ticket_medio
FROM vendas
GROUP BY regiao, EXTRACT(YEAR FROM data_venda), EXTRACT(MONTH FROM data_venda);
Exemplo 2 — MV com JOIN

CREATE MATERIALIZED VIEW mv_cliente_compras
BUILD IMMEDIATE
REFRESH COMPLETE ON DEMAND
AS
SELECT
  c.id AS cliente_id,
  c.nome,
  c.cidade,
  COUNT(p.id) AS qtd_pedidos,
  SUM(p.valor_total) AS valor_acumulado,
  MAX(p.data_pedido) AS ultima_compra
FROM clientes c
JOIN pedidos p ON c.id = p.cliente_id
GROUP BY c.id, c.nome, c.cidade;
Exemplo 3 — ON COMMIT Refresh

A MV é atualizada automaticamente sempre que ocorre um COMMIT nas tabelas base: 


CREATE MATERIALIZED VIEW mv_estoque_atual
BUILD IMMEDIATE
REFRESH FAST ON COMMIT
AS
SELECT
  produto_id,
  nome_produto,
  SUM(quantidade) AS estoque_total,
  COUNT(DISTINCT armazem_id) AS armazens
FROM movimentacao_estoque
GROUP BY produto_id, nome_produto;
Exemplo 4 — Refresh Agendado

Atualiza diariamente às 6h da manhã:


CREATE MATERIALIZED VIEW mv_dashboard_diario
BUILD IMMEDIATE
REFRESH COMPLETE
START WITH TRUNC(SYSDATE) + 6/24
NEXT SYSDATE + 1
AS
SELECT
  TRUNC(data_venda) AS dia,
  COUNT(*) AS total_pedidos,
  SUM(valor) AS faturamento
FROM pedidos
GROUP BY TRUNC(data_venda);

Materialized View Logs — Habilitando Refresh FAST

Para que o Oracle consiga fazer REFRESH FAST (incremental), é necessário criar um Materialized View Log na tabela base. O log registra todas as alterações (INSERT, UPDATE, DELETE) para que o Oracle saiba exatamente quais linhas precisam ser atualizadas na MV, em vez de reconstruí-la completamente.

Criando um MV Log:

CREATE MATERIALIZED VIEW LOG ON vendas
WITH ROWID, SEQUENCE (valor, regiao, data_venda)
INCLUDING NEW VALUES;

Parâmetros importantes:

  • WITH ROWID: registra o ROWID de cada linha alterada — necessário para identificar unicamente as linhas afetadas
  • WITH SEQUENCE: inclui um número de sequência que permite ao Oracle ordenar as operações cronologicamente
  • WITH PRIMARY KEY: alternativa ao ROWID, usa a chave primária da tabela
  • INCLUDING NEW VALUES: registra tanto o valor antigo quanto o novo, necessário para atualizações em MVs com agregação
Exemplo completo — Tabela base + MV Log + MV com FAST Refresh:

-- 1. MV Log na tabela base
CREATE MATERIALIZED VIEW LOG ON vendas
WITH ROWID, SEQUENCE (valor, regiao)
INCLUDING NEW VALUES;

-- 2. Criar a MV com FAST refresh
CREATE MATERIALIZED VIEW mv_vendas_fast
BUILD IMMEDIATE
REFRESH FAST ON DEMAND
AS
SELECT
  regiao,
  COUNT(*) AS total,
  SUM(valor) AS soma
FROM vendas
GROUP BY regiao;

-- 3. Após inserir/atualizar dados na tabela VENDAS, atualize a MV:
EXEC DBMS_MVIEW.REFRESH('mv_vendas_fast', 'F');

Gerenciando Refresh de Materialized Views

DBMS_MVIEW.REFRESH — Refresh Manual

O pacote DBMS_MVIEW oferece procedimentos para gerenciar MVs programaticamente:

Refresh de uma única MV:

EXEC DBMS_MVIEW.REFRESH('mv_vendas_resumo', 'C');
Refresh de múltiplas MVs:

BEGIN
  DBMS_MVIEW.REFRESH('mv_vendas_resumo,mv_cliente_compras');
END;
/
Métodos de Refresh (parâmetro method):
  • 'C' — Complete refresh
  • 'F' — Fast refresh
  • '?' — Force refresh
  • 'A' — Always refresh (equivale a COMPLETE, mas considera atomicidade)
  • 'P' — Refresh usando particionamento (PCT — Partition Change Tracking)
Refresh com verificação de status:

-- Verificar última data de refresh de uma MV
SELECT
  mview_name,
  last_refresh_type,
  last_refresh_date,
  staleness,
  compile_state
FROM user_mviews
WHERE mview_name = 'MV_VENDAS_RESUMO';

-- Verificar logs de refresh
SELECT
  mview_name,
  master,
  log_table,
  log_owner
FROM user_mview_logs;

-- Forçar refresh atômico (tudo ou nada)
EXEC DBMS_MVIEW.REFRESH('mv_vendas_resumo', 'C', atomic_refresh => TRUE);

-- Estimativa de tempo para refresh
EXEC DBMS_MVIEW.EXPLAIN_MVIEW('mv_vendas_resumo');

Query Rewrite — Redirecionamento Automático de Consultas

O Query Rewrite é um recurso poderoso do Oracle em que o otimizador de consultas automaticamente redireciona uma consulta feita às tabelas base para usar uma Materialized View, sem que o usuário ou aplicação precise alterar o SQL. Isso é transparente e pode trazer ganhos de performance de 10x a 1000x.

Habilitar Query Rewrite:

ALTER MATERIALIZED VIEW mv_vendas_resumo ENABLE QUERY REWRITE;
Verificar se Query Rewrite está sendo usado em uma consulta:

-- Executar a consulta com EXPLAIN PLAN
EXPLAIN PLAN FOR
SELECT
  regiao,
  SUM(valor) AS receita
FROM vendas
GROUP BY regiao;

-- Ver o plano de execução
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- Procurar a operação "MAT_VIEW REWRITE ACCESS FULL" no plano

Para Query Rewrite funcionar, são necessários:

  • O parâmetro de inicialização QUERY_REWRITE_ENABLED deve estar como TRUE
  • O parâmetro QUERY_REWRITE_INTEGRITY deve estar configurado (STALE_TOLERATED, TRUSTED ou ENFORCED)
  • A MV deve ter ENABLE QUERY REWRITE
  • O esquema deve ter o privilégio GLOBAL QUERY REWRITE ou QUERY REWRITE 

-- Conceder privilégio de Query Rewrite
GRANT QUERY REWRITE TO nome_usuario;

-- Verificar parâmetros de inicialização
SHOW PARAMETER query_rewrite;

Comparação: Views Convencionais vs Materialized Views

Característica View Convencional Materialized View
Armazenamento Não armazena dados fisicamente. Apenas a query SQL é guardada no dicionário de dados. Armazena os dados fisicamente em segmentos (tablespace). Ocupa espaço em disco.
Desempenho A consulta é reexecutada a cada SELECT. Pode ser lenta em tabelas grandes com JOINs complexos. Os dados já estão pré-calculados e prontos. Leitura quase instantânea, ideal para ambientes analíticos.
Atualização dos dados Sempre mostra o dado atual da tabela base em tempo real. Mostra o dado no momento do último refresh. Pode estar desatualizada (stale) até o próximo refresh.
Uso de índices Usa os índices das tabelas base. Pode ter seus próprios índices, otimizados para a consulta específica da MV.
Custo de manutenção Zero. Não requer manutenção. Requer agendamento de refresh, MV Logs para FAST refresh, e planejamento de espaço.
Casos de uso Segurança, simplificação de queries, abstração de schema. Ideal para sistemas OLTP. Data Warehousing, relatórios, dashboards, replicação. Ideal para sistemas OLAP.
Query Rewrite Não aplicável. Suporta Query Rewrite — o otimizador redireciona automaticamente consultas para usar a MV.
DML na própria view Permitido (com restrições) em views de tabela única sem agregação. Não permite DML. Apenas SELECT.

1. Criar view com JOIN entre 2 tabelas com WITH READ ONLY

Crie duas tabelas: alunos (id, nome, turma_id) e turmas (id, nome_turma, turno). Insira pelo menos 5 registros em cada. Depois, crie uma view chamada alunos_turma que faça JOIN entre elas mostrando: nome do aluno, nome da turma e turno. A view deve ser WITH READ ONLY. Teste tentando fazer um INSERT na view para verificar o erro.

Gabarito: 

-- Criar tabelas
CREATE TABLE turmas (
  id NUMBER PRIMARY KEY,
  nome_turma VARCHAR2(50),
  turno VARCHAR2(20)
);

CREATE TABLE alunos (
  id NUMBER PRIMARY KEY,
  nome VARCHAR2(100),
  turma_id NUMBER REFERENCES turmas(id)
);

-- Inserir dados
INSERT INTO turmas VALUES (1, 'Oracle Básico', 'Manhã');
INSERT INTO turmas VALUES (2, 'PL/SQL Avançado', 'Noite');
INSERT INTO turmas VALUES (3, 'DBA Fundamentals', 'Manhã');

INSERT INTO alunos VALUES (1, 'Ana Silva', 1);
INSERT INTO alunos VALUES (2, 'Bruno Costa', 1);
INSERT INTO alunos VALUES (3, 'Carla Souza', 2);
INSERT INTO alunos VALUES (4, 'Daniel Lima', 3);
INSERT INTO alunos VALUES (5, 'Eduarda Rocha', 2);

-- Criar view com JOIN e WITH READ ONLY
CREATE VIEW alunos_turma AS
SELECT
  a.id AS aluno_id,
  a.nome AS aluno,
  t.nome_turma AS turma,
  t.turno
FROM alunos a
JOIN turmas t ON a.turma_id = t.id
WITH READ ONLY;

-- Teste (deve gerar erro ORA-42399)
INSERT INTO alunos_turma (aluno_id, aluno, turma, turno)
VALUES (6, 'Teste', 'Novo', 'Manhã');

2. Criar Materialized View com agregação e configurar refresh ON DEMAND

Crie uma tabela vendas_curso (id, produto, valor, data_venda). Insira pelo menos 10 registros. Crie uma Materialized View chamada mv_curso_resumo que agrupe por produto com COUNT e SUM, usando BUILD IMMEDIATE e REFRESH COMPLETE ON DEMAND. Insira mais 3 registros na tabela base e execute o refresh manualmente. Verifique que os novos dados aparecem na MV.

Gabarito: 

-- Criar tabela base
CREATE TABLE vendas_curso (
  id NUMBER PRIMARY KEY,
  produto VARCHAR2(100),
  valor NUMBER(10,2),
  data_venda DATE DEFAULT SYSDATE
);

-- Inserir dados iniciais
INSERT INTO vendas_curso VALUES (1, 'Curso Oracle SQL', 1500, DATE '2026-01-15');
INSERT INTO vendas_curso VALUES (2, 'Curso PL/SQL', 2000, DATE '2026-01-20');
INSERT INTO vendas_curso VALUES (3, 'Curso Oracle SQL', 1500, DATE '2026-02-01');
INSERT INTO vendas_curso VALUES (4, 'Curso DBA', 3500, DATE '2026-02-10');
INSERT INTO vendas_curso VALUES (5, 'Curso PL/SQL', 2000, DATE '2026-02-15');
INSERT INTO vendas_curso VALUES (6, 'Curso Oracle SQL', 1500, DATE '2026-03-01');
INSERT INTO vendas_curso VALUES (7, 'Curso DBA', 3500, DATE '2026-03-05');
INSERT INTO vendas_curso VALUES (8, 'Curso PL/SQL', 2000, DATE '2026-03-10');
INSERT INTO vendas_curso VALUES (9, 'Curso Oracle SQL', 1500, DATE '2026-04-01');

COMMIT;

-- Criar Materialized View

BUILD IMMEDIATE
REFRESH COMPLETE ON DEMAND

  COUNT(*) AS quantidade_vendida,
  SUM(valor) AS receita_total,
  AVG(valor) AS ticket_medio
FROM vendas_curso

-- Consultar MV (deve mostrar 3 produtos com seus totais)

-- Agora os novos dados aparecem
SELECT * FROM mv_curso_resumo;

3. Comparar performance de SELECT direto vs Materialized View com EXPLAIN PLAN

Utilizando a tabela vendas_curso criada no exercício anterior e a mv_curso_resumo, execute EXPLAIN PLAN em ambas as consultas abaixo e compare os planos de execução. Verifique a diferença de custo e operações realizadas. Habilite Query Rewrite na MV e execute novamente a consulta à tabela base para ver se o Oracle redireciona para a MV automaticamente.

Gabarito: 

-- 1. EXPLAIN PLAN na tabela base diretamente
EXPLAIN PLAN FOR
SELECT
  produto,
  COUNT(*) AS quantidade,
  SUM(valor) AS total
FROM vendas_curso
GROUP BY produto;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- 2. EXPLAIN PLAN na Materialized View
EXPLAIN PLAN FOR
SELECT * FROM mv_curso_resumo;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
-- Você verá: TABLE ACCESS FULL de MV_CURSO_RESUMO (custo muito menor)

-- 3. Habilitar Query Rewrite
ALTER MATERIALIZED VIEW mv_curso_resumo ENABLE QUERY REWRITE;

-- 4. EXPLAIN PLAN com Query Rewrite habilitado
-- (consulta a tabela base, mas Oracle pode redirecionar para a MV)
EXPLAIN PLAN FOR
SELECT
  produto,
  COUNT(*) AS quantidade,
  SUM(valor) AS total
FROM vendas_curso
GROUP BY produto;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
-- Se aparecer "MAT_VIEW REWRITE ACCESS FULL", o Oracle usou a MV automaticamente
