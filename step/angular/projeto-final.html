Projeto Final — Controle de Estoque

Neste projeto final, você deverá criar uma aplicação Angular chamada controle-estoque, com autenticação, controle de perfil de usuário, consumo de APIs externas e organização por componentes e serviços.

1. Branch do projeto

Crie uma branch chamada projeto-final baseada na branch main.

Todo o desenvolvimento do projeto final deverá ser feito nessa branch.

2. Criação da aplicação

Dentro da branch projeto-final, crie uma nova aplicação Angular chamada: 


  controle-estoque

Após criar a aplicação, entre na pasta do projeto e rode a aplicação para verificar se tudo foi criado corretamente.

3. Componentes obrigatórios

A aplicação deverá conter os seguintes componentes:

  • login
  • dashboard
  • painel-adm
  • formulario-registro

O componente formulario-registro deverá ser criado como componente filho do componente painel-adm.

Isso significa que o formulário de registro não deve ser acessado diretamente por uma rota própria. Ele deve aparecer dentro da tela do painel administrativo.

4. Tela de login

O componente login deverá conter um formulário para entrada do usuário no sistema.

Esse formulário deverá possuir, no mínimo:

  • Campo de e-mail
  • Campo de senha
  • Botão para realizar login
  • Mensagem de erro caso o login falhe

Ao enviar o formulário, o componente deverá chamar um serviço de autenticação responsável por fazer a integração com a API de login.

A API de login que deverá ser utilizada é:


  https://projeto-node-step-git-main-fabios-projects-d2648344.vercel.app/api/auth/login

Quando o login for realizado com sucesso, o token retornado pela API deverá ser salvo no localStorage.

Após o login, o usuário deverá ser redirecionado para a tela de dashboard.

5. Dashboard

O componente dashboard deverá ser responsável por exibir a lista de produtos do sistema.

Essa lista deverá ser carregada por meio de uma integração com a API: 


  https://fakestoreapi.com/products

A chamada para essa API deverá ser feita por meio de um serviço Angular específico para produtos.

O dashboard deverá exibir, no mínimo, as seguintes informações de cada produto:

  • Nome ou título do produto
  • Categoria
  • Preço
  • Imagem do produto

Também é recomendado exibir uma mensagem de carregamento enquanto os produtos estão sendo buscados.

6. Painel administrativo

O componente painel-adm deverá representar uma área restrita da aplicação.

Essa tela só deverá ser acessível por usuários com perfil de administrador.

Dentro do painel administrativo deverá aparecer o componente filho formulario-registro.

O painel administrativo poderá conter também um título, uma breve descrição e um link ou botão para voltar ao dashboard.

7. Formulário de registro

O componente formulario-registro deverá conter um formulário para cadastro de novos usuários.

Esse formulário deverá possuir, no mínimo:

  • Campo de nome
  • Campo de e-mail
  • Campo de senha
  • Campo para seleção do perfil do usuário
  • Botão para registrar o usuário
  • Mensagem de sucesso ou erro após tentativa de cadastro

O campo de perfil deverá permitir escolher entre usuário comum e administrador.

Ao enviar o formulário, o componente deverá chamar o serviço de autenticação responsável por fazer a integração com a API de registro.

A API de registro que deverá ser utilizada é:


  https://projeto-node-step-git-main-fabios-projects-d2648344.vercel.app/api/auth/register

8. Serviços obrigatórios

O projeto deverá possuir, no mínimo, dois serviços:

  • AuthService
  • ProdutoService

AuthService

O serviço de autenticação deverá centralizar toda a lógica relacionada ao usuário.

Ele deverá ser responsável por:

  • Fazer login
  • Fazer registro de usuário
  • Salvar o token no localStorage
  • Buscar o token salvo
  • Remover o token em caso de logout
  • Buscar os dados do usuário logado

Para buscar os dados do usuário logado, deverá ser utilizada a API: 


  https://projeto-node-step-git-main-fabios-projects-d2648344.vercel.app/api/auth/me

Essa requisição deverá enviar o token salvo no localStorage.

ProdutoService

O serviço de produtos deverá centralizar a chamada para buscar os produtos da Fake Store API.

O componente dashboard não deve chamar a API diretamente. Ele deve chamar o serviço de produtos.

9. Guardião de rota

O projeto deverá possuir um guardião de rota chamado roleGuard.

Esse guardião deverá proteger a rota do painel administrativo.

O roleGuard deverá:

  • Verificar se existe token salvo no localStorage
  • Buscar os dados do usuário logado usando a rota /auth/me
  • Verificar o perfil do usuário retornado pela API
  • Permitir acesso apenas se o usuário tiver perfil de administrador
  • Redirecionar usuários não autorizados

Caso não exista token, o usuário deverá ser redirecionado para a tela de login.

Caso exista token, mas o usuário não seja administrador, ele deverá ser redirecionado para o dashboard.

10. Rotas obrigatórias

A aplicação deverá possuir as seguintes rotas:

  • /login — tela de login
  • /dashboard — tela com lista de produtos
  • /painel-adm — tela administrativa protegida

A rota /painel-adm deverá usar o roleGuard.

A rota inicial da aplicação poderá redirecionar o usuário para /login.

11. Requisitos técnicos obrigatórios

  • Usar Angular
  • Usar componentes standalone
  • Usar services para chamadas HTTP
  • Usar HttpClient
  • Usar localStorage para armazenar o token
  • Usar guardião de rota para controle de acesso
  • Usar rotas para navegação entre telas
  • Consumir a Fake Store API no dashboard
  • Consumir a API de login
  • Consumir a API de registro
  • Consumir a API de usuário logado no guardião
  • Os formulários devem utilizar os componentes do Angular Material

12. Fluxo esperado da aplicação

  1. Usuário acessa a aplicação
  2. Usuário é direcionado para a tela de login
  3. Usuário informa e-mail e senha
  4. Aplicação chama a API de login
  5. API retorna um token
  6. Token é salvo no localStorage
  7. Usuário é redirecionado para o dashboard
  8. Dashboard carrega produtos da Fake Store API
  9. Usuário administrador pode acessar o painel administrativo
  10. roleGuard verifica o token e o perfil do usuário
  11. Painel administrativo exibe o formulário de registro
  12. Administrador pode cadastrar novos usuários

13. Entrega do projeto

Ao finalizar o projeto, suba o código para o Git na branch projeto-final.

O commit final deverá indicar que o projeto final foi concluído.


  git add .
  git commit -m "projeto final controle de estoque"
  git push
